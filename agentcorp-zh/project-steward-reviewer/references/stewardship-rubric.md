# 管家评审维度

本维度将项目管家评审员的"所有者品味"转化为可检查的问题。它不是通用最佳实践清单；仅当你能展示当前变更影响项目长期健康时才使用。

## 外部锚点

- Google Engineering Practices 将代码评审的目标定义为使整体代码健康随时间改善，并明确允许评审员拒绝他们不希望纳入系统的功能；它同样强调设计、复杂度、测试、文档和系统上下文。
- Google 的所有权模型将同行评审、代码所有者批准和可读性分开；代码所有者关注变更是否适合其代码区域、是否增加技术债务，以及团队是否有能力维护它。
- Apache 项目成熟度模型是成熟项目的参考：代码可构建且可追溯；强依赖许可不应增加限制；发布过程可重复；质量状态透明；安全、向后兼容和迁移说明被优先考虑；重要决策以书面形式记录。
- Open Source Guides 强调维护者可以友善但坚定地说不：一项贡献可能有价值，但如果它不符合项目的范围、愿景或实现质量，则不应被接受。

## 评审维度

### 1. Project Fit（项目契合度）

问：这项能力是否属于项目核心职责，还是仅被某个调用方、团队、客户或短期场景需要？

强信号：
- 新能力要求项目长期拥有一个新产品概念，但需求无法证明它属于项目的身份。
- 代码为单个业务场景添加了通用入口点、全局配置或跨模块概念。
- 它本可以通过插件、调用方组合、独立服务或本地扩展解决，却被塞进核心。

### 2. Ownership And Maintenance（所有权与维护）

问：谁来维护这部分？团队是否有专业知识、监控、升级、回滚和事件处理路径？

强信号：
- 新增了依赖、外部系统、运行时、定时任务、数据迁移或发布步骤，但没有所有者。
- 变更引入了兼容 shim、映射表、特例列表或双写路径，需要持续更新。
- 关键知识只存在于实现者脑中，从未被捕获到设计记录、注释、契约或运行手册中。

### 3. Architectural Boundary（架构边界）

问：变更是否尊重现有模块边界和信息隐藏，还是将内部细节泄露成长期耦合？

强信号：
- 调用方开始依赖内部状态、内部命名、存储细节或权宜协议。
- 本应保持局部的概念被提升为全局抽象，迫使更多模块了解它。
- 为绕过当前限制而添加了后门、全局标志、隐式回退或跨层调用。

### 4. Public Surface And Compatibility（公共表面与兼容性）

问：新增或修改的公共/共享表面是否值得长期承诺？

强信号：
- 新端点、schema 字段、CLI 参数、配置选项、导出类型、事件或 JSON/RPC 方法没有兼容性策略。
- 破坏性变更没有版本、弃用、迁移说明或调用方影响分析。
- 一个"临时"公共选项没有移除条件，且实际上会被用户或其他模块依赖。

### 5. Change Shape And Reviewability（变更形态与可评审性）

问：这个 diff 是否能让维护者清楚看到真正的语义变更？

强信号：
- 功能变更与大规模重排序、格式化、顺手重构、命名迁移或测试重写混在一起。
- 单个变更跨越太多模块，无法作为一个可评审的故事来解释。
- 关键决策没有对应的 issue、设计工件、注释或提交上下文。

### 6. Debt Ledger（债务台账）

问：如果接受了债务，它是否被记录、有界并附有退出路径？

强信号：
- TODO/FIXME/HACK 只写了"稍后清理"，没有触发条件、所有者或验证方法。
- 兼容 shim、双写、回退或特例分支没有 sunset 计划。
- 为了赶截止日期而牺牲结构，但没有解释为什么这是唯一可接受的临时选择。

### 7. Test And Documentation As Assets（测试与文档作为资产）

问：测试和文档是否帮助未来维护者安全演进，而不仅仅是让本轮通过？

强信号：
- 测试依赖过度 mock，仅断言调用次数，核心行为损坏时不会失败。
- 新公共行为、配置、迁移、发布步骤或事件处理没有文档。
- 文档仅复述如何使用，而没有记录为何这样设计、边界在哪里，以及何时不应使用。

## 输出指导

按以下结构撰写每条发现：

- 长期健康影响：谁将在未来承担什么维护成本。
- 证据：代码/计划/设计/文档路径和行号。当发现对更广泛的仓库做出断言——无所有者、无其他调用方、无设计记录、无兼容性说明——时，包含你运行的搜索命令及其返回；否则将发现置信度降级为 medium。
- 建议行动：缩小范围、移至插件/调用方、添加契约、拆分 PR、添加设计记录、添加 sunset 计划、请求人类所有者接受风险等。
- 路由：`review-fixer`、`implementation-planner`、`solution-architect`、`release owner`、`human owner`。

当问题本质上是项目方向或所有者的权衡时，不要假装工具可以裁决；清晰列出选项并路由给人类所有者。
