# Fable 之后的 Skill Carving：真实语料暴露了“规则越多越稳”的代价

> 主要来源：[garrytan/gstack Issue #1994](https://github.com/garrytan/gstack/issues/1994)，创建于 2026-06-13，检查于 2026-07-16。Issue 当时仍为 Open；本文把它视为社区测量与迁移提案，不把尚未落地的建议写成结果。

## 一、这不是抽象争论，而是一组现有 Skill 的体检

Issue 作者在真实安装的 gstack v1.57.10.0 上检查了 54 个 Skill：

- 54 个 description 元数据合计约 5.2 KB，约 1.3k token，作者认为这一层足够精简；
- 49 个 `SKILL.md` 正文超过作者引用的 500 行建议上限；
- 最大的几个 Skill 达到 1,800 到 2,275 行；
- 单个 `/codex` 流程在进入任务前包含 46 次 `gstack/bin` 调用，作者估算固定开销约 3,000–4,200 token；
- 从 2 月以来的个人 transcript 看，54 个 Skill 中只有 11 个贡献了 100% 的实际调用。

这些数字不能直接证明长 Skill 一定更差，但它们证明了两个工程事实：大部分能力从未被使用；每次会话仍可能为历史脚手架付费。

## 二、Carving 不是机械删字，而是改变 Skill 的职责

作者提议采用 Fable 官方迁移标准：用“目标、理由、边界、自我验证”替代大量 `MUST` 驱动的步骤脚本，并删除只为旧模型弱点存在的规则。

gstack 此前已经进行多轮 `carve`：把常用 Skill 拆成骨架与按需章节，减少常驻加载。这个方向说明 Skill 可以有两层：

- 一个很短的主语义，告诉模型当前角色、目的和承重边界；
- 只有当前情形需要时才读取的参考、协议、模板和脚本。

这比把全部经验塞进主提示词更符合长程任务的注意力经济。

## 三、使用集中度比 Skill 数量更能说明系统形状

54 个 Skill 看起来能力丰富，但如果真实工作只调用其中 11 个，系统可能存在三种问题：

- 多个 Skill 的边界过窄，用户不知道何时调用；
- 一些 Skill 是旧流程留下的死资产；
- 核心能力被拆得太碎，触发和切换成本高于收益。

下一代 Skill 仓库不应把“数量更多”当成熟度。更值得观察的是：哪些能力反复改变结果；哪些只在说明文档中存在；哪些可以被一个高纯度母能力覆盖；哪些必须保留独立权限或验证边界。

## 四、模型感知的渲染配置有诱惑，也有风险

Issue 建议为 Fable 提供更简洁的渲染配置，避免给新模型加载旧模型脚手架。方向合理，但要谨慎：

- 同一模型在不同任务、effort 和运行环境下也会有不同表现；
- “Fable 用短版，旧模型用长版”可能演变成难以维护的提示分叉；
- 如果没有同一组真实 trace 的对比评测，配置只是新的猜测。

更稳妥的顺序是先建立一个最短共享核心，再只为测量到的模型差异添加小型覆盖层。

## 五、这个案例对新 AgentCorp 的直接要求

1. 主 Skill 只保存决定行为的高密度语义，模板和协议按需读取。
2. 每条常驻规则都应能指向一个真实失败、稳定偏好或不可变边界。
3. 统计实际触发和产出价值，定期合并、归档或删除低价值 Skill。
4. 精简必须逐组进行，并在同一批任务上复测；不要一次重写整个 Skill 栈。
5. Skill 迁移成功的标准不是行数下降，而是质量不降、成本下降、无用请示和机械流程减少。

## 六、证据边界

这份 Issue 提供了真实安装的静态测量和作者自己的 transcript 统计，但没有发布 Fable 精简前后的完整 benchmark，也没有证明提案已经合入。因此，它最可靠地支持“现有 Skill 存在显著常驻体积与使用集中问题”，不能单独支持“某种具体精简一定提高质量”。
