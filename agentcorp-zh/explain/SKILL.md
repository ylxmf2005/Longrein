# Explain

这是一个可复用的 AgentCorp 沟通能力，不是 delivery phase，也不是有独立 gate 的角色。任何角色在其输出必须让没读过代码、日志或 artifact 的人看懂时都可以加载它。你不下任何裁决、不过任何 gate——review、验证和验收的结论属于它们各自的 owner；你负责让这些结论能被权衡。当一个解释 branch/PR/diff 的请求实际上是“在合并前让我理解这整个变更”时，推荐 `walkthrough`，而不是把一个 explanation set 撑大。

**你的问题：读完这份解释后，读者能对下一个决定表态吗？** 廉价的失败是信心表演——一段流畅的总结，听起来已成定论，藏起了哪些是确知、哪些是猜测，让读者只顾点头。点头不是标准，参与才是：修不修、发不发、选哪个方案。

## 铁律

```
每个结论都带着它的状态和证据一起交付。
```

已确认、大概率，还是还没验证——外加让读者能自己核对的把手：一行日志、一个 `file:line`、一条命令及其输出、一条测试结果。诚实的空白胜过编造的事实；读者无法核验的解释就是信心表演，写得再清楚也一样。

## 了解你的读者

默认读者：一个很聪明但零上下文的 sponsor——他们可能不知道 repo、领域词汇、历史决策或惯例。但当读者被指明或显而易见时要校准：专家想要把 delta 和证据摆在前面、术语原样保留；新手需要先教正常情况再讲异常情况；混合读者先给普通含义、精确术语放在括号里。无论读者是谁：在依赖术语之前先翻译它，在讲机制之前先说明这一点为什么重要，绝不用居高临下的话（“很简单”“显然”“只是”）来充数。

## 输出模式

- `output_mode: inline` —— 在对话里回答。适合一个小答案、一段短状态、单个概念。
- `output_mode: artifact` —— 写到任务根目录并返回一个简短指针。适合读者会反复打开、标注或逐项决策的多点集合；也适合用户说“落库”“写下来”“做个文档”“方便看”的任何时候。加载 `references/artifact-format.md` 获取路径、set/index 结构和 frontmatter。
- `output_mode: auto`（默认）—— 自己判断，但有两种情况**强制** `artifact`：解释值得配一张图或任何终端无法渲染的东西（丢出未渲染的源码不算交付，为了留在 inline 而砍掉本该有的图也不是选项）；或者请求是读者会逐项处理的多点集合。

## 默认结构

除非用户要求别的结构，就用这个形状；小答案把各节合并成自然段，而不是硬塞标题。

1. **一句话结论** —— 用一两句话说出最重要的结论。
2. **背景** —— 在讲发生了什么之前，先讲系统、功能、测试或 artifact 是*为了*什么：连正常情况都建立不起来的读者，评估不了异常情况。
3. **发生了什么** —— 用普通语言，含义先于原文：先总结日志、trace 和 diff，再引用支撑这个含义的那几行。
4. **为什么重要** —— 影响用户、线上、开发流程、信心，还是只影响某个测试片段；把因果讲清楚（“因为 X，所以 Y 出问题”）。
5. **当前状态** —— 已确认 / 已修复 / 未验证 / 被阻塞 / 仍不确定，分开列。
6. **这对你的下一个决定意味着什么** —— 当一个选择取决于它时，说明什么取决于此、以及你会怎么选。没有依赖时省略。
7. **术语小抄** —— 只收本次解释必须的词，第一次出现时定义。

写 bug、测试进度或实现解释之前，先加载 `references/genres.md` 获取每类的检查单和一个成型示例。

## 例子和图

优先用一个具体例子，而不是抽象叙述：读者会看到、发送、点击或决定什么——贴近当前任务、真实可信。解释三步以上的流程、状态转换和失败路径、跨角色或服务的职责归属，或一棵决策树时，默认配一张紧凑的 Mermaid 图（`flowchart`、`sequenceDiagram`、`stateDiagram-v2`，节点用普通语言命名，配一句解读）。只会重复段落的图就不要画。

## 红旗信号——一旦发现自己这样想就停下

| 念头 | 现实 |
| --- | --- |
| “画图就得落库，用文字讲能留在 inline。” | 砍掉本该有的图既交付了更差的解释，又让规则两头落空。图要交付，落库也随之而来。 |
| “原因很明显，当事实写更干净。” | 没标注的推断就是信心表演。已确认、大概率还是还没验证——带上把手。 |
| “把堆栈贴上去，它自己说明一切。” | 原始输出是证据，不是解释。先说含义；只引用支撑它的那几行。 |
| “读者很急，背景就跳过吧。” | 两句正常情况的说明，比一个对异常情况的错误决定便宜。 |
| “这个分支的解释再多加几个条目文件就够了。” | 围绕同一个变更不断膨胀的集合，是戴着 explain 帽子的 walkthrough 请求。推荐 `walkthrough`。 |
| “读起来很顺，发吧。” | 读起来顺不是标准。读者现在能对下一个决定给出支持或反对的理由吗？ |

## 交付前

每个结论都带状态标签和证据把手；术语第一次出现就有定义；任何本该有的图都在，并落在能渲染的地方；落库的文件遵循 `references/artifact-format.md`（index/item 类型、`task_id`、条目自足、没有任何东西被 stage 或提交）。
