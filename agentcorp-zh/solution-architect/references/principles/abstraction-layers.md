# 抽象层

## 描述

设计时，使用此维度评判你提出的结构在架构层之间分离关注点的程度。在良好设计的系统中，每一层提供与相邻层不同的抽象——追踪一个操作的方法调用时，应在每个层级看到抽象的变化。当相邻层暴露相似的抽象时，它暗示了职责边界混乱与浅薄的设计，增加了接口复杂度却没有成比例的功能增益。

关键关注点包括：
- **层分离**：每一层是否提供了有意义的差异抽象
- **穿透方法/变量**：无转换地穿越各层的方法或变量
- **复杂度放置**：复杂度被推给了调用者还是被拉入了模块

## 核心原则

### 不同层，不同抽象

系统中的每一层应提供与上下层不同的抽象。当相邻层有相似抽象时，它们很可能在设计中没有赢得自己的位置——增加了结构开销却没有足够的功能收益。

**示例**：文件系统的各层提供了真正不同的抽象：变长字节数组（文件）→ 固定大小缓存块 → 设备驱动块 I/O。每个级别都改变了用户对数据的思考方式。

### 把复杂度向下拉，不要向上推

当不可规避的复杂度出现时，模块应内部吸收它，而不是暴露给用户。由于模块通常有很多用户但只有少数开发者，让开发者一次性处理复杂度比让每个用户都应对它更好。优先简单的接口，而非简单的实现。

**反模式**：暴露配置参数，或对模块本可内部处理的情况抛出异常，迫使每个调用者或管理员解决同样的问题。

### 消除穿透方法

穿透方法（简单委托给另一个签名相似的方法）暗示了职责边界混乱。它们让类变得更浅——增加了接口复杂度却没有增加功能，并在层之间创造了不必要的依赖。

**解决方向**：
- 将底层类直接暴露给调用者
- 重新分配功能以消除跨类调用
- 如果无法清晰分离，合并类

### 避免穿透变量

被长串方法传递的变量迫使所有中间层意识到它们并不使用的关注点。这会跨层传播知识，让演化变得困难。

**解决方向**：
- 使用上下文对象存储系统级状态，并在主要对象中保存其引用
- 寻找可承载信息的现有共享对象
- 避免全局变量（它们阻止多系统实例并制造测试问题）

### 匹配内部与外部抽象

类的接口应提供与其内部实现不同的抽象。当接口与实现使用相同的表示时，这个类很可能是浅层的。

**示例**：一个内部将文本按行存储的文本类，仍应暴露字符级接口（在任意位置插入/删除），而不是行级接口（整行读取/写入）。抽象差距代表有价值的功能。

## 危险信号

在你提出的设计中以及你阅读代码时注意以下模式：

### 穿透方法
- 只做用相同或非常相似签名调用另一个方法的方法
- 没有转换或新增逻辑的委托链
- 大多数公共方法是简单转发器的类
- 例外：根据参数选择调用哪个方法的调度器是合理的

### 穿透变量
- 被多个方法签名穿过但只在最深层使用的方法
- 长参数列表中大部分是简单转发的参数
- 必须通过调用链传递的全局状态

### 跨层相似抽象
- 接口签名与实现方式紧密镜像
- 暴露内部表示细节的外部 API
- 相对于接口大小增加极少功能的 wrapper 或 decorator

### 复杂度向上推
- 模块本可内部决定的配置参数
- 模块本可处理的条件却抛出异常
- 要求调用者执行多步协调（拆分/合并、设置/拆卸）的 API
- 要求用户理解内部结构的简单操作

### Decorator 过度使用
- 小功能拆分成独立的 decorator 类，本可合并到基类
- 多个浅 decorator 各增加极少功能
- 被用于所有人都需要的通用功能上的 decorator

### 无价值接口重复
- 多个签名相似却不提供不同功能的方法
- 下层 API 被不经转换地复制到上层
- 连接层却不增加抽象的样板代码

## Issue Tags

在设计产出中标记分层问题时使用以下标签：

- **[Pass-Through Method]**：方法委托给另一个方法而不增加功能
- **[Pass-Through Variable]**：变量穿过不使用它的方法
- **[Similar Layer Abstractions]**：相邻层暴露相似抽象
- **[Complexity Pushed Up]**：模块迫使调用者处理复杂度
- **[Shallow Decorator]**：Decorator 的接口大小相对于增加的功能过浅
- **[Interface Matches Implementation]**：外部 API 镜像内部表示
