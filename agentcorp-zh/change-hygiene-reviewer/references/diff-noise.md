# Diff 噪声审查

仅在任务分配或 diff 信号指向空白、格式化、折行、注释、随手重构或格式化器影响范围扩散时加载。目标是找出没有行为价值、非工具强制、却增加 MR/PR 阅读成本的 hunk。

## 流程

1. 先运行机械扫描——在正确的仓库中、针对正确的范围。
   - 从代码工作区根目录运行扫描器（当任务使用独立检出时为 `code_worktree`/`code_location`）：`python3 <此 skill 的绝对路径>/scripts/diff_noise_scanner.py --json --base <目标分支>`。脚本在当前工作目录中调用 `git diff`，因此必须在被审查的仓库内运行——从 skill 目录运行会扫描 skill 仓库，而非代码仓库。
   - 对于 MR/PR 或功能分支，优先使用任务分配指定的 base/head。没有明确范围但能定位目标分支时，使用 `--base <目标分支>`——脚本会自行计算 merge-base(base, HEAD)。对于原始范围，使用三点形式 `--diff <base>...HEAD`；两点 `<base>..HEAD` 是将 base **tip** 与 HEAD 比较，会混入主线中不在 HEAD 中的提交的反向 hunk，导致你把别人已合并的变更标记为本 MR 的噪声。
   - 仅在发起人或任务分配明确要求审查暂存/工作区/当前候选内容时才查看本地未提交的变更：没有 source 标志时脚本扫描暂存 diff，`--worktree` 扫描未暂存的工作区。注意：没有 source 标志且暂存 diff 为空时，脚本**会静默回退到未暂存的工作区 diff**——输出的 `source` 字段会显示 `git diff (fallback; staged diff was empty)`。在 MR/PR 上下文中始终传入明确的 `--base`/`--diff`/`--file`，使用任何发现前始终检查 `source` 是否匹配所分配的范围。
   - 对于外部 diff 文件，使用 `--file <diff文件>`。
   - 需要逐块浏览比较 diff 时，使用 `--base <目标分支> --chunks hunk|group|line`；`group` 表示同一 hunk 内连续的 `+/-` 变更组，最适合逐块判断每处是否属于本次变更。
   - 调整块大小时，使用 `--unified 0` 细化上下文，或 `--inter-hunk-context <n>` 合并相邻 hunk。
   - 将输出的 `verdict`、`counts_by_category` 和关键发现作为证据；扫描器是证据收集者，不是最终裁判。
2. 确定变更面。
   - 优先使用任务分配给出的 base/head 或 diff 文件。在 MR/PR 上下文中，未提交的工作区默认只是本地状态，不是 MR/PR diff。
   - 如果分配未给出范围，先从目标分支确定 merge-base..HEAD；仅当任务明确为"审查暂存/工作区/当前候选"时才切换到暂存或工作区 diff。
   - 查看 `git diff --stat`、`git diff --name-status` 和关键文件的 `git diff -- <路径>`。
   - 对于怀疑为空白噪声的 hunk，用 `git diff -w -- <路径>` 或 `git diff --ignore-space-change -- <路径>` 比较。
3. 先分类，再判断。
   - `semantic`：实现任务所需的行为、契约、数据、错误处理和测试更新。
   - `mechanical-required`：项目格式化器/lint 必须产生的最小格式化变更。
   - `noise`：不改变行为、非工具强制、增加阅读成本。
   - `mixed`：单个 hunk 混合了语义和噪声；建议拆分 hunk 或回退噪声行。
4. 对每个噪声/混合 hunk 问三个问题。
   - 此需求、Story Spec、审查发现或测试失败是否要求它？
   - 项目格式化器/lint 是否明确要求它？
   - 如果删除这些行，行为、验证和 lint 是否仍然成立？

## 需要关注的问题

- **纯空白变更**：忽略空白后 hunk 消失，且非格式化器/lint 要求。
- **过度折行**：函数调用、字面量、方法链、列表或断言被拆得过窄，而原始代码未超出项目可接受的行宽。
- **缺乏充分理由的重排**：文档字符串、注释、字符串、错误消息或日志文本仅被重新排列或折行。
- **随手格式化**：相邻函数、未触及的分支、无关 import 或已有测试被顺带格式化。
- **随手重构**：当任务不需要时，提取单调用者逻辑为函数、合并/拆分函数、重命名或重排代码块。
- **纯注释噪声**：改写注释、添加解释性注释或删除注释，但未修正过时或错误的信息。
- **生成代码或格式化器影响范围扩散**：一次格式化运行改变了大量无关文件；建议缩小格式化器范围或拆分为单独的 MR/提交。

## 行宽与折行

不要把"更窄"当作"更好"。在 Python/后端代码中，只要满足项目 lint/格式化器和周围文件风格，优先保持紧凑、可读、低 diff 的形式。

报告过度折行前，确认至少一个条件成立：同一表达式只是被模型跨行拆分了；周围代码通常允许更宽的行；当前拆分产生明显噪声；或 `git diff -w` 或手动 hunk 对比显示主要变更来自折行/缩进。

不要报告：格式化器必须拆分的长行、折行明显提升可读性的复杂表达式、为避免过长字符串/SQL/URL/类型签名而进行的必要折行。

## 输出要求

仅报告有可操作修复的噪声。对每个发现，说明文件/行号或 hunk、扫描或 diff 命令证据（包括扫描器实际使用的 `source`）、为什么它增加审查成本，以及建议是删除、回退、拆分提交还是保留但需解释。
