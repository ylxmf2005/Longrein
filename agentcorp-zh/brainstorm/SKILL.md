# Brainstorm

这是一个通用的 AgentCorp 思考能力，不是交付阶段，也不是拥有独立 gate 的角色。任何角色都可以加载它；它的主要归属是 `delivery-orchestrator` 在 `validate-requirements` 期间。任务内形成的决定要折叠回所属阶段的工件（通常是 `requirements/validated-requirements.md`）——绝不要顺手写进实现、架构或下游工件。独立探索可以留在对话中；思考不必硬造出工件或结论才算有价值。

**你的问题：赞助方到底需要什么，才能表达得足以被验证？** "模糊"背后隐藏着两种不同的缺口，它们需要不同的工具：

- **已知未知**——赞助方知道但尚未确定的事实。用**模式 1**关闭：仓库无法回答的聚焦问题。
- **未知已知**——赞助方无法表达但在看到时能识别的偏好。用**模式 2**浮现：完整、真正不同的选项，供其*反应*——反应是廉价的，组合是昂贵的。
- **未知未知**——无人踏勘过的地形——不是你能直接处理的。先运行 `probe`，将其报告作为头脑风暴的基础；在没人踏勘过的地形上采访赞助方，只会把盲点洗白成需求。

## 铁律

```
你的推荐是反应材料，绝不是决定。
```

两种模式都需要计算出一个推荐的默认选项——它的存在是为了让赞助方有具体的东西可以反应。没有任何东西会成为需求，直到赞助方明确选择它，无论你的默认选项看起来多么显而易见，或赞助方回复多么缓慢。未经确认的默认选项被记录为假设并保留问题开放，或者头脑风暴返回 `blocked`，指明缺失的事实。

## 操作原则

- **在确定形状前先验证意图。** 保留赞助方的重要措辞，但挖掘底层问题、目标用户、成功信号和非目标。压力测试问题的真实性、用户价值、范围、风险和更简单的替代方案——但不要把赞助方变成采访对象。
- **打开线索，不把对话导进审问漏斗。** 赞助方还在思考时，浮现几条仍活着的方向、张力或类比，让对话跟着真正有信息量的线索走。新证据改变问题形状时就转向；不要把每轮都塞进预写问卷，也不要过早把多个合理方向压成一个。
- **落在实地上，并在有用时画出来。** 话题背后有真实系统时，先读相关代码和已有工件。紧凑的图、状态草图或比较表若比再写一段话更能暴露结构与权衡，就使用它；可视化是承载证据的思考，不是装饰。
- **绝不问仓库能回答的问题。** 在问题触及赞助方之前，先检查代码、文档、已有工件或 `teamspace/learnings/` 是否已经回答了它。一个仓库本可回答的问题，是把工作推回给原本委托它的人。
- **让赞助方反应，而非组合。** 每个问题附带 2–4 个具体候选答案、它们的权衡和你的推荐默认；每个方向都提供一条完整到足以选择、细化或拒绝的路径。
- **为事实声明提供依据。** 当答案取决于仓库行为、文档或先前任务时，先检查证据——如有探测报告，或本地标准（`AGENTS.md`、`README*`）、先前 `teamspace/` 工件、相关代码——并将凭据（`file:line`、命令+输出、文档路径）带入决策记录。将外部引用克隆到临时位置并阅读作为证据；未经明确授权，不要从中运行任何内容。外部证据若不是快速读一遍就能定论——例如 SOTA、既有方案、某个依赖的真实能力——该交给 `parallel-researcher`，不是留作 brainstorm 的功课。
- **保持需求与实现的界限清晰。** 头脑风暴产品行为、旅程、成功指标和范围；把表、模块、API 和算法留给设计，除非头脑风暴明确是关于技术选择。

## 开放探索姿态

当赞助方想思考、比较、调查或顺着一个直觉走，但还没有要求作出决定时，使用这个姿态。它刻意保持自适应：相关时检查真实代码或工件，挑战假设，浮现多条仍活着的线索，然后跟随赞助方选择的方向。没有固定步骤、强制推荐或必交工件。只有当思考已经凝结到值得保存时才总结。

一旦决定真正形成，就明确完成过渡：选了什么、拒绝了什么以及原因、还剩哪些未知、影响哪份所属工件。可以提议把它收回到那份工件，但不要悄悄另写一份旁支记录。赞助方若要求开始实现，且意图已经足够清楚，就交还所属的交付或实现角色，按它的 gate 继续；不存在人为的“先退出探索才能实现”锁。

## 模式 1：逐问追问

当方向基本清晰但具体事实会改变置信度时使用。先静默构建完整问题列表，按杠杆排序——首先是对架构或数据模型有影响的答案，然后是范围和可观察行为，接着是旅程和 UX，最后是美观——每轮**只问一个**，每个附带候选答案、每个暗示的权衡和你的选择及理由。将每个答案折叠进持续运行的需求图景。**当剩余答案不再会改变方向时停止**；将剩余内容记录为假设或开放问题。如果未解决的事实仍会阻碍设计或测试计划，返回 `blocked` 并精确指出缺失的内容。

## 模式 2：多路径提案

当请求指向了目标但未指定形状，或几个合理的形状会导向不同的已验证需求时使用。展示 2–4 条完整路径并请赞助方选择；加载 `references/proposal-paths.md` 获取透镜集合、每条路径的必备结构、视觉/交互形状的原型指南以及推荐规则。不可协商之处：路径必须真正发散（仅在措辞上不同的两条路径是一条路径）；至少一条路径故意超出赞助方已声明的偏好（隐藏偏好只在边界处浮现）；最终推荐只使用路径中可见的要素。

对单一方向的深度压力测试，加载 `references/grill-lenses.md` 并仅使用适用的透镜。

## 红旗——当你察觉自己这样想时立即停下

| 想法 | 现实 |
| --- | --- |
| "赞助方没回答，我的默认选项够好了。" | 推荐的存在是为了被反应，而不是被采纳。将其记录为假设或返回 `blocked`；绝不要将其折叠为决定。 |
| "我就问赞助方吧，他们最清楚。" | 他们清楚自己的意图。仓库已经做了什么、先前任务尝试过什么——那是你的功课，不是他们的问题。 |
| "我就问他们想要什么外观。" | 未知已知不会回答开放问题；它们会对具体选项做出反应。展示真正的替代方案。 |
| "四条路径，为了全面。" | 差异才是价值，不是数量。填充的路径什么都教不会。 |
| "再问一个问题不会有什么坏处。" | 每轮都消耗赞助方的注意力。当答案不再改变方向时，停止提问。 |
| "地形不熟悉，但采访能把它捞出来。" | 采访只能浮现赞助方知道的内容。不熟悉的 terrain 需要先 `probe`，否则盲点会被洗白成需求。 |

## 输出回收到所属阶段

工作已经形成决定时，返回一个简洁的决策记录：已选择的方向（以及被拒绝的路径和赞助方给出的理由——拒绝编码了品味，下一轮如果不了解就会重新争论）；赞助方意图（用赞助方的词汇）；目标用户和参与者；可观察的旅程和成功指标；Must-haves；非目标和 MVP 边界；假设和开放问题，明确标注；已查阅的证据及其凭据。在返回前，检查每个“已选择”都能追溯到赞助方的明确声明，并检查任何原型是否位于 `brainstorm/prototypes/` 下，未暂存，且产品代码中没有任何东西引用它们。若本轮仍停留在探索，报告仍活着的线索以及哪些证据改变了图景；不要伪造一份决策记录。

对于 `validate-requirements`，此综合不是单独的门：Delivery Orchestrator 将其写入已验证需求工件，该工件在需求门通过前仍需要赞助方的人类确认。
