# Review 因果见证

评审时先从 Task Context 的 Task Operating Envelope 和实际改动表面选择相关见证。它们是构造证据的方法，不是固定审查角色清单；每个类型仍在 coverage ledger 中标明 `reviewed | not-applicable | evidence-gap`。

## 目录

1. Behavior & Boundaries

## Behavior & Boundaries

### 正确性

```text
真实输入或状态 → 实际分支/状态转换 → 错误可观察结果
```

检查边界值、空值、顺序、非法状态、错误传播和需求不匹配；只有 Task Operating Envelope 或实际调用者允许交错时才展开并发。测试通过不能替代自己走路径；测试也可能共享作者的盲区。

### 公共契约

```text
现有调用者依赖的承诺 → 本次形态或语义变化 → 调用者怎样失败
```

覆盖 route、请求/响应、必填/可选字段、错误码、CLI、schema、导出类型、认证方式和迁移窗口。版本号或 changelog 不是迁移路径；纯新增可选字段和兼容默认值通常不是问题项。

### 安全

```text
不可信入口 → 缺少或可绕过的控制 → 敏感操作/数据出口 → 可利用后果
```

没有可达攻击路径就不报漏洞。区分理论加固建议与当前可利用问题；检查身份、授权、注入、敏感信息、令牌/密钥和滥用规模。

### 可靠性

```text
依赖慢、失败或半成功 → 错误传播/状态残留 → 重试、泄漏、重复或级联损害
```

检查 timeout、retry、幂等、补偿、清理、资源释放、部分持久化和 shutdown/recovery。正常路径不能证明失败路径。

### 性能

```text
会增长的维度 × 有来源的真实规模 → 查询、内存、网络或延迟如何越界
```

没有真实规模、热路径和成本计算时，不把“可能慢”写成问题项。用查询计划、复杂度、基准或生产数据支持。

### 对抗与组合

```text
多个单独合理的行为/前提 → 特定顺序、组合或滥用 → 级联失败
```

用于高风险、跨组件、时序和可滥用变化。必须给具体场景，不使用“再想几个极端情况”制造噪声。

## Change Integrity

### 范围与变更卫生

```text
一个 hunk/文件/依赖 → 当前授权源 → 找不到归属或改变了未批准语义
```

区分本轮必要变化、分支历史残留、格式噪声和顺手重构。无授权内容通常应撤回或拆分，不因“顺便更好”留下。

### 明确标准

```text
项目逐字规则 → 当前代码或产物 → 可指认的违反
```

没有原文规则就不能把个人偏好冒充标准。规则与项目现实冲突时说明冲突和改变代价，不静默违反，也不机械服从。影响较小不代表可以忽略：优先级描述后果，是否在当前对象中修复由规则归属、修改成本和处理方式决定。

### 简洁性

```text
新增抽象、选项或间接层 → 它实际隐藏/解决什么 → 删除后承诺是否仍成立
```

通过删除检验才能报告多余复杂度：指出更简单的结构，并证明删掉以后承诺仍成立。正确性、安全、可观察性和显式失败真正购买的复杂度不是问题。

### Cleanliness 与 dead code

```text
未使用、不可达、重复或漂移的代码/注释 → 实际调用与任务承诺 → 删除后是否影响当前行为
```

搜索真实引用和入口后再下结论。新引入或被本次变化遗留的 dead code 属于当前 review；纯既有垃圾只有在本次变化扩散、依赖或掩盖它时才展开。注释只在错误、漂移或缺少代码无法表达的边界时形成问题，不把文案偏好当缺陷。

### 测试与文档资产

```text
本次行为或长期边界 → 现有测试/文档能否让未来修改者看见它 → 缺失怎样让错误安全通过
```

只报告会让真实缺陷逃逸或让维护者误解边界的缺口。完整 TestPlan、执行与覆盖结论属于 `test`；review 只判断本次变化留下的证据和维护资产是否足以支撑前进。

## Architecture & Stewardship

### 架构、形态与品味

```text
当前 hack 或错误概念 → 更诚实的结构 → 已检查的迁移与实现价格
```

最小 diff 可能在错误 schema、模块边界、依赖方向或命名上续命；更大改动也可能只是过度设计。先找到真实模型，再在诚实方案中选择最小者；给出更好形态以前检查它会触及的调用者、数据、迁移和实现价格。

### 项目长期承担

```text
新增公共面、依赖、债务或运维承诺 → 谁长期承担什么成本 → 谁有权接受
```

说不清成本承担者和授权者的担忧只是个人偏好。临时方案需要退出条件、触发器和负责人。

## 对象特定见证

### 文档、设计和计划

代码之外的产物也需要可走通的失败：

- 需求：某个价值或边界不可观察，导致不同实现都能自称完成；
- 设计：缺少 owner、接口、状态、迁移或失败语义，迫使实现者临场决定；
- 计划：故事依赖未确定的结构，顺序无法验证，或把范围外工作静默吸收；
- 测试报告：用例缺失、证据句柄打不开、动作与结论不对应，导致绿色状态无法复核。

### Skill、提示词和长期规则

```text
真实触发输入 → description 是否发现正确能力 → 正文、reference 与相邻规则怎样影响判断 → 可观察行为误差或长期副作用
```

检查触发描述与正文职责冲突、多个 Skill 争抢同一信号、新旧规则互相抵消、临时任务要求被晋升为长期规则、示例或语义锚点外溢，以及复制安装后引用不可达。行为改进必须能在原失败和邻近反例中看见差异；只让一个样本更像预期、没有证明通用原因时，不把措辞偏好报告成改进。

## 问题项记录

```markdown
### [P0-P3] <简短标题>

- 类别：correctness | security | reliability | performance | compatibility | standards | simplicity | shape | scope | stewardship | ...
- 证据状态：confirmed | partial | needs-more-evidence
- Scope relation：introduced | amplified | pre-existing | outside-delivery-scope
- 处理方式：fix-now | follow-up | owner-decision
- 位置：文件与行、接口、测试用例或产物章节
- 触发：具体输入、状态、调用者或扰动
- 路径：实际经过的分支、边界或依赖
- 结果：错误行为或长期成本
- 证据：命令、请求、代码、日志、历史或规则原文
- 建议：最小正确处理方向；结构问题附诚实形态与价格
- 证据边界：仍未确认什么
```

被证据推翻的候选问题不分配优先级、scope relation 和处理方式；只有会影响审查可信度的高信号误报才进入 Finding Audit 的 `overruled` 部分。真实但不在本次落地的问题保持 confirmed/partial，并用 scope relation 与处理方式表达，不弯折 truth。

复现依赖未提供的 schema、配置或外部契约时，明确写出假设。一个在自选 fixture 中通过或失败的实验不能替代真实项目契约；该契约会改变裁决时，把它留作 evidence gap。

没有发现问题也是有效结果，但要说明实际检查的对象、覆盖面、镜头和证据边界；“未发现问题”不等于证明没有问题。
