# 命名与直观性

## 描述

代码应该让人一眼就能看懂。开发者阅读代码时，第一眼的感觉就应该是对的，不需要深入分析，也不需要频繁翻文档。做设计时，用这个维度来判断：你正在敲定的名字和约定，是否通过精确的命名、遵循约定、符合读者预期，把含义表达清楚。

晦涩是复杂度的主要来源。当重要信息被隐藏起来、需要费劲才能发现时，开发者会浪费时间、做出错误假设、引入 bug。好的命名是基础——名字会在读者脑海里形成图像，引导理解。一致性则会放大这种效果，让开发者能把在一个场景中学到的知识迁移到另一个场景。当代码违背预期或使用模棱两可的结构时，读者就不得不放慢速度，去琢磨真正的行为。

设计过程中持续追问：有人能快速读完最终写出的代码并在第一遍就正确理解它吗？还是需要仔细研读才能解码？

## 核心原则

### 1. 选择精确的名字

名字应该让人对所指之物形成清晰的心理图像。所谓精确，就是名字能缩小可能性范围——当有人孤立地看到这个名字时，不需要看文档或使用上下文，就能准确猜出它指什么。

- 像 `count`、`data`、`info`、`status` 这种通用名字通常太含糊
- 名字应当能区分相近概念（例如用 `fileBlock` 和 `diskBlock`，而不是都叫 `block`）
- 布尔值要用能清楚表明 true/false 含义的谓词
- 避免一个名字可能同时指代多种不同事物的情况
- 当很难找到一个精确的名字时，往往说明设计本身就不清晰

### 2. 保持命名一致

一旦为某个用途选定了一个名字，就要在所有地方用它来表示这个用途，绝不做别的用途。一致性降低认知负担——读者学会一个模式，就能复用这份知识。

- 为常见概念建立约定，并在全项目遵循
- 对承担相同角色的变量，在代码库各处使用相同的名字
- 绝不在不同上下文中把一个名字用于不同目的
- 当需要多个相关变量时，加上可区分的前缀（例如 `srcFileBlock`、`dstFileBlock`）
- 约定要足够具体，确保所有使用场景的行为确实相近

### 3. 在代码各处保持一致

除命名外，一致性还体现在结构、风格、模式和不变量上。相似的事情应该以相似的方式做；不同的事情应该看起来不同。

- 在文件中遵循已有的编码风格和结构
- 一致地使用成熟的设计模式和接口
- 统一应用命名约定（camelCase 还是 snake_case、循环变量习惯等）
- 维护不变量，减少针对特例的推理
- 不要总想着"改进"已有约定，除非收益足以抵得上更新所有使用处的工作量

### 4. 让代码符合预期

代码最直观的时候，就是它表现得和读者预期一致的时候。违背约定或常见模式会带来意外和困惑。

- 操作的行为应该与其名字暗示的一致
- 代码结构应该遵循这门语言/框架里的熟悉模式
- 不寻常的行为（例如在构造函数里开线程、main 函数不返回）必须写文档说明
- 在合适的情况下，优先使用标准库的模式和惯用法

### 5. 减少需要的信息

让代码变得直观的最佳方式，就是消除隐藏信息的必要性。设计时让理解行为所需的上下文尽可能少。

- 用抽象隐藏不相关的细节
- 消灭需要读者跟踪额外状态的特例
- 避免会掩盖含义的结构（泛型容器、声明类型与分配类型不匹配等）
- 为阅读方便而设计，而不是为书写方便而设计

### 6. 清晰呈现关键信息

当信息必不可少时，要让它可见。用格式、注释和结构来突出重要内容。

- 用空白分隔逻辑块，让结构可扫读
- 为不明显的行为添加注释（事件处理函数的调用时机、意外的控制流）
- 当代码违背正常预期时，要记录下来
- 确保重要信息位于读者会去看的地方

## 危险信号

### 命名不精确

- **变量名太泛**：`data`、`info`、`result`（在非明显上下文中）、`status`、未经限定的 `count`、用于非坐标概念的 `x`/`y`
- **名字有歧义**：同一个名字用于多个不同目的（例如 `block` 既指 file block 又指 disk block）
- **名字过度具体**：名字暗示的用途比实际更窄（例如一个对任意 range 都有效的方法，参数却叫 `selection`）
- **布尔值含义模糊**：布尔变量没有清楚表明 true/false 代表什么（例如用 `blinkStatus` 而不是 `cursorVisible`）
- **名不副实**：方法或变量的名字暗示了一回事，实际行为却是另一回事

### 命名不一致

- **同一概念，不同名字**：等价的东西在代码库不同地方叫法不同
- **同一名字，不同含义**：把一个常用名字用于概念上不同的事物
- **约定不可预测**：同类结构的命名没有规律
- **破坏已有模式**：新代码无视周围可见的命名约定

### 结构不一致

- **编码风格混杂**：同一代码库里缩进、大括号位置、声明顺序不一致
- **接口实现五花八门**：同一接口的不同实现采用了互不兼容的做法
- **同类操作解法各异**：同一个问题每次都用不同方式解决
- **违背不变量**：本应始终成立的属性，有时却被打破

### 掩盖含义的代码

- **事件驱动回调注册无文档**：handler 的调用上下文不清楚
- **用泛型容器做多值返回**：用 `Pair<Integer, Boolean>` 而不是带明确字段名的具体类型
- **声明类型与分配类型不匹配**：声明为 `List` 却分配 `ArrayList`，隐藏了实际行为
- **令人意外的控制流**：看起来该退出的代码没有退出，或者看起来该停止的代码还在继续
- **不明显的副作用**：操作实际做的事情比表面看起来要多

### 缺失关键信息

- **违背预期却无注释**：不寻常的行为没有在注释里指出
- **排版拥挤**：代码或文档挤在一起，结构难以解析
- **复杂逻辑无注释**：不明显的算法或模式没有解释
- **隐藏假设**：代码依赖的知识在使用处不可见

### 难以命名的实体

- **很难找到简短清晰的名字**：通常说明这个实体没有一个干净的单一职责
- **名字需要冗长的限定词**：说明试图在一个地方做太多事
- **多种合理解读**：名字可能指好几件事，概率差不多

## 问题标签

在设计产物中标记命名与直观性问题时使用以下 tag：

- **[Vague Name]**：名字太泛，无法传达含义（例如 `data`、`info`、`status`）
- **[Ambiguous Name]**：名字用于多个不同目的，或可能指代多种事物
- **[Inconsistent Naming]**：同一概念在代码库中命名不同，或同一名字被用于不同概念
- **[Naming Convention Violation]**：代码破坏了项目已有的命名模式
- **[Style Inconsistency]**：代码与周围文件的格式/结构约定不符
- **[Pattern Inconsistency]**：同类问题用不同方式解决，且未给出理由
- **[Nonobvious Code]**：代码行为无法通过快速阅读看清；重要信息被隐藏
- **[Generic Container Abuse]**：在特定类型能澄清含义的地方，使用 `Pair` 等泛型容器
- **[Violates Expectations]**：代码行为与读者的自然预期不同
- **[Undocumented Surprise]**：不寻常的行为（线程、不返回、副作用等）未在注释中解释
- **[Poor Information Layout]**：排版拥挤或结构不清，导致代码难以扫读
- **[Hard-to-Name Entity]**：难以选出精确名字，说明底层实体的设计不清晰
