# 测试计划审查参考

审查测试计划时常见的红灯——在写决定前，逐条对照计划检查；当出现以下任何一项时，执行该计划很可能也无法建立信心。

- “测试该功能是否工作” —— 没有可验证的断言，没有可证伪之处。
- 面向用户的工作流变更仅由单元测试覆盖，没有端到端验证。
- 公共表面（API）变更没有请求/响应检查。
- 迁移或持久化变更没有前后数据对账。
- 浏览器/UI 工作没有真实交互或视觉证据。
- E2E 用例未声明其执行形式，或者实际上是披着 E2E 标签的 API 调用——它无法证明用户实际看到的东西。
- 一条检查只说明了意图但没有步骤（“启动一个 Plan 并验证恢复”），或把用户输入留空（“输入一个合适的 prompt”）——测试员必须临时编造过程。
- 计划引用的入口点、页面或数据在测试上下文文档（`teamspace/testing-context.md`）中没有来源，所以测试员当场猜测。
- 环境假设被假设掉了，没有说明测试员实际如何运行。
- 覆盖摘要中有一条目既没有 E2E 目标也没有说明省略原因。
