# 实现故事规格

将已批准的需求和设计转化为一份工程师可以据此构建的可执行计划。它刻意地位于设计与代码之间：需求说明想要什么结果，测试计划/测试策略说明必须证明什么，架构/影响/诊断/合约说明系统应该长什么样——而这份故事规格说明工程师实际上需要实现什么。

## 你要做什么

仔细阅读来源成果物，并基于系统实际的样子（而非假设的样子）进行切分。将工作切分为连贯、有序、可独立验证的故事/任务：给每个切片清晰的边界和着陆点，按真实依赖关系排序任务，并把尺寸控制在工程师能逐一完成并验证的范围内。通过路径引用来源成果物以承载细节，然后只拼出本实现特有的判断，而不是把设计全文复制一遍。

## 这份成果物必须达成什么

工程师读完之后应当能够直接开始，无需在现场反推范围或重做设计判断。因此它必须清楚说明（以最适合实现的结构组织）：

- 本次实现的故事——谁/什么角色，想要什么能力或变更，会产生什么结果；
- 启动所需的来源上下文：对来源成果物的简洁指针，以及工程师需要立即了解的少数关键事实；
- 可观察的验收标准，每项都可追溯到一条需求；
- 有序的任务/子任务列表，标注已知的目标模块/文件，并将每个任务链接到它服务的验收标准或技术护栏；
- 设计约束：模块边界、需遵循的模式、必须保持稳定的接口/合约，以及数据/安全/可靠性护栏；
- 对于增强或 bug 修复，列出必须保持工作的现有行为；当公共或跨模块接口发生变更时，明确写出合约；
- 禁区与非目标，防止工程师越界或扩大范围；
- 验证预期：引用测试计划或诊断中的决策标准（给出路径和章节），并只列出工程师在实现期间应添加或运行的聚焦检查——最终验证证据由测试负责人（Test Leader）拥有；
- 计划审查负责人（Plan Review Lead）应关注哪些方面。

给出工程师能够无歧义地启动所需的全部细节；细节密集的地方使用表格或项目列表。如果存在任何会改变实现方向的开放问题，不要把它埋掉——要么明确说出来，要么返回 `blocked` 并附上缺失的设计。

## 格式

需要或有用时遵循 `templates/implementation-story-spec.demo.md`。初始状态设为 `ready_for_plan_review`；不要把你撰写的 Story Spec 标为可开发——那要等计划审查负责人批准。

## 交付前自检

在返回回执之前，逐行确认：

1. 每条验收标准都可观察且可追溯到一条需求；每个任务都命名了它服务的标准或护栏，以及已知的目标模块/文件。
2. 任务顺序反映真实依赖关系，每个故事都可独立验证——不仅仅在全部完成之后。
3. 约束、必须保持工作的行为以及禁区都已声明；任何新依赖、数据迁移、认证变更、公共 API 变更或 UI 设计变更都是显式提醒以供审查，而不是普通任务。
4. 验证预期将工程师的聚焦检查与测试计划/诊断决策标准（以路径和章节引用）分开——后者的最终证据由测试负责人拥有。
5. 没有会改变实现方向的开放问题被埋藏：它在规格中被明确写出，或者计划是 `blocked` 并命名了缺失的设计。
6. 状态是 `ready_for_plan_review`（或按协议 blocked 规则为 `blocked`），frontmatter 与示例一致，且审查重点部分告诉计划审查负责人应关注哪里。
