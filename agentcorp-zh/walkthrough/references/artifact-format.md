# 讲解产物格式

SKILL.md 涵盖为什么；此文件涵盖如何。编写产物前加载。

## 形式

- **HTML（默认）：** 一个自包含的文件——所有 CSS 和 JS 内联，无网络请求，无外部字体或 CDN。以普通 1280–1440px 桌面浏览器设计和验收。决策主线保持纵向阅读；大变更满足下述规则时，同级模块和速查视图使用 tabs。除非发起人明确要求移动端，不做手机截图、断点调优、触摸优化或像素级响应式 QA。代码放入 `<pre>` 块（如果使用 styled `<div>`，必须设置 `white-space: pre-wrap`）。图表为内联 HTML/SVG 示例数据——简化的 UI 示意图、系统草图、前后对比；不用 ASCII 艺术。对关键概念、定义和边缘用例使用高亮框。
- **Markdown（`output_format: md`）：** 相同的五节；图表用 Mermaid（交付前验证语法）；测验选项和答案放在折叠的 `<details>` 块中，使答案在打开前保持隐藏。
- 命名：任务根下 `walkthrough/<slug>.html`，或独立运行时 `teamspace/walkthroughs/<YYYYMMDD>-<slug>.html`。slug 命名变更，不是分支。
- **信息分层：** 主阅读路径承载发起人的决策模型，不是源材料的无损渲染。需要完整覆盖时，把完整内容放进明确标注的工程附录或渐进式展开；附录负责保留可得性，不能成为上文重复源文档的理由。

## 渐进式展开与视觉注意

- **首屏：** 在普通 1280×800 或 1440×900 桌面 viewport 中，展示产物名称、一句话用途、关键不变/关键变化，并至少露出“变化雷达”或下一层导航的开头。标题最多占两条视觉行，不能让大字号吃掉整屏。最小架构图放得下时进入首屏，否则紧跟变化雷达。变化雷达只收行为修正、容易意外、兼容变化和发布风险，不用实现细节占满首屏。
- **知识单元：** 关键概念默认使用关闭的 `<details>`。summary 必须同时包含概念名、用途和一句话结论或后果；读者不展开也能判断是否与自己相关。信息密集的产物可以展示很多关闭状态的 summary 供扫描，但默认展开的知识单元不得超过 5–8 个；概览之后优先全部关闭。
- **展开正文：** 展开后才展示机制、例子、取舍、失败方式和受影响行为。代码、SQL、DDL、完整接口和穷举表格进入概念内部的第二层 `<details>` 证据折叠。
- **Badge 与文字高亮：** Badge 只说明影响类型，不等于高亮。用 `<mark>` 等文字背景高亮指出必须记住的具体短句。每个有实际影响的展开概念，正文通常应有 1–3 处高亮，不能只出现在 summary 或 Badge。
- **高亮预算：** 只有会改变决策、运行结果、兼容预期或发布动作的事实才高亮。只高亮短语或一句话，不染整段；不能只靠颜色传达含义。

## 大变更的导航模型

写 HTML 前先做覆盖地图，把内容分成五类：**决策主线**、**同级模块/操作**、**跨模块流程**、**速查领域**、**证据**。按这张地图选择导航：

- **纵向主线：** 关键不变/关键变化、端到端例子、跨模块不变量、人的裁定、风险和测验保持顺序阅读。这些内容依赖前后关系，不能藏进 tabs。
- **Module Atlas tabs：** 变更包含三个及以上同级模块或操作时，建立一组 tabs，让读者直接跳到关心的专题。适用时，每个 tab 使用同一份紧凑合同：用途、入口/调用者、鉴权、状态变化、产生的事件/输出、兼容影响、一个决定性例子。不适用的字段直接省略，不机械填格子。
- **Reference Desk tabs：** 常量、权限矩阵、状态表、schema/DDL、routes/callers、迁移条件属于速查材料。每个足够大的速查领域放一个 tab，不把所有大表连续堆进叙事正文。
- **Tab 内仍要控制深度：** tab 不是全部展开的许可证。模块 tab 内继续用关闭的概念 `<details>`，代码、SQL 和完整契约再放第二层证据折叠。
- **不用 tabs 的情况：** 只有一两个模块时使用普通章节。单一时间流程、前置链、前后对比或同一个决策不能拆到不同 tabs。
- **交互底线：** tab 按钮必须显示选中状态、支持键盘激活，并保持 panel 尺寸稳定。在一个桌面 viewport 验证切换与深链接。除非明确要求，移动端不在验收范围。

因此，常见的大型 Architecture walkthrough 结构是：**常显决策面 → Module Atlas tabs → 跨模块流程 → Reference Desk tabs → 裁定/风险 → 测验**。这是教学界面，不是 Architecture 章节树的复制品。

## Source / Current / Target 新鲜度

- 标题附近放紧凑 revision stamp：准确 Source revision、准确 Current revision；只有批准设计仍未落地时才显示 Target。能拿到 commit 或 dirty-state 标记时，不使用 `working tree based on` 这类模糊说法。
- 两个时间层不同时，结论显式写明 `Source 曾经...`、`Current 现在...`、`Target 要求...`。不能让 `current` 同时指被检查代码和已批准 Architecture。
- 交付前重新执行支撑每条 `pending`、`not implemented`、兼容 bug 或公开接口结论的路由/调用者/测试搜索。Current 移动后，变化雷达、模块 tabs、风险、路由表和测验必须一起更新。
- Current 已经符合 Target 时，合并两个时间层并删除过时的迁移叙述。历史分歧放进简短证据折叠，不占现在时的决策面。

## 章节契约

1. **背景与稳定合同** —— 先教已经存在的世界，再列出变更后仍成立的 5–8 条关键合同。从新人进入此子系统真正需要的地方开始，但不复述整个系统。读者必须先知道哪些公开行为、所有权边界、工作流或安全合同刻意保持不变，才能评价变化。
2. **直觉** —— 变更的本质，任何代码之前。说明变更解决的问题和解决方案的形状，用通俗语言，然后追踪**一个端到端的玩具示例**：一个具体输入、旧代码做了什么、新代码做了什么、以及为什么差异才是要点。更清晰时优先用小图加示例数据，而非 prose。
3. **关键变化，作为故事** —— 只选真正改变心智模型的 3–7 项变化，讲清为什么存在、带来什么、周围什么仍不变。不按文件顺序、源文档标题顺序或字段清单展开。大变更可以先引入 Module Atlas，把同级专题放入 tabs；跨模块故事仍留在 tabs 外。代码变更中的每个 hunk 引用真实 `path:line`；Architecture/Plan/Review walkthrough 引用源产物章节，只在契约承载决策时展示完整内容。
4. **行为变更和风险** —— 现在行为不同的紧凑、完整记录：新/变更的输出、错误行为、性能或兼容性影响，以及——最有价值的——与 diff 不显示的已有代码路径的交互。没有时各类写“none”。覆盖诚实：范围内的任何未解释事项，都在此处列为明确未覆盖并附原因。
5. **测验** —— 5-8 道选择题，每道 3-4 个选项，答案在读者提交前隐藏（HTML：点击揭示；md：`<details>`）。每个选项的揭示说明为什么对或错。难度：足以要求真正理解，无陷阱，无 trivia。

## Architecture 与 Plan 模式

当来源是 Architecture、Requirements、Implementation Plan 或 Review 产物时，页面默认可见的决策面按以下顺序组织：

1. **关键不变** —— 保持稳定的契约和用户效果。
2. **关键变化** —— 只列真正改变心智模型的结构或行为变化。
3. **一个端到端例子** —— 用真实示例值展示前后差异。
4. **为什么是这个形状** —— 被拒绝方案及其具体代价。
5. **人的裁定与风险** —— 已接受取舍、未决事项和发布后果。

按内容职责转换 Architecture，不按标题逐节搬运：

| Architecture 来源 | Walkthrough 载体 |
| --- | --- |
| Decision Summary + Unchanged Contracts | 常显决策面 |
| 目标组件与各自拥有的接口 | Module Atlas tabs |
| Changed flows 与跨组件不变量 | 纵向跨模块故事 |
| Schema、API、常量、迁移条件 | Reference Desk tabs |
| 兼容性、风险、non-goal、开放问题 | 裁定/风险与测验 |
| Source references 与研究证据 | 嵌套证据折叠 |

跨领域 Architecture 在决策面之后使用 Module Atlas tabs，并另设 Reference Desk tabs。Schema、完整接口、全部迁移条件、模块清单、证据引用和源文档正文放入渐进式展开的速查视图或工程附录。不能用“信息量相同”为由，在主阅读路径重复源文档。

## 测验机制

- 瞄准意外表面：依赖已有代码路径的行为、边缘用例和失败模式、“追踪此输入”问题、“为什么选择形状 A 而不是 B”。至少一道题必须要求**背景**节的知识——这就是你检测跳过背景的读者的方式。
- 禁止：可通过扫 diff 回答的问题——文件名、行数、符号拼写、任何计数。
- 关卡：如 SKILL.md 的“测验关卡”（满分；重读后答对的变体清除 miss；`approved`/`skipped` 在 `task.md` 的 Gate History 中）。独立运行时，产物以最终“Gate outcome: approved <date> / skipped by sponsor <date>”行结束。

## 交付前自检——有任何一项命中，回去重写

- 读者尚未被教会周围系统之前，第一个代码块就出现了。
- 第 3 节的顺序镜像了 `git diff --stat` 的输出，而非想法的结构。
- 页面默认加载时有超过 5–8 个知识单元展开或拥有同等视觉权重。
- 折叠 summary 只有名词或标题，没有说明用途和一句话结论。
- 代码已经折叠，但完整概念解释仍连续铺开。
- 有实际影响的展开概念没有精确文字高亮，或者高亮只存在于 Badge/summary。
- 高亮覆盖整段或密度过高，读者无法分辨哪些内容会改变决策或结果。
- 可见页面镜像了源材料标题，或给每个字段、方法同等显著性。
- 三个及以上同级模块仍被渲染成一条很长的连续页面，尽管读者需要横向查找；或者 tabs 被用来拆散一个必须顺序理解的故事。
- 各模块 tab 使用互不相关的布局，缺少可对比的合同维度。
- 没有明确移动端要求，却执行了手机截图、断点调优或触摸专项打磨。
- 首屏没有同时讲清关键不变和关键变化。
- 在 1280×800 或 1440×900 下，过大的标题/metadata 把变化雷达和下一层导航完全推到首屏之外。
- Source、Current、Target 混写、revision 模糊，或分支移动后仍保留未经复核的 `pending`/`current bug` 结论。
- 附录虽然存在，但主叙事仍重复了其中大部分内容，而不是教学决策。
- 展示的 hunk 没有 `path:line` 锚点，或锚点是估算的而非来自 diff。
- 你知道的行为变更既未出现在第 4 节，也未出现在未覆盖列表中。
- 任何测验题无需阅读产物即可作答（trivia），或没有一道题要求背景节。
- 产物依赖网络资源、外部字体或赞助人可能没有的其他渲染器。
- 讲解被直接交付到终端行内，而非持久化到文件。
- 任何位置出现了未解释的任务内代码（T-xx、F-xx、ST、内部产物名称）。
