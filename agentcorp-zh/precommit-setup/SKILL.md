# precommit-setup

这是 AgentCorp 的一项可复用支撑能力，既不是交付阶段，也不是带自己关卡的角色。**你要回答的问题：这个仓库真正会一直用下去的提交时护栏，是哪些？** Hook 设置会朝两个相反方向死掉：太重（把完整测试套件塞进 pre-commit → 不出一周人人都 `--no-verify` 提交；这道护栏反倒把人训练成绕开护栏）和太空（一份看着像样、却从没跑过的 YAML → 第一次真提交就崩）。两者的病根是同一个——把配置文件、而不是一次亲眼看过的 hook 运行，当成了交付物。

## 铁律

```
没跑过的 hook 等于没有 hook——跑它，否则如实报 UNVERIFIED。
```

## 工作流

1. 选工具之前先摸底：现有的 hook 系统（`.pre-commit-config.yaml`、`lefthook.yml`、`.husky/`、`.git/hooks/`、`package.json` 脚本）、现有的质量命令（`Makefile`、`justfile`、`pyproject.toml`、CI 配置）、现有的提交规则（`AGENTS.md`、`CLAUDE.md`、`CONTRIBUTING.md`）。
2. 现有的 hook 工具永远优先——没有明确要求就绝不迁移。若一个都没有，优先选语言中立的 `pre-commit` 框架；Node 打头的仓库用 Husky；多语言 monorepo 或已经在用 Lefthook 的仓库用 Lefthook。
3. 先在延迟预算内配好快速检查，再是约束，最后把 AI 审查作为一条慢速的可选路径。
4. 把 hook 跑起来，并记下确切的安装/验证命令。

## 快速默认检查

确定、本地、快；优先做暂存文件级检查。**预算：一次典型提交里，整个默认 hook 控制在约 5 秒内；任何单项检查经常超过约 10 秒，就挪到可选路径或 pre-push。** 合适的候选（挑仓库已经支持的——这份清单是举例，不是配额）：空白符/文件结尾/合并标记/大文件的守卫，仓库自带的格式化工具和 linter，只有装得进预算才纳入的类型检查或定向测试，以及在工具已有或极易加上时的密钥扫描。完整测试套件、浏览器测试、装依赖、网络调用和 AI 审查，绝不作为每次提交的强制项——完整套件属于 CI 和测试阶段。

## 提交约束

本地护栏，一看就懂，且故意留有绕过余地。常见的有：拦住没解决的冲突标记；拦住明显的密钥或凭证文件；拦住只改了生成产物却没动源文件的提交（仓库有这条惯例时）；拦住 `teamspace/` 之类 AgentCorp 运行时交付物，除非用户就是想提交它们；仅在已有约定或用户提出时，才在 `commit-msg` 里强制提交信息风格。每条失败信息都要讲清：什么没过、为什么要紧、以及重跑的命令。

## 可选的 AI 提交审查

绝不要把 Codex/Claude 审查接成无条件的 pre-commit 步骤。触发方式可选：**手动命令**（`make ai-commit-review`）、**环境开关**（`AI_COMMIT_REVIEW=1`——默认关，配一个手动命令）、**按路径/风险**（认证、支付、迁移、公共 API——仅在用户点明区域时），或 **pre-push**。它检查暂存的 diff，回一个简洁的通过/不通过，并把详细笔记写到一个被 gitignore 的路径，如 `.agentcorp/commit-review/`——忽略条目你自己加上；笔记里含 diff 片段，绝不能进历史。失败时的行为：默认只告警；要阻断得显式开启，且永远配一个绕过口子（`SKIP=ai-commit-review git commit ...` 或工具自带的跳过方式）。AI 提交审查 hook 绝不顶替评审阶段的代码审查。

## 配置模式

动手写任何 hook 配置之前，先读 `references/config-patterns.md` 并从里面抄——即拷即用的 pre-commit / Husky / Lefthook 模板、AI 审查的守卫脚本，还有失败信息的写法。有两条规矩跟着每份模板走：`repo: local` 只用于快速的仓库命令，绝不写 `make test`；只用用户手上真有的 CLI——写参数前先用 `codex --help` / `claude --help` / 看项目脚本确认过。

## 验证

在当前环境里实打实地跑：安装器（`pre-commit install` 或等价命令）；完整 hook 跑一遍（`pre-commit run --all-files`）——若超了预算，就跑一次定向的暂存文件检查，并说清跳过了什么、为什么；CLI 存在时，在跳过模式和开启模式下各跑一遍 AI 审查命令；`git status --short` 确认只有该动的文件动了。如果工具在这里装不上或跑不了（没网、没二进制、沙箱 shell），就报 **UNVERIFIED**，并附上用户必须自己跑的确切命令。报告要含：装了哪个工具、增/改了哪些文件、跑了哪些命令及结果、还剩哪些慢速检查、绕过/开启的说明，以及状态（verified / partial / UNVERIFIED）。

## 危险信号——发现自己这么想时，停

| 想法 | 现实 |
| --- | --- |
| "配置是标准 YAML，肯定能跑。" | 没执行过的 hook 会栽在路径、缺二进制和拼写上。 |
| "`pre-commit` 更好，我把现有的换掉。" | 现有工具优先。迁移得有明确要求。 |
| "hook 里放 `make test` 能逮住更多 bug。" | 它也会让每次提交都变慢，直到人们干脆绕过 hook。 |
| "`run --all-files` 我跳过吧，八成很慢。" | "八成"不是测量。先跑一遍计时；只有拿预算当证据才跳过，并说明。 |
| "笔记目录早晚会被忽略掉的。" | 除了你没人会去忽略它——下次提交就把 diff 片段带进历史了。 |

## 输出

Hook 配置、脚本和失败信息，一律随目标仓库的语言和惯例；任何 AgentCorp 报告类交付物则保持 zh-CN 的人类正文，标识符和命令原样照录。绝不要把任务交付物写进技能目录；`teamspace/` 留在本地（未被追踪就加进 `.git/info/exclude`；绝不暂存、提交或推送）。
