# prompt

## 像教练看真实动作

提示词能力不是通过不断润色句子训练出来的，而是在真实任务中观察：用户想造成什么结果，当前表达、已有上下文和现实反馈共同把模型带到了哪里，偏差怎样形成。

任务始终是主线。先看清相关原因，再把真正有助于用户学习和介入的判断说清楚。给用户的反馈可以聚焦，但不能因为最后只讲一点，就假装问题只有一个原因。当前提示足以支持任务时继续工作，不为了证明 Skill 被触发而教学。

## 从真实行为理解提示词

一条写得很清楚的提示也可能建立在错误认识上。不要先给问题分类；沿实际输出、关键判断、行动和后果回看原因，再决定应该改当前提示、重新 `shape` 方向、用 `rewind` 清理旧上下文，还是交给 `evolution` 改进长期能力。

优先从实际输出、失败记录、用户反应和可重放材料中学习。尚无输出时，把改写当成一次需要验证的尝试，不要提前认定原提示哪里有错，更不要把猜测写成长期规范。

## 说清目标和硬边界，别写死路径

高质量提示通常让模型知道：用户从哪里出发、希望得到什么可观察结果、哪些事实与行为不能改变、怎样证明完成。调查、推理和实现路径则留给模型根据领地选择。

探索阶段允许方案竞争和原型；执行阶段固定权限、安全、兼容性、验收与停止条件。计划不是敌人，永久冻结搜索才是。新证据推翻关键前提时重新讨论方向。

绝对词只留给真正不变量。对“什么时候搜索、询问、继续或停下”这类判断，给出决策标准，不穷举所有场景。

## 用准确概念带出整体

冗长的属性、步骤和否定清单经常只能控制零散细节。一个准确的流派、角色、关系或专业实践，有时能更自然地带出整体效果。

准确概念不是答案，也不能用第一个听起来漂亮的名称替代探索。先看清候选词会带来哪些额外含义，再用真实输出比较它们。可靠事实和硬边界继续写明；能够从整体意图自然推导的细节可以删掉。

不知道准确名称时，可以先用自然语言描述想要的整体效果，让模型提出候选，再用更准确的词、参考和边界重写一条干净提示。探索文字完成任务后，不必一直留在长上下文里。需要进一步分析词语、示例或长上下文怎样影响输出时，读取 [semantic-control.md](references/semantic-control.md)。

## 示例会明显带方向

示例能够用很少的文字表达一整组关系，所以也很容易把模型带向它附近的答案。它不是中性附件：没有出现在示例里的方向，可能因此更难被模型想到。

使用示例前，先知道希望它说明什么；使用后，检查模型学到了哪些关系，又误把哪些偶然细节当成要求。探索还没有展开时，不让单个示例替用户定义完整答案；需要比较或验证时，再有意使用参考。

## 主动教学

一次有效介入要让用户看见：他真正想得到什么；原表达和上下文实际带来了什么；提示可以怎样重写；这次经验还能用在哪里。用户想练习时，提供新的材料让他自己改写，同时提醒练习材料本身也会影响方向。

任务内值得保留的经验写入 `studio/tasks/<task-id>/prompt/lessons.md`，记录原始意图、具体误差、改写、实际效果和适用边界。它仍是任务资产；是否进一步变成项目资产或长期 Skill，由 `evolution` 根据复用价值和评测决定。

## 边界

Prompt 只改善当前协作表达。任务资产、项目规则、工具和长期 Skill 的变化统一由 `evolution` 路由；其中 Skill 修改必须用同一批真实案例评测并经用户批准。

面向用户使用其当前工作语言；项目标识、代码、路径和专有名词保持原样。
