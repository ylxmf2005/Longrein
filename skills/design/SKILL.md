# design

## 让承重决定在变昂贵以前显形

把已经形成的方向带进真实系统，使数据属于谁、边界在哪里、状态怎样变化、接口承诺什么、失败后怎样处理和先从哪里施工，在它们变昂贵以前显形。不要让实现者写到一半才无意间发明架构，也不要用很长的待办清单掩盖设计还没成立。

设计不是一次写定的蓝图，而是当前证据支持的系统模型。架构说明系统为什么应变成这种形状，计划让这个形状通过可观察增量接受现实检验。两者默认放在同一份权威产物中；新证据推翻模型时，一起修订结构、顺序和受影响的承诺。

## 先读清现有系统

当前 Session 已宣布 Active Task 时，先读取其绝对路径对应的 `task.md`、Task Operating Envelope、Git 意图和 Artifact Map 输入，再核验真实代码基线。随后打开改动会触及的模块、调用者、接口、数据定义、测试、历史决定和必要运行证据。任务描述、既有计划和文件名只是入口，不能替代代码阅读；任何会影响设计的判断都要能回到实际来源。

把材料区分为三种状态：

- **现状事实**：已从代码、契约或运行结果确认；
- **设计决定**：本轮明确选择并将约束实现；
- **假设或开放问题**：尚未验证，不能伪装成下游事实。

如果真正要实现的结果、范围或可接受代价仍未稳定，回到 `shape`。如果代码现实推翻了已经形成的方向，带着证据重新塑形，不在错误目标内设计一套更完整的系统。

Task Operating Envelope 决定需要设计到什么现实包络：正常负载、可信并发、平台重试、故障恢复、公共调用者或攻击面只有在项目中可达时才成为设计约束。不要为有证据支持的 explicit exclusion 增加锁、重试、扩展点或通用框架；项目现实推翻包络时先提出 Context 修订，不把新约束静默吸收到设计 scope。

## 让决定先取得资格

并非所有未知都应该靠继续写文档解决。结构取舍如果依赖尚未观察的库行为、性能边界、数据分布、迁移结果、用户反应或失败模式，先寻找能让现实回答它的办法：继续走调用链、运行探针、制作可丢弃原型、比较小型实现或测量真实数据。实验的代价应小于它准备消除的不确定性。

最先成功的方案不自动成为架构，最先失败的方案也不自动证明相反路线正确。把结果放回候选解释、项目约束和用户价值中比较。证据仍不足时，保留开放问题和下一次取得反馈的入口，不用完整语气制造虚假确定性。

用户不需要裁决局部代码组织，但需要看见会改变产品行为、长期成本、兼容承诺或责任归属的选择。把这些介入点放在设计前部，给出事实、真实代价和明确推荐，不让用户从完整技术文档里自己寻找决定。

## 纯诊断在因果链闭合处停下

用户只要求诊断时，从具体触发、输入与状态出发，走通实际调用链、状态转换和失败传播，区分已确认根因、促成条件、影响范围与仍待验证的解释。结论必须能解释可观察故障并回到代码、日志、数据或复现证据；时间上相关、看起来可疑或只能解释部分症状的因素不冒充根因。

纯诊断交付事实、因果链、证据边界和修复将触及的结构，不自动修改代码，也不把候选修复写成已决定方案。用户随后授权修复时交给 `dev`；修复前仍有承重结构需要决定时，继续完成设计。

## 把会影响实现的结构定清楚

只处理会改变实现、兼容性或长期成本的结构问题。非平凡任务读取 [架构与实现路线](references/architecture-and-implementation-route.md)，根据真实影响选择需要说明的内容，例如所有权、模块边界、数据和状态、接口、错误处理、权限、恢复、迁移、可观察性和测试入口。不要为了显得完整而把每一项都写满。

保留现有行为时，明确写出不变契约。改变行为时，说明旧形态、新形态、受影响消费者和过渡方式。数据和公共接口需要精确到字段、键、默认值、读写所有者、错误形态与兼容窗口；抽象口号不能替代契约。

设计范围要与根因相称：小问题不借机扩建系统，错误的基础模型也不能只靠小补丁维持。需要比较路线时，只展开真正会改变取舍的候选，比较实际成本后给出明确推荐。超出本轮目标的重构、迁移或平台能力作为独立提案，不偷偷塞进当前任务，也不为了避免讨论而降级成临时补丁。

## 用实现增量继续检验设计

当前证据足以开始施工后，按真实依赖切成可逐步落地的增量。增量既交付价值，也继续检验设计；优先让高风险假设、迁移边界和跨模块契约在仍然便宜、可回退时接触真实反馈。每个增量说明：

- 它产生的可观察结果；
- 主要落点和涉及的边界；
- 依赖哪些前置决定或数据变化；
- 必须保持的契约与禁止触碰的区域；
- 实现者应在本步运行的聚焦检查，以及交给 `test` 的外部证明入口。

让每个增量完成后系统仍可理解、验证并在需要时回退。检查结果不仅决定这一步是否完成，也决定后续设计是否仍可信。不要把所有价值压到最后一步，也不要预定局部函数名和机械编辑。

计划过程中发现新的结构缺口，直接回到设计部分补齐；不要在任务列表里埋一个“实现时决定”。会改变方向的问题交给用户或 `shape`，不会改变结构的局部选择留给实现者。

## 产物随任务缩放

简单诊断或无结构争议的任务可以在对话中给出结论，不制造文件。当前 Session 有 Active Task 且需要保存或交接时使用：

```text
<task-workspace>/design/
├── design.md
├── diagnosis.md  # 纯诊断或设计依赖独立根因记录时创建
└── evidence/     # 只有设计依赖运行、探针或复现证据时创建
```

`design.md` 承载当前模型、关键决定、不变契约、精确接口或数据形态、影响与迁移、实现增量、验证入口、非目标和开放问题。`diagnosis.md` 承载触发、因果链、根因证据、影响范围、反证和未知；纯诊断不必创建 `design.md`。存在当前 Active Task 时标明相关 revision。证据与调查记录放在引用或 `evidence/` 中，不复制进正文。

产物落盘后更新当前 Active Task 的 Current Work 和 Artifact Map，再告诉用户绝对路径，并说明哪些决定已经有证据、哪些仍需裁决。没有 Active Task 时，只写用户指定的路径或在对话中交付，不创建任务目录。

## 允许新证据改写设计

设计是目前最可靠的结构说明，不是永久权威。实现、`review` 或 `test` 发现新事实时，修订受影响的决定、计划和验证入口，并明确哪些旧内容失效。方向改变时回到 `shape`；旧设计已经持续误导后续工作时使用 `rewind`。

`design` 不把产品实现当成交付物，不给自己作独立批准，也不拥有最终测试结论。为了取得设计证据，可以运行探针或制作明确可丢弃的实验；它们不能被悄悄包装成正式实现。完成时，实现者无需猜测架构、范围、契约或下一段施工路线，用户看得见真正需要裁决的取舍，所有人也知道什么证据会迫使当前设计改变。
