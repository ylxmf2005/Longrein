# 抽象层

## Description

做设计时，用这个维度来判断你正在提出的结构在架构各层之间隔离关注点的能力。设计良好的系统里，每一层都与相邻层提供明显不同的抽象——沿着一次操作的方法调用链路逐层深入，应该能看到抽象层次的变化。如果相邻层暴露的抽象过于相似，说明职责边界模糊、设计过浅，增加了接口复杂度，却没有带来相应的功能收益。

核心关注点包括：
- **层间隔离**：每一层是否提供了真正有区分度的抽象
- **透传方法/变量**：未经转换就穿越多层的方法或变量
- **复杂度归属**：复杂度是被推给调用方，还是被收进模块内部

## Core Principles

### 层不同，抽象也不同

系统中每一层都应该与上下相邻层提供明显不同的抽象。如果相邻层的抽象过于相似，那这一层很可能"名不副实"——它增加了结构开销，却没有带来足够的功能价值。

**示例**：文件系统的各层提供了真正不同的抽象：变长字节数组（文件）→ 固定大小的缓存块 → 设备驱动的块 I/O。每一层都在改变用户看待数据的方式。

### 复杂度下沉，不要上浮

当复杂度不可避免时，模块应当内部消化，而不是暴露给用户。因为模块通常是多用户、少开发者，让开发者一次性处理好复杂度，远比逼着每个使用者各自去应对更划算。接口简洁优先于实现简洁。

**反模式**：把本可以在内部处理掉的场景暴露成配置参数或抛异常，逼着每个调用方或运维人员去重复解决同一个问题。

### 消除透传方法

透传方法（直接以相似签名委托给另一个方法的方法）意味着职责边界模糊。它们让类变得更"浅"：接口变复杂了，功能却没增加，还在层与层之间制造了不必要的依赖。

**解决思路**：
- 直接把底层类暴露给调用方
- 重新分配功能，消除跨类调用
- 如果拆不清楚，就把类合并

### 避免透传变量

那些在漫长方法链中被一路传递、中间层根本用不到的变量，会逼着每一层都去关心与其无关的事。这种把知识铺散到各层的做法，会让后续演进变得困难。

**解决思路**：
- 用一个 context 对象存放全局状态，在主要对象里持有它的引用
- 找现有共享对象来捎带这些信息
- 避免全局变量（它们会阻止多实例化，还会带来测试问题）

### 让内部实现与外部接口的抽象拉开差距

类的接口应当与其内部实现提供不同的抽象。如果接口和实现用了同一套表示方式，那这个类大概率是浅的。

**示例**：一个文本类即使内部按行存储文本，对外也应该暴露面向字符的接口（在任意位置插入/删除），而不是面向行的接口（整行 get/put）。这个抽象差距本身就是有价值的功能。

## Red Flags

在你正在提出的设计里、以及你为设计而阅读的代码里，留意以下模式：

### Pass-Through Methods
- 除了以相同或极为相似的签名调用另一个方法外，什么都不做
- 没有转换、没有附加逻辑的一长串委托链
- 大部分 public 方法都是简单转发器的类
- 例外：根据参数选择调用哪个方法的 dispatcher 是合理的

### Pass-Through Variables
- 在多个方法签名里被传递，却只在最深层被使用的变量
- 参数列表很长，其中大部分只是被继续往下传
- 必须被串过整个调用链的全局状态

### Similar Abstractions Across Layers
- 接口签名与实现思路高度雷同
- 外部 API 暴露了内部表示细节
- wrapper 或 decorator 相对于接口体量只增加了微不足道的功能

### Complexity Pushed Upward
- 模块本可以内部拍板的决策，却暴露成配置参数
- 模块本可以兜住的异常，却抛出去
- API 逼着调用方做多步协调（split/join、setup/teardown）
- 简单操作却要求用户了解内部结构

### Decorator Overuse
- 为了小功能单独搞个 decorator，其实完全可以并到基类里
- 多个浅 decorator，每个只加一点点功能
- 把所有人都需要的通用功能也做成 decorator

### Interface Duplication Without Value
- 多个签名相似、却没有提供区分功能的方法
- 层与层之间照搬下层 API，没有任何转换
- 连接各层的胶水代码，没有带来任何抽象提升

## Issue Tags

在设计产物中标记分层问题时使用以下 tag：

- **[Pass-Through Method]**：方法直接委托给另一个方法，没有增加功能
- **[Pass-Through Variable]**：变量被串过根本用不到它的方法
- **[Similar Layer Abstractions]**：相邻层暴露的抽象过于相似
- **[Complexity Pushed Up]**：模块把复杂度推给调用方去处理
- **[Shallow Decorator]**：Decorator 相对于接口体量只增加了微不足道的功能
- **[Interface Matches Implementation]**：外部 API 直接镜像了内部表示
