# 测试计划（TestPlan）

在实施开始前确定验证策略。目的是在有代码之前和变更仍然廉价时，明确说明如何证明本次任务是正确的——让实现者知道什么必须保持可测试性，让测试员知道如何证明高风险行为。

## 你反对的敌人

漏掉一个风险，和在错误的地方验证，是两个敌人。覆盖必须跟随风险：集中精力在真正重要的关键路径、边界、失败模式和回归上，而不是均匀分散。同时，让计划证明行为，而不是某种内部实现风格——排除那些会随实现细节漂移的脆弱断言，否则重构期间测试会批量出现假阳性。

还有第三个敌人：能写“测什么”但不能写“怎么测”的计划。一条只有意图没有步骤的检查，迫使测试员当场编造操作——而编造出来的操作不一定等于你评估过的风险。因此该产物的通过线是**可逐字执行**：测试员可以运行所写内容，无需编造任何步骤。

## 工作顺序

1. **阅读输入** —— 经确认的需求（或诊断标准）、约束、环境备注、现有测试产物。
2. **检查上下文** —— 读取 `teamspace/testing-context.md`；如果它不存在，或没有覆盖本次任务需要测试的表面，先按照 `references/testing-context.md` 第 0-5 步探索和填补缺口，然后继续。
3. **排名风险** —— 设定 Must-Haves、禁止区域、P0 关卡和执行顺序；对于缺陷类任务，原始失败输入以逐字形式作为显式检查出现，因此修复是建立在实际失败的输入之上的，而不仅仅是代理样本。
4. **编写三份执行手册** —— 具体化标准如下。E2E 手册引用的入口点、页面和控制文本，必须追溯到**实际走访过**的页面地图中的条目；任何来源仅是代码推断的页面步骤，要么送回探索和实地验证，要么在该流程中明确标记“页面入口未验证”并列出在整体策略的开放问题中——验证绝不能静默留给测试员。
5. **指定结果报告** —— 每份说明书都声明测试员在每次检查后必须写下的内容：确切动作/输入、确切请求/响应（如相关）、观察表面、证据路径、清理和证据限制。
6. **编写整体策略** —— 覆盖摘要将每个需求映射到检查 ID 和其所在的文件。
7. **交付前自检** —— 每个 AC 都有所有者；E2E 执行形式明确；三份手册通过“可逐字执行”标准；缺陷类任务携带原始失败输入作为检查；结果记录期望明确；环境忠实描述；每个省略和缺口都有书面原因。

## 先上下文，再计划

当“怎么测”不够具体时，根因几乎总是缺少项目运行上下文：系统在哪里被进入、如何登录、页面长什么样、测试数据遵循什么约定。这些不应该在每个案例中当场猜测；它们位于项目级测试上下文文档 `teamspace/testing-context.md` 中，跨任务复用，增量维护。

开始前读取它。如果文档不存在，或没有覆盖本次任务需要测试的表面（例如一个新页面、一个新接口），先按照 `references/testing-context.md` 中的探索指南填补缺口，然后开始规划。当计划引用入口点、页面或数据时，应该追溯到上下文文档中的来源；本任务特有的补充（特意创建的数据、临时配置）放入对应的执行手册，而不是回灌到项目级文档——那个文档只收集跨任务稳定的事实。

## 产物形状：一组文件，不是一个文件

测试计划是一组文件——整体策略加三份执行手册——写入持有指派 `output_path` 的 `test/` 目录：

- `test/test-plan.md` —— 整体策略：风险排名和执行顺序、Must-Haves、禁止区域、覆盖映射、环境、指派、残余风险。面向人类关卡、测试计划审查员和测试负责人的全局视图；不要堆积用例细节。
- `test/api-test-plan.md` —— API 契约测试员的执行手册：契约检查和数据/迁移验证。
- `test/e2e-test-plan.md` —— E2E 测试员的执行手册：完整用户流程，逐步编写。
- `test/regression-test-plan.md` —— 回归测试员的执行手册：爆炸半径、要运行的现有套件、相邻检查。

当某个手册对本任务确实在范围之外时（例如纯后端任务没有用户流程），在整体策略中说明省略原因；不要创建空文件。每个文件的章节形状遵循 `references/templates/` 下对应的演示文件。

## 三份手册各自的具体化程度

判断具体化程度只有一个标准：**指定测试员手持这份说明书和测试上下文文档，无需编造任何步骤即可开始运行。**

- **API 手册** —— 每条检查逐字给出可直接执行的请求或 SQL（方法、路径、参数、请求体）、预期状态、响应形态和断言、它需要的先决数据，以及如何处理失败（停止、标记阻塞或继续）。"调用列表接口并验证返回"不通过；通过的形式是 curl 或 SQL 就在手册里。
- **E2E 手册** —— 首先声明执行形式（默认浏览器操作为主要证据，见下一节），然后将每个流程写为编号步骤：此步骤在哪个页面上、采取什么动作；凡是用户输入内容的地方（搜索词、表单值，尤其是发送给系统的 prompt），给出**逐字文本**，不留空白如“输入一个合适的 prompt”；每步预期的页面行为和截图点；支持证据（API/DB/日志）检查什么；在步骤中通过可见文本加位置来识别控件以消除歧义（"Submit (personal-info section)"），而非 css/xpath。每个流程的最后一步必须是结果的确认（页面上看到什么、数据中找到什么）；不能停在“动作已完成”。错误路径也要作为步骤写出；不要一笔带过。
- **回归手册** —— 此变更的爆炸半径落在哪些模块和契约上；要运行的现有套件的 literal 命令；从受影响表面挑选的相邻检查，每条都说明为什么被选中；要添加的回归检查（理想形式是“变更之前失败，变更之后通过”）；通过标准。

## 结果记录标准

每份执行手册都必须说明测试员如何记录结果。对 API 驱动、浏览器驱动、CLI 驱动和混合实时检查使用同一标准：

- 每条检查记录以背景/用户目标、环境、入口点和可能执行的写入开始。
- 在结论之前包含具体动作。对于 API 或页面控制台检查，包括方法、路径、载荷、凭证/会话模式、状态、关键响应体字段和 trace/request ID。
- 记录证明用户可见或运行时结果的观察表面：UI 状态、截图、DB 回读、日志行、审计事件、通知内容或手动用户确认。
- 对于异步或外部结果，命名观察窗口和谁/什么确认它。触发请求返回成功不足以证明邮件/聊天/推送/调度器行为。
- 对于负向检查，说明观察了哪些来源以及缺少什么匹配信号。如果缺席无法可靠观察，预期结果应为 `needs_more_evidence` 或 `blocked`（命名缺失的观察），而不是 pass——`needs_more_evidence` 是测试员在 Blocked checks 下记录的检查级别标记；产物级状态保持在测试员枚举内。
- 每条检查以清理/恢复证据和证据边界结束：此检查证明了什么，不能证明什么。

## E2E 执行形式：浏览器作为主要证据

对于面向用户的任务，E2E 默认为浏览器操作作为主要证据——测试员通过真实浏览器逐步驱动流程，以截图和页面状态为主要证据，API/DB/日志为辅助。当环境未就绪（服务未启动、路由未切换、数据未准备）时，标记对应流程为 blocked 并说明缺少什么；**不要把 API 驱动的检查写成 E2E**：此类检查无法证明用户实际看到的东西，应属于 API 手册。当确实只能 API 驱动时（纯接口产品、无 UI），在 E2E 手册中明确声明，并说明该层证据的边界。

当任务输入与此标准冲突时（例如任务声明“本轮前端只是可选入口点，API/日志为主要证据”），不要默默选择一个：E2E 手册仍按此标准编写浏览器版本步骤，在整体策略的开放问题中命名此冲突，说明每种形式能证明什么和不能证明什么，把执行哪个的决定留给人类关卡。

## 验证分层和风险排名

验证自然分层；安排检查时，让它们各归其位：

- **能力层** —— 每个 Must-Have 和每个失败/边缘用例都应有一个直接检查来证明它。
- **集成/API 层** —— 每对通信的模块和每个公共契约，至少有一个成功路径检查和一个错误传播检查。
- **E2E 层** —— 每个面向用户的能力至少出现在一个完整的用户目标中，每个目标都走过快乐路径和错误路径，并在每步留下验证。

当较低层仍有未解决的失败时，不要急于在较高层堆积检查——如果底部不稳定，高层的证据也无法成立。按风险（P0/P1/P2）对整个检查集排名，让最重要的先被看到，并在整体策略中声明执行顺序和关卡：哪个 P0 失败使哪些后续检查直接阻塞。

另外两类风险有专门负责人：bug 修复和高风险已有行为必须由回归检查后盾；任何涉及迁移、持久化、回填、回滚、保留或隐私敏感存储的，必须有明确的数据验证，包括审计/日志信号和安全/令牌约束（当需要时）。

## 环境

在整体策略中用纯 Markdown 清晰描述运行环境，让测试员可以通过遵循它来启动环境：环境类型（本地、docker、ssh、托管或其他）；如何运行命令；工作目录、端口和服务 URL（如相关）；按名称的环境变量（除非某个特定的非秘密值确实需要）；凭证仅引用，绝不打出秘密；以及环境当前是否就绪或需要即时设置。如果完全没有环境，标记 e2e 为 blocked 或仅限本地，并列出因此削弱的证据。

## 输出

将产物集写入持有指派 `output_path` 的 `test/` 目录，形状遵循 `references/templates/` 下的每个演示。

前置数据 `confidence` 为 `HIGH` | `MEDIUM` | `LOW`：HIGH = 每个 Must-Have 都可追溯到实际走访的入口，且没有开放问题；MEDIUM = 存在代码推断条目或开放问题；LOW = 需求缺口在开放问题中已命名。

此计划在 Must-Haves 全部可观察、禁止区域已具体绘制、集成检查覆盖真实边界、e2e 没有无理由的缺口、所有三份手册通过"可逐字执行"标准、测试员角色对测试负责人的建议真正可操作时，才就位。

如果需求或诊断标准模糊到无法自信地进行风险排名和验证设计，返回 `blocked` 并具体说明缺少什么证据，而不是编造。
