# 信息隐藏

## 描述

信息隐藏是把设计决策封装在模块内部使其不出现在接口中、对其他模块不可见的技术。每个模块应隐藏关于实现机制的知识，包括数据结构、算法与设计假设。其反面是**信息泄漏**，同一设计决策反映到多个模块中，造成需要协调变更的依赖。

良好的信息隐藏简化接口并让系统更容易演化。当信息被隐藏时，对该信息的变更只影响包含它的模块。

## 核心原则

### 封装设计决策

- 每个模块应封装代表设计决策的特定知识片段（例如，如何解析 JSON、如何在 B-tree 中存储数据、如何实现 TCP）。
- 隐藏信息包括数据结构、算法、低层细节（如页面大小）与高层假设（如"大多数文件很小"）。
- 目标是让信息完全隐藏，使其对模块用户无关且不可见。部分隐藏（仅少数用户需要的信息，通过单独方法访问）也有价值。

### 私有 vs 信息隐藏

- 将变量和方法声明为私有并不等同于信息隐藏。私有元素有帮助，但不保证隐藏。
- 公共 getter 和 setter 对私有变量的暴露程度，与变量直接 public 相差无几。
- 真正的信息隐藏意味着内部数据的性质与使用方式完全不通过接口暴露。

### 信息泄漏

- 信息泄漏发生在同一设计决策反映到多个模块中时。对该决策的变更需要修改所有相关模块。
- 泄漏可以是显式的（通过接口）或后门的（两个模块都知道某种文件格式，即使它不在它们的接口中）。后门泄漏更隐蔽，因为不明显。
- 如果在类之间遇到泄漏，重新组织使知识只影响单一类。选项包括合并类，或将共享知识提取到一个具有简单、抽象接口的新类中。

### 避免时序分解

- **时序分解**按操作发生的时间顺序构建系统（例如，为读取、修改和写入文件分别设立类）。
- 这会导致信息泄漏，因为设计决策（如文件格式）在执行的多点出现。
- 关注每个任务所需的知识，而非操作顺序。将相关知识合并到单一模块中，即使这些知识在不同时间使用。

### 为隐藏而设计接口

- 更简单的接口与更好的信息隐藏相关。复杂接口暴露更多信息并创造更多依赖。
- 不要暴露内部数据结构（例如，返回用于存储参数的内部 Map）。内部表示的变更不应影响接口。
- 提供隐藏内部表示并减少调用者负担的更深方法（例如，`getParameter(String name)` 而不是 `getParams()` 返回内部 Map）。

### 默认值与"做正确的事"

- 为常见场景提供合理的默认值。默认值是一种部分信息隐藏——调用者在正常使用时不需要了解它们。
- 类应自动"做正确的事"而不需要被显式要求。例如，文件 I/O 中的缓冲应是自动的，而非 opt-in。
- 让常见 case 尽可能简单。罕见场景可以要求特殊方法。

### 为更好的隐藏让类更大

- 信息隐藏常常可以通过让类稍大来改进，将特定能力的所有相关代码集中在一起。
- 合并相关功能可以提升接口层级（例如，一个方法完成整个计算，而不是三个方法做三个步骤）。
- 这会产生接口更简单的深层模块。

## 危险信号

### 信息泄漏指标

- **重复知识**：多个类理解相同的文件格式、协议或数据结构
- **后门依赖**：类依赖同一设计决策，即使它不在它们的接口中
- **协调变更**：修改一个设计决策需要修改多个模块

### 时序分解指标

- **结构中的操作顺序**：执行的不同阶段（读取、处理、写入）位于不同的类中
- **共享格式知识**：读取类与写入类都需要理解相同格式
- **基于阶段的命名**：如 `RequestReader` 和 `RequestParser` 这种反映顺序步骤的类名

### 浅接口指标

- **暴露内部**：方法返回内部数据结构（Map、List）用于存储状态
- **内部结构的 getter**：`getParams()` 返回内部参数 map 而不是 `getParameter(name)`
- **表示泄漏**：内部表示变更时接口也随之变更
- **额外调用者工作**：调用者在获取数据后必须进行额外查找或转换

### 缺失默认值

- **强制指定**：调用者必须提供本可自动确定的值（例如，HTTP 响应版本匹配请求版本）
- **罕见选项变必需**：罕见配置选项在常见场景中也必须被指定
- **过度暴露**：使用常见功能迫使使用者了解很少使用的功能

### 过度分解

参见 `cohesion-separation.md` 获取完整的合或分危险信号与标签集合。

- **每个阶段一个类**：多步流程（如读取与解析）的每一步都在独立类中
- **重复解析**：相似解析或验证逻辑出现在多处
- **复杂使用模式**：调用者必须按特定顺序调用多个方法才能完成一个任务

## Issue Tags

在设计产出中标记信息隐藏问题时使用以下标签：

- **[Information Leakage]** - 同一知识或设计决策出现在多个模块中
- **[Temporal Decomposition]** - 系统结构反映执行顺序而非知识边界
- **[Exposed Internals]** - 内部数据结构或表示在接口中可见
- **[Shallow API]** - 接口暴露过多细节或提供的抽象不足
- **[Missing Defaults]** - 常见场景需要显式指定本可自动的值
