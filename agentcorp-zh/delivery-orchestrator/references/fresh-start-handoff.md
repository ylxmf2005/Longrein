# 全新启动交接

当当前对话或工作树可能污染后续工作时使用此能力：一个漫长的多轮调试/重构会话、同一底层问题反复修复却不能稳定、需求散落在多个轮次中或中途变更、早期架构假设被推翻、工作树充斥着跨模块的探索性未提交变更，或者发起人说"重新开始 / 新会话 / 上下文一团糟"。目标不是总结所有内容，而是判断当前线程是否已成为负担，并在发起人同意后——产出一个干净的交接 prompt，让新会话（或新的子代理分配）避免继承污染。

核心操作：将零散的上下文转化为自包含的唯一真相源 prompt，同时隔离并明确标记探索内容、失败尝试和过时假设，而非继续将它们写成叙述。

## 何时触发（评分指南）

在自然阶段边界进行快速检查：大重构之前、反复调试失败之后、提交之前、需求变更之后、对话开始变得冗长过时时。

- +3：发起人明确或隐含表示"重新开始"、"新会话"、"交接"或"上下文一团糟"。
- +3：同一底层问题连续两次或更多次修复失败。
- +3：你发现早期的架构假设或诊断是错误的。
- +2：需求只有跨多轮才拼凑完整，或在实现开始后被变更。
- +2：下一步依赖于记住若干早期约束或例外。
- +2：工作树存在跨模块的探索性未提交变更。
- +2：工作正从探索/调试转入干净的实现阶段。
- +1：对话中包含矛盾结论、膨胀的事后分析或"不要再做 X"的修正。
- +1：验证结果是混合的、过时的或不清晰的。

≥3 分：暂停并咨询；≥5 分：强烈建议重新启动。不要在小的单步变更时打断，也不要在上下文仍然干净时打断，或在发起人刚拒绝且风险未实质增加时打断。

## 咨询（这是一个人工门禁）

如果发起人明确要求交接 prompt，直接编写；否则只问一次，附一个具体理由，提供选项：A) 仅 prompt；B) git 隔离方案 + prompt；C) 留在当前会话继续。如果他们选择 C，继续原始任务且不再询问，除非风险升级。绝不为发起人做重启决定；没有发起人的明确批准，绝不丢弃、重置、暂存、提交或创建分支。

## 产出交接 Prompt

在开始编写之前，将本轮合格的经验教训捕获到 `teamspace/learnings/`（参见 `references/learnings.md`）——重启应该只丢弃被污染的对话，而不是连带丢弃经验教训。

1. **盘点当前真相。** 信任源码、测试、`git status`、diff、日志和发起人的明确指示，而非对话的记忆；如果能查询仓库，先查询再编写；如果不能，诚实地说明未知事项。
2. **将信息分桶。** 目标 / 完成定义 / 已验证事实（附证据）/ 相关文件和入口点 / 已接受的约束和决策 / 失败尝试（作为教训和禁区，而非作为继续的起点）/ 可疑或未验证的假设 / 工作树状态。
3. **确定工作树立场。** 新工作从何处开始，四选一：干净的基线分支（探索性工作归档为只读参考）；当前脏树（未提交变更作为候选工作对待，而非已验证事实）；检查点分支；仅作为历史参考保留的归档。
4. **编写 prompt 并直接交给发起人。** 使用下面的骨架，删除不适用的部分，但保留唯一真相源和反污染措辞；放在 fenced markdown 块中以便发起人立即复制，不要埋在冗长的事后分析下面。

```markdown
你从零开始，不依赖任何先前对话。将此 prompt 和当前仓库状态作为唯一的真相来源；当两者冲突时，信任仓库和测试，并在修改代码之前报告差异。

## 目标
## 完成定义（可观察的成功标准，必须通过的测试/构建/手动检查）
## 工作树立场（四选一，附对应的分支/归档位置）
## 相关文件和入口点（路径 + 为何相关）
## 命令和验证
## 已验证事实（已验证：事实 — 证据：文件/测试/日志/发起人指示）
## 已接受的约束和决策（已接受：……）
## 失败尝试，不要盲目重复（失败：方法 — 证据 — 教训）
## 可疑/未验证假设（未验证：假设 — 如何验证）
## 建议路径（先确认事实，然后最小变更，然后运行验证；如果验证失败，根据证据修正而非根据旧假设修正）
## 护栏（在破坏性 git 操作、大重写、依赖升级、迁移、文件删除之前询问；不要重用失败的代码除非你能解释为何该失败不再适用；当上下文缺失时，提出针对性问题而非猜测）
```

## 常见错误

- 不要写"聊天中发生了什么"的流水账。
- 不要将失败的代码作为起点传递给下游，除非它已被归档并明确标记。
- 不要将未验证的内容写成"我们知道 X。"
- 不要将旧的变通方案和新计划混在同一条指示中。
- 不要让新会话在不知情的情况下继承一棵脏的工作树——要么清楚解释，要么建议隔离。
- 不要列出你接触过的每个文件，只列与下次尝试可能相关的文件；工作树状态另行说明。
