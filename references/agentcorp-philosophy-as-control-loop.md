# 旧版 AgentCorp 全技能的共同哲学：证据、制衡与可恢复的交付循环

> 来源范围：2026-07-16 对旧版 AgentCorp 全量 Skill 的结构扫描，并重点复核 Delivery、Probe、Brainstorm、Grill、Research、Review、Test、Walkthrough、Compound 与 Skill Evolution 家族。旧版现保存在 `/Users/ethan/skills-hubs/AgentCorp.bak`。本文提炼跨 Skill 原则，不复制各角色的阶段细节。

## 一、AgentCorp 不是一组角色名称，而是一组对自我欺骗的断路器

几十个 Skill 表面上分别负责需求、设计、实现、评审、测试和交付，但它们反复出现的是同一类问题：模型很容易把名称、计划、信心和绿色状态当成事实。

因此，AgentCorp 最稳定的母原则不是“多代理”，而是：任何状态变化都必须由外部可检查证据支持。

- Acceptance：没有打开证据，就没有评审；
- Review：没有走通具体路径，就没有发现；
- Test：没有实际执行请求或用户流程，就没有通过；
- Delivery：任何工作都不能只凭作者自己的话进入下一阶段；
- Compound：记忆是待核验假设，真实轨迹和产物才记录失败与代价。

这与反馈控制高度一致：状态词不是反馈，只有能被重新观察的输出才是反馈。

## 二、第一条精华：具体路径优于抽象担忧

Correctness、Security、Reliability、Adversarial、Performance 等 Reviewer 都拒绝模式匹配式结论。一个发现必须包含具体输入或扰动、实际经过的代码或状态路径，以及最终错误结果。

这正是 Skill 进化应采用的误差单元：

```text
代表性输入 → Skill/模型作出的关键判断 → 实际输出 → 与预期的具体差异
```

“这个 Skill 太长”“模型不够主动”“感觉容易越权”都只是方向。只有路径可重放，改进才有靶点。

## 三、第二条精华：调查权、建议权和执行权必须分开

Probe 可以探索用户没有要求的地形；Scope Challenger 可以调查更大或更激进的路线；Brainstorm 可以给鲜明推荐；但这些产物都只是反应材料，不能自行改变范围或落地代码。

Skill Evolution 更进一步：即使已经捕获到明显改进，也必须把具体 diff 展示给人，得到针对该 diff 的明确批准，才能写入长期 Skill。

这条边界直接解决“一次性要求被误写成永久规则”：当前任务允许模型尝试和建议，不代表允许它修改持久化行为。

## 四、第三条精华：作者与裁决者分离

Delivery Orchestrator 的铁律是作者不能给自己盖章。Review Researcher 又在评审和修复之间增加断路器：评审发现先被独立重走路径，确认、部分确认、误报或需要人工，然后 Fixer 才能修改。

这种分离并非每个小任务都需要多代理，但它揭示了一个认知事实：实现者已经建立了自己的路径惯性，同一上下文的“再检查一遍”很容易继续相信原模型。

下一代 Skill 可以把独立性按风险调节：低风险任务由同一 Agent 验证并诚实标注；高风险任务使用新上下文、另一模型或人工裁决。

## 五、第四条精华：失败必须显式，偏离必须留下轨迹

AgentCorp 反复禁止 silent fallback、吞错和伪成功。实现遇到计划外边界时，需要记录“计划是什么、发现了什么、做了什么、为什么”。测试无法执行时必须标记 blocked，而不是缩小覆盖后继续宣布绿色。

这使 Rewind 成为可能。没有偏离记录和失败证据，后来只能凭记忆猜哪个节点开始失真；有轨迹，才能把可靠资产与错误惯性分开。

## 六、第五条精华：交付必须让人重新获得参与资格

Explain 要给结论状态和证据；Walkthrough 不按文件顺序复述 diff，而是重建用户理解，并用测验确认人是否还能参与下一次决定。Delivery 把 sponsor understanding 当作真实 pipeline state。

这反映了用户的核心哲学：AI 不是替人把所有判断做完，而是制造更多高质量判断样本，同时把人带回循环。产物路径、证据和自包含解释都服务于这一点。

## 七、第六条精华：沉淀必须改变未来行为

Compound 不把“总结了经验”当沉淀。修复应成为回归测试，稳定偏好应进入规则，评审模式应形成可批准提案；如果没有值得保留的资产，也应明确写“无可沉淀”。

Skill Evolution 同时阻止自动自改：提案可以自动捕获，长期行为修改必须人工批准。这正好把反馈学习和治理分开。

## 八、旧版 AgentCorp 不能直接搬进新 AgentCorp 的部分

AgentCorp 的优势来自强制分工和阶段产物，但它也有明显成本：

- 大量角色重复“地图不是领地”和证据要求；
- 每个阶段都有状态、frontmatter、handoff 和 gate，简单任务容易被流程吞没；
- 许多规则依赖独立代理、真实 diff 或运行环境，放进 Shape 会变成没有对象的预演；
- 详细输出契约会增加上下文并诱导模型按模板填满，而不是做最小有效工作。

新 AgentCorp 应吸收控制原则，不吸收旧组织图。主 Skill 需要知道什么时候必须取得证据、什么时候需要独立性、什么时候要人批准；具体角色、文件格式和严重度留给按需 reference 或后续专门 Skill。

## 九、适合带入 Shape、Prompt、Rewind 的压缩版

- Shape：未知先从领地调查，建议只是反应材料，方向需要知情选择。
- Prompt：长期规则必须指向真实误差或稳定价值；临时指令无权自行持久化。
- Rewind：失败和偏离保留证据，过时结论失去权威；作者不能用新叙述覆盖旧轨迹。
- 三者共同：每个重要判断都给出可检查句柄；证据不足时保留未知，不用完整语气填洞。
