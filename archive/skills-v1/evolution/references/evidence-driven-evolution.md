# 证据驱动的 Skill 进化

Skill 会长期影响许多未来任务。一次行为规则修改可能带来很大范围的变化，因此不能因为“看起来更好”就直接写入。断链、格式、历史状态和 UI 元数据等不改变行为语义的维护，应使用能直接证明它的静态检查，不机械制造模型评测。

## 一起看失败、成功和反例

真实轨迹提供方向：模型怎样理解输入、作出什么判断、采取什么行动、产生什么后果。不要只围绕最先出现的失败建立解释；同时寻找仍然有效的行为、邻近反例、用户纠正、成本变化和可能的共同原因。

这一阶段不要太早确定原因。项目实现、提示词、上下文、工具环境和 Skill 可能共同作用；先解释现象，再决定应该改哪里。

## 找原因和做实验是两个阶段

找原因时要允许不同解释竞争。形成可以验证的假设后，实验才需要控制变量：固定代表性输入、模型、effort、工具和环境，只改变正在验证的那部分 Skill。

同一批材料重放候选版本，并加入能够暴露过拟合的反例。比较的不只是任务是否通过，还包括用户介入、意外动作、完整性、证据、延迟、成本和长上下文副作用。

控制变量是做实验的方法，不是找问题前的搜索边界。

## 改多少由原因决定

证据支持哪一层原因，就修改哪一层。变化可以落在触发描述、核心说明、按需参考、工具边界、项目规则或自动执行结构；范围由证据决定，不机械追求最短文字或最少文件。触发、判断、动作、权限或产物语义变化使用行为评测；引用、格式、元数据和历史标记使用对应静态验证。

隐去当前项目名称、对象和任务专有词后，候选规则仍然准确描述同一种行为误差，才可能是通用 Skill 变化；否则应留在任务、项目或用户资料中。

删除已经不再有用的旧流程，避免用新规则覆盖旧规则。新增指令还要检查是否与相邻 Skill 的触发和职责冲突。

## 临时经验不能自动变成永久规则

当前任务中有效的要求，可能只适合这一次。会改变长期行为的 Skill 变化需要可重放问题、适用范围、前后评测和失效条件，并由用户针对具体 diff 或明确变化范围批准。调用 Evolution、认可问题或只要求研究，都不等于批准长期修改。

行为变化落地后重新运行相同评测；机械维护重新运行相应静态检查，并保留路径与结果。只有证据表明未来行为已经改善，才能称为进化；只有文字发生变化时，只能称为已修改。
