# 命名与显见性

## 描述

代码应能立即被理解。当开发者阅读代码时，他们对代码做什么的第一直觉猜测应是正确的，无需深入分析或不断查阅文档。设计时，使用此维度评判你敲定的命名与约定是否通过精确命名、遵循约定、与读者期望对齐来使含义清晰。

晦涩是复杂度的首要原因。当重要信息被隐藏或需要努力发现时，开发者浪费时间、做出错误假设并引入 bug。良好命名是基础——名称创造引导理解的 mental image。一致性通过允许开发者将知识从一个上下文转移到另一个上下文来放大这一点。当代码违反期望或使用模糊构造时，它迫使读者放慢速度并 puzzling out 真实行为。

设计时，不断问自己：有人能快速阅读结果代码并在第一次 pass 就正确理解吗？还是需要仔细研究才能解码？

## 核心原则

### 1. 选择精确的名称

名称应创造它们所代表事物的清晰 mental image。精确意味着名称缩小可能性——当有人在孤立情况下看到名称时，他们可以在不看文档或使用上下文的情况下准确猜测它指什么。

- 像 `count`、`data`、`info`、`status` 这样的通用名称通常太模糊
- 名称应区分相似概念（例如，`fileBlock` vs `diskBlock`，而不是只用 `block`）
- 对布尔值，使用清晰表明 true/false 含义的谓词
- 避免可能合理指代多个不同事物的名称
- 当难以找到精确名称时，这通常表明设计不清晰

### 2. 一致使用名称

一旦为特定目的选择了名称，在所有地方都为该目的使用它，绝不用作其他用途。一致性减少认知负荷——阅读者学习一次模式就可在各处复用。

- 为常见概念建立约定并在整个代码库中遵循
- 对服务相同角色的变量在整个代码库中使用相同名称
- 绝不在不同上下文中为不同目的复用名称
- 当需要多个相关变量时，添加区分前缀（例如，`srcFileBlock`、`dstFileBlock`）
- 约定应足够窄，使所有用途都有真正相似的行为

### 3. 在代码中保持全局一致性

除名称外，一致性还适用于结构、风格、模式与不变量。相似事物应以相似方式完成；不同事物应看起来不同。

- 在文件中工作时遵循现有编码风格与结构
- 一致使用已建立的设计模式与接口
- 统一应用命名约定（camelCase vs snake_case、循环变量约定等）
- 维持减少特殊 case 推理的不变量
- 抵制"改进"现有约定的冲动，除非收益足以 justify 更新所有使用

### 4. 让代码符合期望

代码在按读者期望表现时最为显见。违反约定或常见模式会创造 surprise 与 confusion。

- 操作应按其名称暗示的方式工作
- 代码结构应遵循语言/框架中的熟悉模式
- 不寻常行为（构造函数中 spawn 线程、不返回的 main 函数）必须被文档化
- 在适用时优先使用标准库模式与习语

### 5. 减少所需信息

让代码显见的最佳方式是消除对隐藏信息的需求。设计应让理解行为所需的上下文更少。

- 使用抽象隐藏无关细节
- 消除要求阅读者跟踪额外状态的特殊 case
- 避免模糊含义的构造（泛型容器、声明与分配类型不匹配）
- 为易读设计，而非易写

### 6. 清晰呈现关键信息

当信息至关重要时，使其可见。使用格式、注释与结构来突出重要内容。

- 使用空白分隔逻辑块并让结构可扫描
- 为不显然行为添加注释（事件 handler 调用、意外控制流）
- 记录代码何时违反正常期望
- 确保重要信息位于阅读者会查看的位置

## 危险信号

### 不精确命名

- **通用变量名**：`data`、`info`、`result`（在非显然上下文中）、没有限定的 `status` 与 `count`、非坐标概念的 `x`/`y`
- **模糊名称**：单一名称用于多个不同目的（例如，`block` 同时指 file block 与 disk block）
- **过于具体的名称**：暗示比实际用途更窄使用范围的名称（例如，适用于任何 range 的方法却使用 `selection` 参数）
- **模糊布尔名称**：不能清晰表明 true/false 含义的布尔变量（例如，`blinkStatus` 而不是 `cursorVisible`）
- **与行为不匹配的名称**：方法或变量名暗示一件事，实际行为却不同

### 命名不一致

- **同一概念，不同名称**：代码库不同部分对等价概念使用不同叫法
- **同一名称，不同含义**：在概念不同的事物上复用常见名称
- **不可预测的约定**：相似构造如何命名没有规律
- **打破已建立模式**：新代码忽略周围代码中可见的命名约定

### 结构不一致

- **混合编码风格**：同一代码库中不一致的缩进、括号位置、声明顺序
- **接口实现变体**：同一接口的不同实现使用不兼容的方法
- **相似操作使用不同模式**：同一问题每次以不同方式解决
- **违反不变量**：应始终成立的属性有时被破坏

### 模糊含义的代码

- **无文档的事件驱动回调注册**：handler 的调用上下文不清晰
- **泛型容器用于多值返回**：使用 `Pair<Integer, Boolean>` 而不是具有有意义字段名的特定类型
- **声明与分配类型不匹配**：声明 `List` 却分配 `ArrayList`，隐藏实际行为
- **令人惊讶的控制流**：代码看起来应该退出时却没有退出，或看起来应该继续时却停止了
- **非显然副作用**：操作做的比表面看起来更多

### 缺失关键信息

- **未记录的期望违反**：不寻常行为在注释中未被指出
- **密集格式**：代码或文档挤在一起，使结构难以解析
- **复杂逻辑缺少注释**：非显然算法或模式没有解释
- **隐藏假设**：代码依赖在使用点不可见的知识

### 难以命名的实体

- **难以找到简短清晰名称**：通常表明实体没有干净单一目的
- **需要冗长限定的名称**：暗示在一个地方试图做太多
- **多种合理解释**：名称可能以相同概率指代几种不同事物

## Issue Tags

在设计产出中标记命名与显见性问题时使用以下标签：

- **[Vague Name]**：名称太泛，无法传达含义（例如，`data`、`info`、`status`）
- **[Ambiguous Name]**：名称用于多个不同目的或可能指代多种事物
- **[Inconsistent Naming]**：同一概念在代码库中命名不同，或同一名称被用于不同概念
- **[Naming Convention Violation]**：代码打破项目中的已建立命名模式
- **[Style Inconsistency]**：代码与周围文件的格式/结构约定不匹配
- **[Pattern Inconsistency]**：相似问题以不同方式解决而没有正当理由
- **[Nonobvious Code]**：代码行为无法从快速阅读中看清；重要信息被隐藏
- **[Generic Container Abuse]**：在特定类型能澄清含义的地方使用 `Pair` 或类似泛型类型
- **[Violates Expectations]**：代码行为与读者自然期望不同
- **[Undocumented Surprise]**：不寻常行为（线程、不返回、副作用）在注释中未解释
- **[Poor Information Layout]**：密集格式或不清晰结构使代码难以扫描
- **[Hard-to-Name Entity]**：难以选择精确名称暗示底层实体设计不清晰
