# evolution

## 让这轮代价改变下一轮

Evolution 不是复盘日记，也不只负责修改 Skill。它读取真实产物、测试、日志、diff、决定和失败轨迹，寻找哪项经验能让未来 Agent 更早看见原因、更快证明结果，或不再重复同一错误。

先把失败还原成完整行为路径：相关能力是否被发现和加载，是否正确触发，加载后是否被遵循，还是 Skill 本身确有缺陷。只有最后一层在重放中成立，才修改 Skill 正文。当前任务的对象、名词和临时要求只是样本，不是通用规则；候选变化必须指向能够跨任务复现的机制。

“自行改进这个 Skill”没有现实反馈，通常只会得到更完整、更严谨也更重的文字。一起看失败、仍然有效的行为、反例和成本，才能判断真正出了什么问题。最先出现的失败只是线索，不自动代表根因、完整范围或正确改法。

没有真实案例时，可以形成研究假设，不能直接改变长期行为。

## 主动捕获，但不抢走当前任务

发现可复用信号时，先完成或保护当前主任务。只有缺陷正在继续放大错误时才立即打断；其他情况在自然阶段结束时告诉用户：观察到什么模式、它影响了什么、证据在哪里、建议落到哪一层。

一次产品 bug 不自动等于 Skill bug。项目实现、当前提示、旧上下文、工具环境和长期 Skill 可能单独出错，也可能互相影响。沿真实记录检查原因，再决定该改哪一层：项目问题留在项目，当前表达由 `prompt` 改善，旧上下文由 `rewind` 清理，跨任务可重现的触发、判断或流程问题才进入 Skill 评测。

没有高价值资产时，简短说明“无可沉淀”及原因即可。不要为了证明 Evolution 被触发而制造文件或规则。

## 把经验放到正确的位置

同一个发现可能有不同的保存方式：

- 值得永久重放的具体失败：符合项目规则的回归案例、检查或最小复现；
- 会影响未来工作的已验证决定：最接近作用域的项目规则、设计记录或参考；
- 已经稳定的重复劳动：脚本、模板或工具，并保留验证方法；
- 只属于当前任务的事实与决定：留在 `studio/tasks/<task-id>/`，不支配其他任务；
- 可复用 Skill 的触发、判断或流程错误：进入下面的 Skill 评测闭环。

一次有效不等于永久有效。长期保存的内容应说明证据、适用范围、维护成本和失效条件；遵守当前任务对测试、脚本、规则与提交范围的授权，不能借“沉淀经验”扩大交付。

真实失败是发现问题的好入口，但保存下来的内容应尽量记录原因和可检测边界，让系统以后不必等同一错误再次发生。能用权限、结构或自动检查执行的经验，不要只增加一段更强语气。

## 用对照测试证明 Skill 真的变好

保存足以重放和判断问题的材料：输入、模型与 effort、相关 Skill 版本、可用工具、关键输出、实际后果和预期差异。隐私或敏感信息仍只保留必要片段。

然后：

1. 确认当前 Skill 已被发现、加载并触发，再重放基线；误差不能在这一条件下复现时，修复发现、触发或运行环境，不修改 Skill 正文；
2. 结合失败、成功、反例和成本，形成能够解释现象的候选原因；
3. 候选原因已经展开后，再通过控制变量验证它们；
4. 在相同环境重放原失败，并加入一个邻近反例，防止只修单样本；
5. 比较正确性、完整性、用户介入次数、意外动作、token、延迟和产物质量；
6. 删除不再改变行为的旧规则，而不是用新规则覆盖旧规则。

迁移到新模型时尤其要分清改善来自哪里：探索原因时不把第一个错误当答案；进入实验后保持模型、effort、工具和任务样本稳定，只改变正在验证的部分。需要建立或审查 Skill 评测时，读取 [evidence-driven-evolution.md](references/evidence-driven-evolution.md)。

## 提出具体 Skill 变化

评测支持改进后，向用户展示：

- 具体误差和可检查证据；
- 根因位于触发描述、核心说明、reference、工具还是保存规则；
- 与根因和影响范围相称的 diff；
- 前后评测结果和仍未覆盖的范围；
- 这条变化可能带来的副作用或新成本。

提案是反应材料，不是批准。用户必须对具体 diff 明确说是；调用 Evolution、要求“处理改进”或认可问题存在，都不授权长期 Skill 自我修改。

## 落地后保持系统一致

获批后更新源 Skill、发现入口和必要 reference，检查所有相邻 Skill 的职责与触发描述，避免同一信号被多个能力争抢或相互否定。新增、改名或删除 Skill 时，同步仓库现有的发现结构和符号连接。

重新运行同一批评测，并确认：目标行为改善；低风险任务没有新增仪式；临时指令没有被写成永久规则；产物路径仍可发现。没有验证结果就只报告已修改，不能宣称已改进。

## 产物

任务内使用 `studio/tasks/<task-id>/evolution/evolution.md`，独立研究使用 `studio/evolution/<YYYYMMDD>-<slug>.md`。文件记录准备保存的经验、证据、保存位置、授权与实际验证；涉及 Skill 时再保存可重放材料、基线、候选 diff、评测和用户决定。状态使用 `candidate | testing | proposed | approved | rejected | landed`，不复制整段私密会话。

每个实际落地的资产都告诉用户路径；未获授权、尚未验证或只形成假设的内容保持候选状态。

## 边界

Evolution 统一处理“这轮学到的东西怎样改变未来”，但不重新打开已经完成的产品范围。`prompt` 只改善当前表达；`shape` 只改变方向；`rewind` 只修复时间线；`soul` 约束所有落点是否真实、克制并值得维护。

面向用户使用其当前工作语言；代码、路径和专有名词保持原样。
