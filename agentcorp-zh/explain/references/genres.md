# 解释体裁

三种最常见的解释请求的按体裁检查清单。本文中的每个示例都遵循铁律：结论必须附带状态及其证据句柄。

## 解释 Bug

在解释 bug 或错误时：

- 说明用户或系统当时试图做什么。
- 说明哪里出了问题。
- 用平实语言命名最可能的原因。
- 说明其状态是已复现、已修复、部分修复还是仍在调查中。
- 引用支持该结论的证据。

示例：

> 保存看起来成功了，但数据库没有变化。前端发送了 `userId`，而后端只接受 `user_id`，因此后端拒绝了请求。页面没有将该拒绝暴露给用户，导致失败看起来像成功。通过重放请求确认——后端对 `userId` 返回 400（`logs/api.log:212`）；尚未修复。

## 解释测试进度

在解释测试或验证时：

- 说明测试了哪个用户旅程、代码分支或风险。
- 说明结果：通过、失败、阻塞、跳过或未运行。
- 如果测试失败，说明该失败指向什么行为。
- 将"本切片测试通过了"与"整个系统安全了"分开。
- 命名剩余风险及下一步验证动作。

示例：

> 正常登录流程在浏览器中工作（通过从头到尾走通旅程验证）。过期会话处理未测试，因此该结果对 happy path 有置信度，但对超时行为没有。下一步：在签署前驱动一次过期会话登录。

## 解释实现

在解释某个功能或技术设计如何工作时：

- 从用户可见或运营层面的目的开始。
- 命名主要组成部分，以及每个部分负责什么。
- 按顺序走通流程。
- 指出关键的不变量或保护机制，不能随意移除。
- 仅在权衡会影响用户的决策或审查者的信心时才提及权衡。
