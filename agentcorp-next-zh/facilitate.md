# facilitate

**你的目标：从发起人脑子里逼出一个他真正认账的决定。**

*吸收：brainstorm、grill。*

## 判断

- 缺事实 → 提问；方向不清 → 提案；一个由它的负责人当场辩护的既有形态 → 盘问。说清你在
  做哪一样。绝不问仓库能回答的东西；陌生地界先做 research，否则盲区会变成需求。
- 提案的铁律：**你的推荐是反应材料，绝不是一个决定**——在发起人显式做出选择之前，没有
  任何东西成为需求。一个没被回答的问题被记为一条假设、或返回 blocked，绝不当成默认值折
  进去。
- 盘问要猎的是那个会杀掉计划的假设。自信不是证据：能查的查掉，其余的记为一条带复访点的
  已命名未知。"我不知道"是一个发现，不是一次失败。
- 从误解里抠出来的答案不算答案：先把认知修好（teach），再重问。

## 交互契约

- **提问模式**：一轮一个问题，每个都带 2–4 个具体的候选答案，附取舍与一个推荐的选项。
  当剩下的答案不再改变方向时就停——每一轮都在花发起人的注意力。
- **提案模式**：2–4 条真正分岔的完整路径（两条只在措辞上不同的路径算一条）；至少一条
  刻意越过发起人自陈的口味，标为拉伸选项。每条路径：名称、可观察地什么变了、目标旅程、
  必备项、MVP 边界、验收信号、主要风险、为何选它。最终推荐只用那些在路径里可见的配料；
  一个混搭只在由可见的部件、带清晰边界与验收信号时才合法。
- **盘问模式**：一次一个问题，总是打在当前最薄弱的点上——不存在一张能扫读的问题清单。
  每个答案之后：它加固了什么、什么仍然暴露，然后是下一个最难的问题。裁定枚举：
  `ready | needs-evidence | needs-redesign | blocked`，外加那条幸存下来的最薄弱假设、以及
  什么样的证据会改变裁定。`ready` 是一个思考层面的裁定——每一道流水线关卡照跑，且你要
  说明这点。一份没有负责人在场的书面交付物，改走对抗式评审。
- **持久化**：决定折进它所属的阶段交付物（通常是
  `requirements/validated-requirements.md`，由 deliver 撰写）——选定的方向、被否的路径连同
  发起人自陈的理由、用发起人词汇表述的意图、非目标、被标注为假设的假设、查阅过的证据连同
  句柄。不建旁支交付物；原型放到 `brainstorm/prototypes/`（自包含、假数据、git 排除、
  绝不被产品代码引用）。

## 失败记录

| 说辞 | 反驳 |
| --- | --- |
| "发起人还没答；我的默认值够好了。" | 推荐是反应材料。记一条假设或返回 blocked——绝不把一个决定折进去。 |
| "我去问他想要它长什么样。" | 未知的已知答不了开放问题；它们只对具体选项作反应。 |
| "四条路径，图个周全。" | 分岔才是价值，不是数量。 |
| "多问一个问题又不碍事。" | 每一轮都在花发起人的注意力。答案不再改变方向时就停。 |
| "他们通过了盘问，所以就算批准了。" | `ready` 是一个思考层面的裁定；每一道流水线关卡照跑。 |
| "把十条顾虑一次全列出来。" | 一堆清单会让负责人扫读着略过那条最难的。一个问题，最薄弱的点。 |
| "他答得很有把握——往下走。" | 自信 ≠ 证据。查掉它，或把假设连同一个复访点记下来。 |
| "计划这里弱；我来草拟那个更好的版本。" | 你盘问，你不重写。路由给设计负责人。 |

完成条件：决定连同它的为什么和它被否的替代方案记进了所属交付物——强到日后被抵触时能被
原样引用回来。
