# Test 证据链标准

本文件供非平凡测试、跨系统变化、异步终态、完整 E2E 和正式 Test Report 使用。它规定测试必须留下什么，不要求每个小改动填满全部章节。

## 一、风险覆盖镜头

先从改动的承诺和影响面选择相关镜头，不机械全跑：

| 镜头 | 需要追问的事实 |
|---|---|
| 主路径 | 用户能否从真实入口完成完整目标，终态是否正确 |
| 错误路径 | 无效输入、依赖失败、冲突、超时和重试如何表现 |
| 权限与所有权 | 不同身份能否只读写自己应拥有的资源，错误码是否稳定 |
| 生命周期 | 创建、读取、更新、软删除、恢复、硬删除和后续引用是否一致 |
| 组合操作 | move、clone、fork、bind、detach、favorite 等相邻操作是否受共享模型变化影响 |
| 并发与幂等 | 重复请求、并发更新、回调重放和部分失败会否产生重复或卡死状态 |
| 数据与迁移 | 新旧版本、默认值、回填、审计、回滚和持久化终态 |
| 公共契约 | 请求/响应、错误语义、CLI、schema 和已有消费者兼容性 |
| 高级用户 | 合法但不常见的组合、长流程、完成后继续操作和跨资源关系 |
| 回归 | 原失败、受影响现有行为和明确不应变化的边界 |

表中的操作名只是常见例子，不是所有项目都必须存在。测试真实产品能力，而不是为了覆盖表格发明场景。

## 二、可直接执行的用例

```markdown
### <用例 ID> — <场景>

- 背景与风险：为什么测。
- 优先级：P0 | P1 | P2，或项目已有等级。
- 环境与身份：入口、调用者、权限、依赖、允许的写入。
- 前置数据：真实资源或创建步骤。
- 实际动作：可直接执行的请求、命令或编号 UI 操作，保留字面输入。
- 预期：用户可见结果、状态码/业务码和持久化终态。
- 证据：响应、截图、SQL、日志、审计或外部确认的位置。
- 失败处理：停止、继续，或阻塞哪些后续用例。
- 清理：恢复动作和确认方法。
```

“调用列表接口并检查结果”不合格；给出方法、路径、参数、payload 和断言。页面操作使用可见文字和所在区域，不把脆弱选择器当作测试意图。

## 三、逐用例执行记录

```markdown
### <用例 ID> — <场景> — <status>

- 背景 / 风险：
- 前置条件：
- 预期：
- 实际动作：
- 实际响应与观察：
- 终态回读：
- 清理与清理回读：
- 证据：
- 证据边界：本项证明……；不能证明……
```

### API、CLI 和自动化套件

保留准确命令或请求、目标环境、退出码或状态码、业务错误、关键字段和原始日志路径。自动化套件可以合并记录，但必须列出命令、目标、退出码、具体失败和重跑历史。

### 浏览器与真实用户流程

记录页面、实际输入、每一步操作、观察到的页面状态和截图。API、数据库和日志可以支持解释，但不能替代用户界面本身的证据。视觉检查只在视觉行为属于承诺时执行。

### 异步与外部结果

记录触发时间、观察窗口、轮询或回调标识、最终状态和确认来源。超出窗口仍没有终态时为 `partial` 或 `blocked`。

### 负向检查

说明观察了哪个来源、多久、预期缺少什么信号。无法可靠证明“没有发生”时，明确留下证据缺口。

## 四、覆盖台账

| 用例 ID | 场景 | 状态 | 执行记录 | 最强证据 |
|---|---|---|---|---|
| `<id>` | `<risk / user goal>` | `passed / failed / partial / blocked / skipped` | `<anchor>` | `<path / link>` |

实际改动产生新风险时新增用例并说明来源。原用例因设计变化失效时保留编号并标记 `skipped`，说明由什么替代。每个必须实现的结果、受影响消费者和承重风险都能追到至少一个用例。

## 五、完整 Test Report 结构

1. 结论摘要：能否交付、失败、阻塞和剩余风险。
2. 被测对象：commit/ref、未提交差异、构建或实际运行代码。
3. 环境与测试数据：服务、配置、依赖、隔离方式和真实资源 ID。
4. 身份、角色与资源：调用者、所有权、租户/项目/记录关系。
5. 必要业务背景：状态转换、派生值、审计和错误码。
6. TestPlan 覆盖台账。
7. 逐用例执行记录。
8. 失败分析、根因证据状态和精确重测范围。
9. partial、blocked、skipped 与尚未完成的动作。
10. 清理证明。
11. 原始证据和重放入口。
12. 当前环境交接状态。

报告正文必须自包含地说明做了什么、怎么做、结果是什么。附件保存大体积原始证据，不替正文承担叙事。

## 六、测试现有能力而不只测试 diff

先列出改动直接触及的模型、边界和消费者。一个共享文件模型、权限入口或状态字段被重构时，所有依赖该结构的操作都需要影响分析，即使它们自己的代码没有修改。

影响分析可以得出三种结果：

- 行为明确被改变：新增用例或重跑原用例；
- 行为表面不变但依赖承重结构：运行回归用例；
- 入口、路径、数据和终态均不受新 delta 影响：记录理由后沿用旧证据。

这避免两个极端：只测新功能而遗漏消费者，或每次上游更新都机械重跑整个产品。

## 七、判断测试是否足够

分别判断：

- **完整性**：承诺、受影响消费者、风险和计划 ID 是否全部有状态；
- **正确性**：证据是否直接证明所声称的行为；
- **一致性**：需求、设计、实现、计划和实际观察是否仍一致；
- **可重放性**：另一个测试者能否依照报告复现请求、失败和终态；
- **环境诚实度**：实际运行代码、数据库、路由和依赖是否与报告描述相同。

测试结论不使用一个总体百分比掩盖 P0 缺口。任何成功声明都能写成：

```text
声明 → 用例 → 实际动作 → 终态证据 → 支持边界
```

无法形成这条链，就降低结论，而不是增强措辞。
