# agentcorp-next

**一个把自己写成目标与契约、而非脚本的软件交付组织。**

agentcorp-next 跑的是与 [AgentCorp](../README.md) 相同的那条可信交付流水线——需求
→ 测试计划 → 设计 → 实现 → 独立评审 → 验证 → 验收 → 交付——但对每一步*怎么做*
尽可能少说。它是为足够强的模型准备的：强到手写的流程反而要跟模型自己的规划正面
竞争，而且输。留下来的，是任何模型强度都替代不了的东西：这活为什么值得做、怎样才
算做完，以及所有人——代理、校验器、人类发起人——一致约定要在哪里会合的那组精确
坐标。

## 理念

给代理的指令语料里有三类内容，随着模型变强，它们的寿命各不相同：

| 类别 | 例子 | 在这里的归宿 |
| --- | --- | --- |
| **过程**——活儿怎么干 | 步骤序列、排查清单、把角色分类当成能力拐杖 | **删除**——模型规划得比脚本好 |
| **判断**——什么才算好 | 意图、价值、不变量、验收哲学 | **保留并压缩**——只留目标与边界，不留方法 |
| **契约**——约定好的坐标 | 交付物路径与形态、枚举、量化门槛、关卡机制、失败记录 | **原样保留**——一条约定的价值*就在于*它被共享且稳定 |

判定第三行的标尺：一个同样强的模型，独自作业，会不会自己独立得出同一个答案？更好的
实现计划——会，所以这里不规定任何计划。而路径 `review/research/00-index.md`、关卡词表
`approved | skipped | revised | blocked`、门槛"5–8 道测验题、其中一道非读 Background
一节不可"——不会。这些是任意选定的约定，它们的价值恰恰在于每个代理、每一次运行、
发起人的肌肉记忆都认同同一套。抹掉它们，每次运行就各自发明*一种*看似合理的格式，
而不是系统其余部分所期待的*那一种*格式。

失败记录——每份章程里那张"自辩说辞与其反驳"的表——也是契约：它们是被证伪的假设，
是关于这条工作流在现实中曾经从哪里断掉的事实。没有哪个模型能推断出一个系统的历史。

**首要指令：用户可见的效果不变。** 无论哪一代模型在干活，发起人看到的都是同一批关键
交付物、同样的路径、同样的形态，同样的关卡、同样的词表，同样的证据纪律。凡是发起人
永远看不到的东西——规划、拆解、推理——都归模型自己管。

## 一个任务如何流转

**deliver** 掌管整条流水线：它与发起人一起把"做完"钉死（成功标准、绝不可破的红线、
基线），把任务归入四种范式之一，并走完阶段序列——升起每一道人类关卡、把投入预算
编译成具体的数目、且从不给自己的交付物签字放行。一路上它召集其余各份章程：

- **research** 在需求定型之前把未知地界勘察成图；**facilitate** 把开放的方向逼成一个
  发起人真正认账的决定。
- **build** 产出设计、计划与代码——用正确的形态做最小的改动，每一行都可追溯到已获批
  的意图。
- **review** 在全新上下文里从十一个角度进攻结果，随后一道断路器式的 research 环节
  在任何修复落地之前对每一条发现做对抗式核验。
- **verify** 拿一份在代码存在之前就写好的测试计划，在这条主张所要求的环境里，证明
  被改动的行为。
- **teach** 确保发起人真的理解自己在批准什么——对要紧的改动，可上到一道以测验把关的
  走查。
- **compound** 把跑完的这一轮变成会改变未来任务的资产：回归测试与仓库规则直接落地；
  对系统自身的改动则是由发起人把关的提案。

每份章程内部的扇出、排序与用哪些工具，是模型自己的决定。隔离不变量则不是：一个改动
的作者永远不裁定对它的评审，证据以可打开的句柄传递，任何关卡都不会悄无声息地放行。

## 阅读顺序

| 文件 | 装着什么 |
| --- | --- |
| `constitution.md` | 跨越模型世代都成立的不变量——证据、隔离、同意、真相、契约、范围、留痕 |
| `artifacts.md` | 共享坐标系：任务布局、基线、frontmatter、每一套封闭词表、交接保真度 |
| `deliver.md` | 流水线掌管权：范式、关卡、投入编译、并行编排、交付报告 |
| `review.md` `build.md` `verify.md` | 三份重量级阶段章程 |
| `research.md` `facilitate.md` `teach.md` `compound.md` | 支援型章程 |

每份章程都是同一副骨架：**目标**（一句加粗的话）→ **判断**（它掌管的那些问题与价值
——不含方法）→ **交付物契约**（关键交付物：路径、必需形态、量化门槛）→ **失败记录**
（说辞 → 反驳，取自真实的翻车）→ **完成条件**（可检验）。

## 治理规则

- **删除。** 过程可以随便删。判断可以压缩但不可丢弃。契约与失败记录绝不因重写而被
  删除——契约只在深思熟虑后变更，且在同一次落地里更新每一个消费方。
- **重新加回。** 脚手架只有在"一个强模型不带它跑出了一条失败轨迹"面前才回来。"模型
  说不定用得上"不算一条轨迹。任何被加回的东西都把它那条轨迹贴在旁边，好让它日后能带
  着理由再被删掉。
- **等效。** 这套语料以效果、而非篇幅来验证：把跑完过的历史任务放进它里面重跑，再把
  产出的交付物集合与在产语料的产出对拆比对——同样的关键路径、同样的章节形态、同样的
  枚举值、同样的关卡。文字可以不同；坐标不可以。

## 沿革

agentcorp-next 把 37 个技能、约 10.6k 行的 AgentCorp 语料，浓缩进十一个文件、约 1k 行：

| 这里 | 吸收自 agentcorp/ |
| --- | --- |
| constitution.md | 那几条铁律 + "这个组织怎么思考"的箴言 |
| artifacts.md | workflow.md 的交付物布局、枚举、基线、交接规则 |
| deliver.md | delivery-orchestrator、workflow.md、intake、fresh-start-handoff |
| review.md | 11 位专项评审、code/plan/acceptance-review 主评、test-plan-reviewer、review-researcher |
| build.md | solution-architect、implementation-planner、implementation-engineer、review-fixer、comment-optimizer |
| verify.md | test-planner、test-leader、api-contract-tester、e2e-tester、regression-tester |
| research.md | probe、parallel-researcher |
| facilitate.md | brainstorm、grill |
| teach.md | explain、walkthrough |
| compound.md | compound、skill-evolution |
| （未纳入） | precommit-setup、authenticated-browser-session、semantic-core-translation——操作型工具；需要时按原样移入 |
