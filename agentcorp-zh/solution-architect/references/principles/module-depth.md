# 模块深度

## 描述

模块深度指功能与接口复杂度的比率。**深层模块**在简单接口后提供大量功能，向用户隐藏实现复杂度。**浅层模块**的接口相对于提供的功能过于复杂，在管理复杂度方面收益甚微。

目标是在最小化成本（接口复杂度）的同时最大化收益（功能）。深层模块是良好的抽象，因为只有一小部分内部复杂度对用户可见。

## 核心原则

### 深层 vs 浅层模块

- **深层模块**拥有简单接口，隐藏了显著的实现复杂度。它们让开发者无需理解内部细节就能使用强大功能（例如，Unix 文件 I/O 用 5 个简单系统调用管理数十万行代码）。
- **浅层模块**的接口与实现几乎一样复杂。它们没有隐藏多少复杂度，在没有补偿收益的情况下强加认知负荷。

### 接口设计

- 接口应比它暴露的实现简单得多。
- 接口包括形式元素（签名、类型）与非形式元素（行为、使用约束，在注释中记录）。
- 接口代表模块对系统其余部分施加的复杂度。更小、更简单的接口引入更少复杂度。

### 通用接口

- 适度通用的接口比专用接口更深。它们应支持当前需求，同时足够灵活以用于多种场景。
- 通用方法减少 API 中的总方法数，使其更易学习与使用。
- 专用接口在模块之间泄漏信息并创造紧耦合。UI 抽象不应出现在低层模块中。

### 常见场景简单化

- 接口应让最常见场景尽可能简单。默认行为应匹配大多数用户所需。
- 罕见场景可通过不 clutter 典型用户接口的可选机制来支持。

### 避免过度分解（Classitis）

- 把类拆成许多小片段会制造浅层模块并增加整体系统复杂度。
- 每个小类都有自己的接口，这些接口累积起来创造巨大的系统级复杂度。
- 小类常常导致代码冗长、样板过多。

## 危险信号

### 浅层模块指标

- **琐碎 wrapper**：简单委托给另一个调用而几乎没有转换的方法（例如，`addNullValueForAttribute(String attr) { data.put(attr, null); }`）
- **文档比实现还长**：如果解释接口比阅读代码更费力，它很可能是浅的
- **没有抽象收益**：接口暴露了实现的所有复杂度
- **一次性方法**：为单一特定场景设计的方法

### 接口复杂度指标

- **大量窄用途方法**：很多方法，每种服务一个特定场景
- **通用模块中的专用操作**：UI 概念（如"backspace"）出现在文本存储类中
- **样板蔓延**：要求多个对象或调用来完成简单任务（例如，Java 的 FileInputStream → BufferedInputStream → ObjectInputStream）

### 过度分解指标

参见 `cohesion-separation.md` 获取完整的合或分危险信号与标签集合。

- **类数量过多**：许多功能极少的类
- **穿透链**：仅为传递给其他构造函数而创建的对象
- **冗长设置**：常见操作需要多步初始化

## Issue Tags

在设计产出中标记模块深度问题时使用以下标签：

- **[Shallow Module]** - 接口复杂度几乎等于实现复杂度
- **[Trivial Wrapper]** - 方法没有提供有意义的抽象
- **[Special-Purpose API]** - 接口为单一狭窄场景设计而非通用场景
- **[Complex Interface]** - 接口为提供的功能拥有过多方法或参数
- **[False Abstraction]** - 接口看起来简单但隐藏了用户需要知道的重要细节
