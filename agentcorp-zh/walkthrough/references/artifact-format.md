# Walkthrough Artifact 格式

SKILL.md 讲为什么；这里讲怎么做。写 artifact 之前加载。

## 形态

- **HTML（默认）：** 一个自足文件——所有 CSS 和 JS 内联，不发网络请求，不用外部字体或 CDN。带章节标题和目录的长文单页；顶层结构不要用 tab。响应式程度足以在手机上阅读。代码放 `<pre>` 块（若改用带样式的 `<div>`，必须设置 `white-space: pre-wrap`）。图是带示例数据的内联 HTML/SVG——简化的 UI mockup、系统草图、before/after 对比；绝不用 ASCII art。关键概念、定义和边界情况用 callout 框。
- **Markdown（`output_format: md`）：** 同样的五个章节；图用 Mermaid（交付前验证语法）；quiz 的选项和答案放在折叠的 `<details>` 块里，答案在展开前保持隐藏。
- 命名：任务根目录下 `walkthrough/<slug>.html`，独立使用时 `teamspace/walkthroughs/<YYYYMMDD>-<slug>.html`。slug 描述变更，不描述分支。

## 章节契约

1. **背景**——教已经存在的东西，从宽到窄。从这个子系统的新人需要的起点讲起（它是干什么的、谁调用它、正常路径是什么），再收窄到变更落点的确切上下文。在读者能把旧世界装进脑子之前，不得出现变更本身的任何细节。所有任务内代号（story ID、finding ID、内部 artifact 名）首次出现时都要展开，或者更好：干脆不用。
2. **直觉**——在任何代码之前讲变更的本质。用平实语言说明变更解决的问题和解法的形状，然后**从头到尾走完一个玩具例子**：一个具体输入、旧代码怎么处理它、新代码怎么处理它、以及为什么这个差异正是重点。当带示例数据的小图更清楚时，优先用图而不是文字。
3. **作为 story 的变更**——按*想法*展开的顺序叙述变更：先讲关键决策，或先讲数据形状的变化——哪种顺序能让后续每一步都显得水到渠成，就用哪种。绝不按文件顺序，绝不按字母顺序。叙述把读者从一个 hunk 带到下一个：这一块为什么存在、它接下来解锁了什么。每个展示的 hunk 都标注真实的 `path:line` 锚点（取自实际 diff——不要手数行号）；不承载想法的 hunk（重命名、机械性连带改动）每个用一行总结，附路径，不粘贴。
4. **行为变化与风险**——紧凑而完整地说明现在哪些行为不同了：新的/变化的输出、错误行为、性能或兼容性影响，以及——最有价值的——与 diff 没有展示的既有代码路径的交互。没有内容的类别写"无"。覆盖要诚实：scope 内但没有解释的内容，在这里明确列为未覆盖，并给出原因。
5. **Quiz**——5–8 道选择题，每道 3–4 个选项，答案在读者作答之前保持隐藏（HTML：点击揭示；md：`<details>`）。每个选项揭示后都解释它为什么对或为什么错。难度：足以要求真正的理解，不设陷阱题，不出琐碎题。

## Quiz 机制

- 从 reviewer 的视角出题，而不是作者的视角："X 变了会坏什么"、"这与哪条既有路径交互"、"输入是 Y 时会发生什么"、"为什么选形状 A 而不是 B"。
- 瞄准意外面：依赖既有代码路径的行为、边界情况和失败模式、"追踪这个输入"式的题目。至少一道题必须需要**背景**章节的知识——这是你检测读者是否跳过了它的方法。
- 禁止：扫一眼 diff 就能答的题——文件名、行数、符号拼写、任何数数题。
- Gate：满分。答错时点名要重读的章节，然后就同一概念出一道**变体**题。在任务内时按标准 human-gate 词汇把结果记录到 `task.md` 的 Gate History：满分记 `approved`，sponsor 显式跳过记 `skipped`。独立使用时，在 artifact 末尾记一行「Gate outcome: approved <日期> / skipped by sponsor <日期>」。

## 交付前自检——命中任意一条就回去重写

- 第一个代码块出现时，读者还没被教过周围系统在干什么。
- 第 3 章的顺序照搬 `git diff --stat` 的输出，而不是想法的结构。
- 某个展示的 hunk 没有 `path:line` 锚点，或锚点是估算的而不是取自 diff。
- 一个你知道的行为变化既不在第 4 章，也不在未覆盖清单里。
- 任何一道 quiz 题不读 artifact 也能答（琐碎题），或没有任何一道题需要背景章节。
- artifact 依赖网络资源、外部字体，或 sponsor 可能没有的渲染器。
- walkthrough 被直接输出在终端里，而没有持久化成文件。
- 任何地方出现未解释的任务内代号（T-xx、F-xx、ST、内部 artifact 名）。
