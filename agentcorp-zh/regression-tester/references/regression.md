# Regression Testing 参考

在证明 bug 仍然被修复、或某项高风险行为仍然兼容时，使用这份参考。它规定每项检查怎么跑；verdict 和边界规则在 `SKILL.md` 里。

## "变更前失败 / 变更后通过"的执行顺序

对每个 bug 或高风险行为，在变更的两侧运行检查，顺序如下：

1. **说明目标。**写明要验证的 bug 或高风险行为，以及本应发现问题的那个具体检查——复现步骤、一个测试、一次请求、一条命令。
2. **获取变更前的代码。**checkout 到 base commit，或在本地 stash/revert 掉这次变更——取决于 working tree 允许哪种方式。"变更前"指这次变更之前的代码，不是当前代码树。
3. **证明检查在那里失败。**在变更前的代码上运行检查并捕获失败。这是让后面的通过有意义的前提；一个在变更前从未触发过的检查，对这次变更什么也证明不了。
4. **恢复变更并证明检查通过。**重新套上变更（checkout 回分支、unstash、撤销 revert），运行同一检查并捕获通过。
5. **如果变更前状态无法获取**——base commit 未知、变更无法干净地 revert、环境无法重建——写明原因，只在变更后的代码上运行检查，并把这条例外记录在结果 artifact 的 Residual risk 下。绝不把只跑了变更后一侧的运行当作前后对比的证明来呈现。

对从未坏过的高风险行为（兼容性而非 bug 修复），同一顺序照用，只是把"变更前失败"换成"变更前后行为一致"：先捕获变更前的观察，再捕获变更后的，并明确对比两者。

## 挑选周边检查

除定向检查之外，运行变更 blast radius 所要求的回归测试集。只要变更涉及多个 module、任何共享的 utility、schema 或配置，或某个公共 API 或 contract，blast radius 就算不小——此时必须把受影响模块或 contract 周边已有的测试拉进来一起跑。当某项高风险行为缺少覆盖时，把缺的检查写出来并运行；一个从未断言过该行为的绿色测试集是沉默，不是证明。

## 什么样的 regression 证据才算合格

- 原始复现步骤，以及现在观察到的结果。
- 一个变更前失败、变更后通过的测试；当变更前状态无法获取时，原因记录在 Residual risk 下。
- 命令、请求、截图、日志或 artifact。
- 从受影响模块或 contract 中挑选的周边检查。
- 明确的 pass/fail 状态；对 flaky 或依赖环境的结果，附上按观察记录的重跑历史。
