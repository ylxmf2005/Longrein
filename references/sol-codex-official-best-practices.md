# Sol 与 Codex 官方最佳实践：配置一个会持续变好的工程队友

> 主要来源：[OpenAI Codex Manual](https://developers.openai.com/codex/codex-manual.md)、[Codex Best Practices](https://learn.chatgpt.com/guides/best-practices)、[Prompting](https://learn.chatgpt.com/docs/prompting)、[GPT-5.6 Prompting Best Practices](https://developers.openai.com/api/docs/guides/latest-model#prompting-best-practices)。本文按实际工作流重新组织，不是逐字转载。
>
> 时效：2026-07-16。官方当前把 `gpt-5.6-sol` 定位为复杂编码、研究、计算机操作与高价值开放任务的旗舰模型；默认 Power 档使用 Sol + medium reasoning。

## 一、什么时候选择 Sol

Codex 的 GPT-5.6 系列分为 Sol、Terra 和 Luna：

- Sol：复杂、开放、模糊、高价值，重视分析、判断和成品质量；
- Terra：日常工程的能力与成本平衡；
- Luna：清晰、重复、批量的抽取、分类和转换。

不确定时，官方建议从 Sol 开始。但不是所有任务都需要 Sol 的最大深度。Sol 最适合：复杂代码修改、深度调研、重要评审、跨系统工作、精修文档和需要持续工具使用的 Agent 任务。

reasoning effort 应选择“足以做对的最低档”：

- Low/Light：快速、边界清晰；
- Medium：需要一定规划和检查；
- High/Extra High：多步骤、多来源、多取舍；
- Max：单任务的最困难问题，质量比速度和成本重要；
- Ultra：超出单代理运行，用子代理并行处理可拆分的大任务。

先用熟悉任务测试低一档 effort，再根据结果上调，比默认最高档更经济可靠。

## 二、第一条原则：描述结果，不要遥控每一步

官方 Prompting 指南建议从所需结果开始。只有过程本身具有约束意义时，才规定过程。

一个可靠的任务提示通常包含四项：

- Goal：要构建、改变或回答什么；
- Context：哪些文件、目录、文档、错误、示例和历史决定会改变结果；
- Constraints：架构、安全、兼容性、权限和不可破坏行为；
- Done when：哪些测试、行为和证据能够证明完成。

这四项不是必须填写的表格，而是检查提示词是否给了模型足够的方向。简单任务一句话就够；大型或高风险任务才需要完整结构。

过度规定实现步骤会使模型在新证据出现时仍沿错误路线执行。优先说明可观察结果和硬边界，让 Sol 自己调查代码库并选择路径。

## 三、提示词越强，不代表越长

OpenAI 针对 GPT-5.6 的内部编码 Agent 评测显示，精简 system prompt 在部分样本上既提高成绩，又显著降低 token 与成本。官方给出的数字是：约 10–15% 的评测提升、41–66% 的 token 降低、33–67% 的成本降低；这些数字只应当视为方向性结果，自己的工作流仍需单独评测。

精简方法：

1. 从一个已经有效的提示词和工具集开始；
2. 每次删除一组重复指令、示例或工具；
3. 对相同任务重新运行评测；
4. 每条规则只说一次；
5. 只暴露当前任务需要的工具；
6. 只有当示例表达产品要求或修复实测缺陷时才保留；
7. 同时观察初始上下文和长会话中的重复内容。

不要用大量同义句反复强调“先询问”“不要修改”“保持简洁”。重复规则会放大其权重，导致安全动作也频繁请示，或把有用信息压缩掉。

## 四、把授权范围写成一段紧凑政策

Sol 能主动、持续地完成多步骤任务，因此需要清楚区分不同请求授权了什么：

- 回答、解释、评审、诊断、规划：读取和检查相关材料，报告结论，不自动实施；
- 修改、构建、修复：执行请求范围内的本地修改，并运行相关非破坏性验证；
- 外部写入、破坏性操作、购买、权限变化、重大扩展范围：执行前确认。

同时明确哪些本地动作本来就是安全工作的一部分，例如读取文件、检查日志、编辑范围内代码和运行测试。授权规则集中写一遍，避免模型对预期动作也不断请示。

## 五、复杂任务先规划，但规划必须服务执行

当任务复杂、模糊或难以表达时，官方推荐先使用 Plan mode。它允许 Codex 先收集上下文、提出关键问题并形成计划，再实施。

另外两种方法：

- 让 Codex 采访你：一次问一个会改变架构的问题，挑战假设，把模糊想法变成可观察要求；
- 使用 `PLANS.md` 或 execution plan 模板：适合长时间、多阶段任务，把关键决策、验证和进度写入稳定文件。

计划的价值不是篇幅，而是提前暴露最可能变化的部分。好的计划应优先展示数据模型、接口、用户流程、风险和验证方法；机械重构可以简写。

规划完成后应推进执行。不要反复重述计划、重新争论已经批准的选择，或把“下一步会做什么”当成交付物。

## 六、用 AGENTS.md 沉淀真正重复的经验

一项提示模式被验证有效后，不要在每个任务里复制。把它放进 `AGENTS.md`。

官方把 `AGENTS.md` 定义为 Agent 的开放格式 README，适合保存：

- 仓库布局和重要目录；
- 启动、构建、测试和 lint 命令；
- 工程规范和 PR 预期；
- 禁止事项和安全边界；
- 完成定义与验证方法。

它可以分层：

- `~/.codex/AGENTS.md`：个人跨仓库默认；
- 仓库根目录：团队共享规则；
- 子目录内：局部规则，距离当前文件更近的规则优先。

短而准确的 `AGENTS.md` 优于长而空泛的规则集。先记录最基本的事实，只有反复出现真实错误时才加规则。Codex 同一个错误出现两次时，可以让它做 retrospective，再更新相应规则。

如果主文件越来越大，把计划、评审、架构等专门规则放入独立 Markdown，由主文件引用。

## 七、正确区分 prompt、AGENTS、config、skill 和 MCP

把行为放在最小合适的层级：

- 当前任务的特殊要求：prompt/thread context；
- 仓库长期约定：`AGENTS.md`；
- 模型、reasoning、sandbox、approval、MCP 等运行默认：`.codex/config.toml` 或全局 config；
- 可跨任务复用的工作流程：skill；
- 实时外部数据或动作：MCP server / app connector；
- 稳定且需要定期运行的流程：automation；
- 必须机械拦截的生命周期规则：hook。

不要把所有东西塞进 system prompt。事实、偏好、工作流、连接和执行政策应进入各自最合适的表面，既减少上下文噪声，也便于版本控制。

## 八、尽早把 Codex 配成真实工作环境

许多“模型质量问题”其实是环境问题：工作目录错误、没有写权限、默认模型不对、缺少工具或连接器。

官方建议：

- 个人默认放在 `~/.codex/config.toml`；
- 仓库特定设置放在 `.codex/config.toml`；
- 命令行覆盖只用于一次性情况；
- 新用户先保持默认 sandbox 与 approval；
- 只对可信仓库和明确工作流放宽权限；
- CLI、IDE 和桌面 App 使用同一组配置层级。

配置应尽早与真实构建、测试、浏览器、外部系统和权限匹配，否则模型会在错误环境中做出看似合理的判断。

## 九、验证不是附加步骤，而是完成定义的一部分

不要只要求 Codex “修改代码”。同时告诉它如何证明修改正确：

- 新增或更新相关测试；
- 运行适当测试套件；
- 检查 lint、格式化和类型；
- 对照用户请求确认最终行为；
- 审阅 diff，寻找回归和风险；
- 对无法运行的验证明确说明原因。

`/review` 可以针对基础分支、未提交改动或某个 commit 进行评审，也可以使用自定义 review 指令。团队可把 `code_review.md` 由 `AGENTS.md` 引用，以保持评审标准稳定。

验证结果必须基于实际输出。补丁工具报告成功不代表文件内容正确；测试命令退出零也不代表覆盖了目标行为。检查工具结果与验收标准之间是否真的建立了证据链。

## 十、为长程 Agent 建立持续性与透明度

OpenAI 对 GPT-5.6 长程任务建议三个核心机制：

### 1. 规划与持续完成

任务开始时拆出全部子请求，并在每次重要工具结果后确认还有哪些要求未完成。只有全部解决或确实被用户独有输入阻塞时才结束。

### 2. 重要工具调用前给简短说明

不是每次命令都写旁白，而是在重要阶段说明为什么要做：开始调查、准备修改、运行高价值验证、遇到改变方向的证据。这样用户能跟上决策，而不被操作日志淹没。

### 3. 使用 TODO 或 rubric 跟踪进度

多步骤任务应有可检查的计划或评分标准，防止完成一部分后提前收尾。计划状态需要随着证据更新，而不是保持最初假设不变。

## 十一、控制回答长度时，规定保留什么

GPT-5.6 默认比上一代更简洁。笼统的“尽量短”可能使它漏掉证据和限制。

更可靠的方式是给信息优先级：

- 先给结论；
- 保留支持结论所需的证据；
- 保留实质性风险和限制；
- 保留下一步行动；
- 优先删除开场、重复、泛泛安慰和可选背景。

API 中可以用 `text.verbosity` 设置默认细节等级，再用提示词说明当前任务必须包含的结构和内容。

语气也应描述具体写作行为，而不是只写“友好”：直接回答；用户报告问题时先承认具体问题；只在相关时安慰；不写通用夸奖和多余结尾。

## 十二、工具调用应按任务形状设计

工具越多不一定越好。只暴露相关工具，并准确说明输入、输出和错误行为。

Programmatic Tool Calling 适合代码可以批量处理大量中间结果并压缩为小型结构化输出的阶段，例如过滤、连接、排序、去重、聚合和验证。

以下情况优先直接工具调用：

- 一个调用就够；
- 中间输出本来就小；
- 每个结果都可能改变下一步语义判断；
- 动作需要审批；
- 最终必须保留原生引用或文件产物。

如果两种路线并用，要明确哪一段用程序化调用、允许哪些工具、输出 schema、并发和重试限制、停止条件，以及哪一段保留给模型的直接判断。避免完成后换路线重复工作。

## 十三、评测提示词，而不是凭感觉维护提示词

模型输出具有非确定性。生产提示词和模型升级都应该有代表性 fixtures、测试和评测套件。

OpenAI 建议：

- 生产模型固定到特定 snapshot，以获得一致行为；
- 提示构造放在代码里的小模块中；
- 动态值使用 typed arguments 或 schema；
- 修改提示前准备代表性样本；
- 通过常规代码评审和发布流程上线；
- 对需要渐进发布的改变使用配置或 feature flag；
- 比较正确率、完整性、证据、token、延迟、成本、调用和重试。

提示词、工具说明和 Agent 规则都是程序的一部分，应接受版本控制、测试和回归检查。

## 十四、把有效流程升级为 skill 与 automation

当一套工作不再只是一次性提示，而是包含固定步骤、参考资料、脚本、验证和输出格式时，应做成 skill。

适合 skill 的例子：

- 固定格式的代码评审；
- 发布前检查；
- 真实样本回归；
- 特定内部平台操作；
- 文档、表格、演示文稿生成；
- 研究和证据综合。

当流程已经稳定、输入来源明确、失败处理清楚，并需要按时间或事件重复运行时，再升级为 automation。不要自动化尚未被人工运行验证过的模糊工作流。

## 十五、可直接复用的 Sol 工作法

1. 用 Sol 处理复杂、开放、高价值的工作；清晰批量任务换更轻模型。
2. 提示词写目标、关键上下文、硬边界和完成证据。
3. 结果导向，不遥控可以由模型自行判断的实现步骤。
4. 复杂任务先计划或采访，再执行；计划不能替代执行。
5. 删除重复规则和无关工具，每次精简后跑相同评测。
6. 集中定义自主执行与确认边界。
7. 把反复有效的仓库经验写进短而准确的 `AGENTS.md`。
8. 把运行默认放 config，把工作流放 skill，把外部连接放 MCP。
9. 要求测试、检查、diff review 和可引用的完成证据。
10. 长任务用计划/TODO 保持完整性，在重要阶段给用户简短更新。
11. 回答长度按信息优先级控制，不用一句“尽量简短”解决。
12. 用真实任务评测模型、effort、提示和工具变化。

Sol 最适合被当作一个会调查、执行、验证并从团队规则中学习的工程队友。真正的提升不来自一条万能提示，而来自任务上下文、运行配置、持久规则、工具连接和反馈评测共同构成的系统。
