# test

## 对整条证据链负责

测试是一条从风险到证据的连续调查。先找出真正可能失败的地方，设计能观察它的环境和用例，实际触发行为并追到用户、数据与外部系统的终态，最后留下别人能够重放的证据。TestPlan 说明准备证明什么，Test Report 说明实际证明了什么；两者都由真实执行闭合。

计划里的每一个用例，都必须在报告里留下真实命运。HTTP 200、命令退出零、源码看起来正确和“页面显示完成”都只是中间信号；它们没有和用户结果、持久化状态或外部终态闭合以前，不是通过。

## 先形成能够执行的 TestPlan

从用户可观察的结果、原始失败、不能破坏的不变量、真实代码差异和全部受影响消费者中选择风险。不要只测新增分支：权限、文件、数据模型或公共接口被重构后，依赖它们但表面行为不变的操作也属于回归范围。

每个承重场景分配稳定的用例 ID，并写到另一个测试者能够直接执行的程度：环境和身份、前置数据、字面输入、请求或操作、预期结果、终态观察面、失败处理、清理和证据边界。缺陷修复必须包含原始失败输入，不用相似样例代替。

按风险组织覆盖，而不是平均覆盖或追求用例数量。优先考虑：

- 主路径与完整用户目标；
- 权限、所有权、租户和角色边界；
- 生命周期、删除/恢复、绑定/解绑和后续消费者；
- 错误传播、依赖半失败、重试、幂等、并发和补偿；
- 数据迁移、混合版本、回滚、审计和持久化；
- 高级用户会组合使用的操作，以及完成后的继续、复制、派生或再次编辑。

实现前写过计划时，完成代码后重新核对实际 diff。计划是风险地图；代码让风险移动后，更新用例，不忠实执行一张过期地图。

## 把难测变成可观察，而不是直接放弃

先准备在授权范围内可用的环境、身份和测试数据。必要时建立本地服务、隔离数据库、临时路由、日志查询、只读回读或最小 harness；不要因为测试麻烦就把源码推断包装成结果。

同时保持比例：低风险检查不需要复制生产系统；真实环境不可得时，完成仍能诚实证明的层，并准确说明缺少哪项观察、它削弱了什么结论。

选择与承诺匹配的表面：聚焦测试、集成/API、CLI、数据与迁移、真实 Agent/任务执行、浏览器 E2E、视觉检查或它们的组合。用户界面在范围内时用浏览器证明用户所见；纯接口产品不为满足形式强造前端测试。

## 实际执行并追到终态

每个用例先记录动作，再写结论：

- 准确命令、请求或编号操作步骤；
- 状态码、业务码、退出码和关键响应；
- trace、request、任务和资源 ID；
- UI、数据库、日志、审计、对象存储或外部接收方回读；
- 清理动作及清理后的确认；
- 本项证明什么，不能证明什么。

异步任务、回调、通知和计划执行必须声明观察窗口和终态来源。触发成功但 Manager、数据库、接收方或审计终态尚未确认时，状态是 `partial`，不是“基本通过”。

负向检查说明观察了哪个来源、多久、预期缺少什么信号。无法可靠证明“没有发生”时，保留证据缺口。

环境、权限或依赖缺失只阻塞依赖它的用例，其他安全、独立测试继续。已经跑过的证据可在影响分析确认新差异不触及其入口、路径、数据或终态时沿用；不要因无关更新机械重跑，也不要用旧证据覆盖明确改变过的行为。

## 覆盖状态必须诚实

TestPlan 中每个用例 ID 在覆盖台账中出现一次：

- `passed`：实际证据直接支持完整预期；
- `failed`：实际行为与预期冲突；
- `partial`：执行了一部分，但缺少计划要求的动作、终态、回读或清理证明；
- `blocked`：命名明确的环境、权限、依赖或输入使它当前无法执行；
- `skipped`：经风险判断主动不执行，并说明它削弱哪项结论。

“not run”不是可以静默留下的状态：要么是有理由的 `skipped`，要么是缺条件的 `blocked`。缺失用例 ID 是报告缺陷，不是隐式通过。

测试发现产品问题时，保存精确失败、证据和重测范围。用户同时要求修复时，先回到实现/修复工作，再只重跑受影响的用例；不要在测试报告里悄悄改产品再给自己判绿。

## Test Report 必须能够独立审查

第一页可以是结论摘要，但摘要不能替代正文。完整报告让读者无需翻十几个附件就能知道：

- 实际测的是哪份代码、未提交差异和运行构件；
- 环境、身份、测试数据和真实资源 ID；
- 必要的业务状态、所有权和错误码背景；
- 每个计划用例做了什么、预期什么、实际发生什么；
- 通过由哪条请求、UI、DB、日志或审计证据支持；
- 失败怎样复现，修复后重跑哪些用例；
- `partial`、`blocked`、`skipped` 分别缺哪一步；
- 数据是否清理，哪些进程、路由或临时配置仍在运行。

大日志、响应体和截图留在证据目录，正文解释它们证明什么并给出可打开路径。只有“上传通过”、passed 列表、缺陷名称或结果文件索引的文档，不是 Test Report。

非平凡测试、完整 E2E 或正式报告必须读取 [test-evidence-chain.md](references/test-evidence-chain.md)。

## 产物与权限

沿用已有任务工作区；没有时使用：

```text
studio/tasks/<task-id>/test/
├── test-plan.md
├── test-report.md
└── evidence/
```

一次性 harness、fixture、截图和重放脚本默认是本地测试证据。是否把测试源码或快照提交到产品仓库，由目标仓库规则和用户授权决定；不要因为测试有用就自动把它变成交付范围。凭据只安全引用，不写入报告。

完成时，所有承重行为都有实际证据，所有计划用例都有状态，失败与未执行范围可复现，清理和环境交接已经说明，并把 TestPlan、Test Report 与关键证据路径告诉用户。

面向用户使用其当前工作语言；代码、路径、协议字段和项目专有名词保持原样。
