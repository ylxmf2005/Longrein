# 回归测试参考

在证明 bug 仍被修复，或处于风险中的行为仍然兼容时使用本参考。它规定每个检查如何运行；verdict 和边界规则在 `SKILL.md` 中。

## 失败前 / 通过后序列

对于每个 bug 或处于风险中的行为，按以下顺序在变更两侧运行检查：

1. **明确目标。** 命名被验证的 bug 或处于风险中的行为，以及应捕获它的确切检查——复现步骤、测试、请求或命令。
2. **获取变更前代码。** 检出基准提交，或本地 stash/回退变更——以工作树允许的方式为准。"变更前"指变更前的代码，而非当前树。
3. **展示检查在那里失败。** 在变更前代码上运行检查并捕获失败。这使后续的通过有意义；变更前从未触发的检查无法证明变更修复了什么。
4. **恢复变更并展示检查通过。** 重新应用变更（切回分支、unstash、取消回退），运行同样的检查，并捕获通过。
5. **如果变更前状态不可获取** —— 基准提交未知、变更无法干净回退、环境无法重建——说明原因，仅在变更后代码上运行检查，并将例外记录在结果工件的 Residual risk 下。绝不要将仅变更后的运行作为变更前后证明呈现。

对于从未损坏的处于风险中的行为（兼容性而非缺陷修复），同一序列适用，将"变更前失败"替换为"变更前后行为一致"：捕获变更前观察，再捕获变更后观察，并显式比较。

## 选择邻近测试

除目标检查外，运行变更的爆炸半径所要求的回归套件。当变更触及多个模块、任何共享工具、schema 或配置，或公共 API 或契约时，爆炸半径是非平凡的——必须拉入受影响模块或契约的邻近现有测试。当处于风险中的行为缺少覆盖时，编写缺失的检查并运行它；绿色套件从未断言该行为只是沉默，不是证明。

## 什么是好的回归证据

- 原始复现步骤及当前观察到的结果。
- 变更前失败、变更后通过的测试；当变更前状态不可获取时，原因记录在 Residual risk 下。
- 命令、请求、截图、日志或工件。
- 从受影响模块或契约中挑选的邻近检查。
- 清晰的通过/失败状态，以及对于不稳定或环境依赖的结果，如实观察到的重试历史。
