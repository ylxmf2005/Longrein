# Fable 长程工程案例：人不再遥控步骤，但必须持续裁决

> 主要来源：[Simon Willison, “sqlite-utils 4.0rc2, mostly written by Claude Fable (for about $149.25)”](https://simonwillison.net/2026/Jul/5/sqlite-utils-fable/)，发表于 2026-07-05。本文按协作循环重新组织，不是逐字翻译。

## 一、起点只是一句高层目标

Simon 在手机上的 Claude Code for web 中给出了一句提示：对即将发布的稳定版 4.0 做最后评审，重点发现那些如果以后再修就会形成破坏性变更的问题。

这条提示没有规定要读哪些文件、运行哪些测试或按什么步骤评审，却包含了三个高密度信息：这是发布前最后窗口；破坏性兼容风险优先；目标不是普通代码清理，而是决定能否稳定发布。

Fable 首轮报告发现了五个“发布阻塞项”。最严重的一项是 `delete_where()` 不提交事务，并让后续写入也停留在未提交状态，数据库关闭后多项操作一起丢失。报告不仅命名问题，还给出了可运行的端到端复现。

## 二、真实工作不是“一条神提示”，而是 37 次裁决

整个过程包含 37 个提示、34 个提交、30 个文件，代码变化约为新增 1,321 行、删除 190 行。作者并没有预先遥控全部步骤。他在模型每次完成一段工作后检查结果、选择下一项、决定哪些设计变化值得接受。

这说明高层提示和人类参与并不矛盾：

- 模型获得了调查、实现和验证的路径自由；
- 人保留兼容性、设计取舍、风险排序和发布资格的裁决权；
- 交互发生在“新证据改变下一步”的节点，而不是每个机械动作之前。

用户介入循环的最佳位置，不是告诉模型下一行怎么写，而是在证据出现后回答：这个结果改变了什么？这个代价值得吗？这个行为是否属于我们愿意长期维护的契约？

## 三、文档是人重新取得系统理解的入口

作者提到一个非常实用的方法：先审阅模型修改的文档，用它建立这次变化的初始心智模型。正是在阅读事务文档时，他注意到 Python 3.12 的 `autocommit` 选项没有被真正处理；进一步测试发现，这个差异会导致几乎整个测试套件失败。

这不是“文档比代码更重要”，而是文档强迫实现声明自己的契约。人先读契约，再用代码和测试核验它，能够比从数十个 diff 文件开始更快地恢复判断力。

对 Skill 的启示是：交付物不仅要包含代码结果，还应提供一个足以让人重新参与下一轮决策的理解入口。Walkthrough、契约说明、变更日志和验证证据都属于控制回路，而不是包装。

## 四、新上下文复核确实能发现新问题

Fable 完成主要工作后，Simon 又让 GPT-5.5 以高推理档评审自上一个 RC 以来的变化，并确认 changelog。这个独立模型发现了两个值得调查的 P1 问题，包括一个方法在抛出错误前已经提交写操作，以及一个生成器未完全消费时事务提交行为异常。

作者原本认为“让一个模型评审另一个模型”带有迷信色彩，后来因为它反复发现有价值的问题，才把它变成习惯。

这里的关键不是模型品牌互换，而是新上下文没有继承实现者的路径惯性。独立复核者更容易把代码当成未知对象，重新从行为契约出发。

## 五、成本暴露了 Skill 编排的另一条边界

作者估算整轮成本约 149.25 美元，并明确反思：应该更多使用较便宜的子代理。强模型适合承重判断和复杂综合，不意味着所有检索、计数、分片检查都应由最昂贵模型完成。

下一代 Skill 应把任务分成两类：

- 需要开放判断、跨证据综合、架构取舍的工作；
- 边界清楚、可批量、可机械验证的工作。

前者交给强模型，后者可以交给轻量模型、脚本或并行检查。模型选择和工具选择也属于流程设计。

## 六、这个案例真正证明了什么

它没有证明“一句提示就能自动完成大型发布”。相反，它证明的是一套更成熟的循环：

1. 用高层目标和风险定义启动；
2. 让模型通过真实代码和测试制造具体误差信号；
3. 人在重要证据出现时裁决方向；
4. 文档帮助人恢复理解；
5. 新上下文复核实现者的盲区；
6. changelog、提交和测试把过程保存为可审查轨迹。

高能力模型减少了人类对执行步骤的控制需求，却增加了对证据、契约、成本和最终判断的要求。
