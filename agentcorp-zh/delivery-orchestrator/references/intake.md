# 本地分诊参考

当待处理工作以 issue、bug 报告、用户反馈或需要分配出去的模糊请求形式到达时，使用此参考。

## 接收原则

在分配任何单个事项之前先阅读整批。保留报告者的观察并添加分类元数据，而非将报告改写为你自己的理论。按用户影响和技术严重性分类，而非按报告者的职级或时间先后。在分配之前合并重复项，以避免产生冲突的实现工作。分配出去的工作项必须自包含，以便下一个代理无需阅读无关线程即可开始。

## 分类

| Issue 信号 | 类型 | 范式 |
| --- | --- | --- |
| 以前能用，现在坏了 | `bugfix` | `bugfix/hypothesis-driven` |
| 不符合文档或预期行为 | `bugfix` | `bugfix/hypothesis-driven` |
| 崩溃、数据丢失或安全漏洞 | `bugfix` | `bugfix/hypothesis-driven` |
| 用户想要一个尚不存在的能力 | `enhancement` | `enhancement/delta-design` |
| 用户希望现有行为以不同方式工作 | `enhancement` | `enhancement/delta-design` |
| 小的独立能力，不改变接口 | `addition` | `addition/simple` |
| 全新系统或重要子系统 | `greenfield` | `dev/architecture-first` |

当 issue 是设计争议、问题，或依赖你不具备的领域专业知识时，升级而非强制分类。

## 去重

强重复信号：相同的复现步骤；相同的错误消息出现在相同的界面上；相同的失败出现在相同的页面、端点、命令或工作流上；一个报告是另一个的子集。合并时保留最清晰的标题、最完整的复现步骤、最高严重性以及所有来源 issue ID。

## 优先级

| 优先级 | 含义 |
| --- | --- |
| P0 | 生产环境对大量用户不可用、数据丢失或正在发生的安全入侵。需要发起人确认。 |
| P1 | 重大用户影响或无变通方案。需要发起人确认。 |
| P2 | 有变通方案的真实问题，或有意义的功能缺口。 |
| P3 | 轻微摩擦、外观问题或积压改进。 |

对回归、阻塞项、安全问题和数据丢失提升优先级。每个优先级给出一句话理由。
