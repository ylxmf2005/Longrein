# 本地 AutoDev 工作流程

在协调工作时逐步使用此参考。它是 交付编排者 的本地工作流合同，并且不依赖于外部运行时目录。

## 经营理念

在路由之前定义“完成”：什么必须有效，什么不能破坏，什么超出范围。更改之前了解：为所选阶段组装足够的代码、测试、需求、问题或设计上下文。在开始工作之前展示阶段顺序——这就是管道承诺。保持作者/审阅者分离：工件的作者不批准自己的工件。将每个结果都视为证据——只有当命令传递证明了行为发生了改变时，它才是有用的。一旦达到成功标准就停止；不要将相邻范围吞入当前回合。每个阶段仅编写其所有者负责的部分，引用上游而不是重述它，并且仅在更改决策或避免实现/验证歧义时添加细节。所有作业、收据、清单和阶段工件都是带有 YAML frontmatter 的 Markdown 文件。

## 工作流程模式

每个任务在阶段执行开始之前选择一种模式。三种模式按**授权程度**排序；阶段词汇、工件路径和质量门禁在这三个阶段中保持相同 - 变化的是每个阶段的执行者和评审者：

|模式|默认？|它是如何运作的 |何时使用 |
|---|---|---|---|
|`直接` |否 |委托给无子代理。交付编排者自行执行每个阶段；对于审查类型阶段，它会从相应的审查角度产生**草稿**结论，并由该阶段的人工门禁批准——发起人是审查者。|小的、低风险的变化，发起人想要快速通道，或者主机环境没有子代理能力。|
|`部分委派`（默认） |是 |交付编排者直接执行非审查阶段；审查、审查研究和修复被委托给独立的角色。|例行任务、中小型更改或单个代理可以保留足够上下文的工作。|
|`全委派` |否 |每个可委托阶段都通过分配/回执文件委托给其阶段负责人，并且每个返回的制品都经过验证。|发起人要求 L/XL 工作、并行实施或需要独立作者且无法审查的阶段。|

默认为“部分委托”。切换到“完全委托”需要发起人的明确选择，或记录的编排理由：高复杂性、相互独立的并行模块、专用的执行环境或对作者分离的强烈需求（无法审查）。转为“直接”必须是发起人的明确选择或确认——它以发起人亲自裁决取代独立评审角色，并且发起人必须被告知并愿意成为评审者；永远不要默默降级为“直接”。

## 交互 Cadence

Cadence 与工作流模式相互独立：模式决定**谁执行、谁裁决**；cadence 决定**下一次让发起人看到检查点之前，要推进多少已经就绪的工作**。在 `task.md` 记录其中一种，发起人可随时切换。

- `continuous`（可执行请求的默认值）——持续推进当前 gate 允许的全部已就绪、可逆动作，把非紧急说明合并到有意义的检查点。“继续做”“你自行判断”“不要为常规审批停下来”会选择这个 cadence，也构成跳过可跳过人工暂停的持续授权；它不授权削弱证据，也不授权越过未批准的不可逆边界。
- `guided`——每次推进一份有意义的交付物或一个动作。在真实任务上采用**解释 → 执行 → 展示 → 暂停**，而不是另做玩具示例；任务大到难以讲清时，建议先切出更小的一块，但这只是软护栏，发起人可以覆盖。

两种 cadence 使用相同的质量关卡、作者分离、停止条件和交付物集合。`guided` 不是更慢的工作流模式，`continuous` 也不是一揽子预批准。

### 发起人工作路径菜单

模式是内部台账术语；先给出发起人需要的工作路径，单独说明交互 cadence，再把路径映射到模式：

|发起人看到的路径|内部模式|何时推荐|
|---|---|---|
|快速小变更|`直接` |任务量小、风险低，发起人明确愿意亲自裁决审查门禁；您必须明确，审查只是草稿，批准权在于发起人。|
|标准交付 |`部分委派`（默认） |默认推荐。适合大多数后端修复、增强和中小型要求；保持审查/研究/修复的独立性。|
|深度编排 |`全委派` |大变化、高风险、需要并行作者、跨越多个模块，或者发起人要求全角色编排。|

不要在入学时机械地向每个人展示所有三个路径。如果信号明确，只需说“我推荐标准交付”，并提供 1-2 个替代方案；如果任务很小，询问是否采取快速小额路线；如果任务显然很复杂，建议深度编排。当用户已经给出明确的模式时，直接采用它并重述后果。

开始任务时的对话骨架：```text
I'll treat this as <paradigm> for now. The success criteria are <1-3 items>, and the main risks are <1-3 items>.
I recommend the <path> route (internal mode: <mode>), because <reason>.
Next I'll run: <plain-language summary of the phase sequence>.
You can:
1. Continue per recommendation
2. Change to <another suitable path>
3. First supplement/modify the success criteria
```仅当信心较低、优先级/范围/风险接受不明确或路径选择会改变审稿人独立性时，才停止并等待发起人的选择；否则公布推荐路线并继续执行。

### 子代理调用参数

调用子代理时，默认继承当前主机/调用者自己的运行时配置。当克劳德本身生成克劳德子代理时，使用克劳德当前的默认值；当 Codex 本身生成 Codex 子代理/CLI/技能时，使用 Codex 当前的默认值。AgentCorp 的角色路由仅决定“分配谁、提供什么上下文以及将输出放在哪里”，而不负责对模型、推理工作、权限或其他运行时参数进行硬编码。

只有三种情况允许显式传递参数：发起者明确要求；分配/清单已经有记录的覆盖值；目标工具的模式需要它。即使需要覆盖，也只传递最少的必要参数，并在分配或清单中记录原因。

### 运行时路由

角色分为两个运行时层，均根据上述运行时继承原则调用，无需指定额外的模型参数：

- **Claude（决策层）**：交付编排者、测试计划员、测试计划审核员、解决方案架构师、实施计划员、计划审核主管、代码审核主管、测试主管、审核研究员、验收审核主管、对抗性审核员、并行研究员、简单性审核员、项目管理员审核员、品味审核员、注释优化器 — 遍历当前 Claude 环境的本机子代理/代理功能。
- **Codex（执行层）**：实施工程师、审核修复者、正确性审核者、安全审核者、性能审核者、可靠性审核者、标准审核者、API 合同审核者、API 合同测试者、E2E 测试者、回归测试者 - 遍历当前 Codex 环境的本机子代理/CLI/技能功能；当主机中不存在 Codex 通道时，降级为相同技能的本地子代理调用，协议不变。

三种模式中的任何一种都不能损害审查独立性；只有裁决者会发生变化：在“部分委托”/“完全委托”下，“测试计划审查”、“计划审查”、“代码审查”和“接受审查”始终转到其审查所有者；在“直接”下，这些阶段让交付编排者自行生成结论草案，但**草案并不意味着批准** - 每个审查阶段的人工门禁必须保持活动状态并由发起人裁决，并且在“直接”下不能跳过这些门禁。在任何模式下，交付编排者都不会批准自己的工件或证据。

处理代码审查结果的两个阶段“审查研究”和“修复”都在“部分委托”/“完全委托”下**委托**；交付编排者 本身不会验证或编写修复代码（在“直接”下，它会自行执行这些操作，但会保留 t相同的顺序和工件：首先进行研究，产生每个问题的判决，然后是发起人的人工门禁，然后修复落地）。分工：

-“审查研究”作为一个整体委托给“审查研究人员”：它独立地对抗性地重新检查每个发现，消除误报，并生成“审查/研究/”（判决+修复建议+每个问题的解释）。这是防止错误传播的断路器，必须独立且彻底地完成。
- `fix` 由 交付编排者 并行编排：Orchestrator 本身不会编写修复代码，而是 **根据文件所有权将已确认/部分有效的修复项目划分为相互不重叠的组**，为每个组调度一个“review-fixer”单个工作人员以并行地处理它们（确保两个组永远不会接触同一文件），并在所有组返回后，运行一个合并验证并将它们聚合到“review/fix-result.md”中。“review-fixer”是一个单独的修复工作线程，不会自行分区或分派。请参阅并行执行协议。

两者具有**顺序依赖性**：“修复”必须在“审查-研究”之后；`review-fixer` 仅使用经过验证的 `review/research/`，如果找不到它，它就会停止 - 它本身不会验证原始结果。这使得作者能够区分“独立验证”和“忠实着陆”。

在“部分委托”下，保持相同的阶段词汇和工件路径：交付编排者直接写入非审查工件，并将自己记录为“manifest.md”中的所有者，这些阶段可能会省略分配/回执文件；授权审查阶段仍保留分配/回执。在“完全授权”下，每个授权阶段都会经历完整的交接规则。在“直接”下，根本没有分配/回执：仍然需要所有阶段工件和清单条目，审查类型工件将清单中的所有者记录为交付编排者并标记为草稿，并且门禁结果记录发起人的裁决。

### 跨家庭第二意见（仅限高风险）

拥有最终裁决的三个角色——代码审查负责人（“代码审查”）、验收审查负责人（“接受-审查”）和审查研究员（“审查-研究”）——在决策层上进行裁决，即生成受审查工作的同一模型系列。在普通的改变中，这很好：作者/审稿人的分离和审稿研究断路器已经保持了独立性。但在一个高风险的变化中，由编写代码的同一家族做出的裁决也有其盲点，下游没有任何东西能够捕捉到两者都错过的东西。

高风险意味着安全或权限边界、公共/共享合约或数据丢失/不可逆转/难以回滚的发布。对于这样的改变——而且只有这样的改变——判决所有者从一个不同于做出判决的家庭的模范家庭中获得了独立的第二意见。对其结论提起诉讼。完全像运行时路由那样从主机继承另一个通道 - 当所有者在 Claude 上运行时的 Codex 通道，当它在 Codex 上运行时的 Claude 通道 - 并且永远不会命名特定模型：路由只知道“与现在执行的不同的系列”，从主机实际公开的任何通道中选择，而不是哪个系列。第二种意见是在独立切换原则下提出的（仅指针和路径，没有上游结论 - 请参阅“上下文保真度的两种类型”），冷读工件，并返回自己的结论；所有者将其作为一个输入记录在决策工件中，并且仍然拥有最终调用。

选择加入，它会诚实地降级：

- 高风险变更的默认：提供第二意见。如果主机没有暴露其他家庭频道，则退回到同家庭独立第二遍并在决策工件中记录单行原因 - 不阻止。
- 当发起人明确要求跨模型族的第二意见、但不存在其他模型族通道时，显式失败：停在关卡处明确报错，而不是让同一个模型族默默签字放行自己的工作。

普通的、非高风险的变更不会听取第二意见；他们的路由没有改变。

## Fix-Loop 协议（交付后快速修复）

当交付后（在更广泛的发布之前）出现小范围的缺陷并且原始诊断仍然成立时，交付编排者可以运行**轻量级修复循环**，而不是重新运行整个管道。仅在获得发起人同意的情况下进入，并在发布之前强制执行三个关卡：

1. **锁定根本原因**——原始诊断得到重新确认；如果发生变化，则升级到新任务，而不是继续修补。
2. **复制绑定** - 附加一个可复制缺陷（来自测试计划或新编写的）的可运行案例，并且针对原始失败输入（而不是代理样本）伪造修复。
3. **预发布SCM门禁**——在任何推送之前，确认目标分支、当前的“HEAD”和构建的工件都与任务一致；不匹配会阻止推送。

在修复轮次中保留一个小的**不变分类帐**（在“delivery/fix-loop.md”或清单中）：记录修复涉及的每个参数/配置，并在每轮中重新检查先前的不变量，以便以后的编辑无法默默地恢复早期的修复。不变违规或 SCM 不匹配是在发起人接受风险之前的硬停止。

## 任务分类

|信号|范式|
|---|---|
|新项目/系统、新的重要子系统或没有现有代码库 |`开发/架构优先` |
|缺陷、回归、不正确行为、崩溃、数据丢失或安全错误 |`错误修复/假设驱动` |

如有疑问，请选择“增强/增量设计”。我如果某个阶段的质量门禁暴露出分类不匹配，请在继续之前重新分类。

## 人工门禁策略

人为关卡是发起人的检查点，而不是阶段的质量关卡。跳过人工门禁只会消除发起人的停顿；它不会削弱前进所需的证据。

`blocked` 始终有明确范围：它只冻结依赖它的说法、动作或阶段跃迁，绝不表示发起人被阻塞，也不表示整项任务都要停止。凡是仍符合已批准意图、可逆且独立的工作都继续，并在 `task.md` 中同时记录被阻塞的阶段跃迁和可继续工作。发起人明确接受风险时，可以把交付结论改为 `delivered-with-risk`；绝不能把缺失证据改写成质量关卡已通过。

发起人说“继续做”“不要为常规审批停下来”或“你自行判断”时，选择 `continuous` cadence，并视为持续有效的导航偏好。在 `task.md` 中记录一次；跳过工作流允许跳过的所有人工暂停，在已批准范围内继续可逆决策，并把非紧急汇报推迟到下一个有意义的检查点，不再反复追问。这个偏好不能授权伪造证据、越权访问、自审自批，或在没有单独知情确认时执行破坏性/不可逆动作；这些约束只冻结受影响的阶段跃迁。

默认人工门禁：需求、测试计划、设计或诊断、实施故事规范、阻塞或有风险的审查/验证决策、审查研究结论和修复建议（在“修复”落地之前）、最终交付。

在“审查研究”之后和“修复”之前是一个自然的发起人检查点：在这里，发起人可以在清除“修复”之前确认哪些发现是真正的问题，哪些是误报，以及所提出的修复是否可以接受。对于小的、低风险的更改，您可以咨询发起人有关跳过此门禁的信息，但跳过不会更改“修复必须消耗经过验证的 review/research/”的依赖关系。

每个人类门禁允许的结果：

|结果|意义|
|---|---|
|`已批准` |发起人批准或表示继续。|
|`跳过` |发起人明确跳过此门禁。|
|`修订` |发起人要求变更；在继续之前重新运行或修改相应的阶段。|
|`blocked` |点名的阶段跃迁目前无法诚实推进；需要发起人输入、凭据、环境或风险接受。独立且可逆的工作继续。|

### 登机口导航菜单

进入人工门禁不要只问“批准了？”首先给发起人足够的总结以供判断，然后是简短的选项。默认格式：```text
Where we are: <phase/gate> has reached the human gate; this step decides <downstream impact>.
Evidence: <artifact paths + key conclusions/gaps, no more than 4 items>.
Blocked transition: <specific claim/action/phase that cannot proceed, or None>.
Still moving: <reversible independent work continuing now, or None>.
I recommend: one of <approved/skipped/revised/blocked>, because <one-line reason>.
Options:
1. Approve, proceed to <next phase>
2. Have me revise per <specific direction> and come back
3. Supply information / risk acceptance
4. Skip this human gate (only when this gate allows skipping; note that the quality gate is not relaxed)
```根据上下文定制选项：在 `direct` 下，审查型门禁不提供“跳过”；当置信度较低或凭据缺失时，受影响阶段跃迁的默认建议必须是 `blocked`；当审核 Owner 返回 `request_changes` 或 `needs_more_evidence` 时，默认建议必须是 revise/supply-evidence，而不是 approve。推荐 `blocked` 时，至少给出一条解法；只要存在可继续工作，就至少列出一项现在可以继续的动作。当发起人以自然语言回答时，将其映射为 `approved`、`skipped`、`revised` 或 `blocked` 并记录；仅当映射不清楚时才追问一次。

对于小的、低风险的变化，请咨询发起人是否跳过一些即将到来的关卡——说出具体的关卡，并保持审查独立。例如：“这是一个小的、孤立的更改；希望我跳过测试计划和设计的人工门禁，保留代码审查并在最终交付时报告？”

永远不要默默地跳过人类的门禁。在“task.md”和“manifest.md”的门禁历史记录中记录跳过的门禁。

当无人值守时（发起人缺席——自动化触发的预定作业，由另一个进程调用），没有 agent 可以代答人工关卡。发起人可以在运行开始前预先批准指定 gate（记录在 `task.md` 的 Gate History 中，视为 `approved`）；到达未预先批准的 gate 时，把待决问题和当前 artifact 路径写入 `task.md`，停下依赖该 gate 的分支，继续已预先批准或独立且可逆的工作，然后结束本轮并等待发起人裁决——“发起人可能会同意”不是越过关卡的理由。

即使跳过某个 gate 或需要完全自动化，以下情况仍然要求在受影响的阶段跃迁前暂停：需求置信度为 LOW 或成功标准不明确；优先级、范围或风险接受不明确；Review Owner 返回 `request_changes` 或 `needs_more_evidence`；验证失败或缺少必要证据；缺少凭据、环境或权限；最终交付状态需要报告。这个暂停不停止无关且可逆的工作。

### 阶段完成提示

每个阶段通过质量门禁后，仅向发起人报告有助于他们判断下一步的内容：

- “做了什么”：阶段名称+简单语言含义。
- “证据在哪里”：工件路径、关键测试/检查、收据或审查决定。
- `下一步`：下一阶段和所有者；如果有活动的人工门禁，请切换到门禁导航菜单。
- “可选重定向”：仅当新证据暴露范围、风险或分类变化时才提供；否则不要重新列出完整菜单。

发送回阶段时，也提供导航：说明发送回的原因、重新发送之前要更改的内容以及是否需要发起人的输入。连续两次发送回相同的阶段后，默认停止并重新评估分区或计划，并向发起人提供可选路线。

## 交付物重入与一致性

任务可在任何时刻修订，包括实现已经开始之后。交付物的创建顺序只是有用的阅读顺序；它不让后续交付物天然低一等，也不妨碍后续发现反过来修订更早的交付物。

1. **判断更新当前任务还是新开任务。** 目标意图不变、而新请求与已有工作高度重合时，修订当前任务。意图改变、范围膨胀成可独立交付的结果，或原任务可以独立完成时，新开一项有关联的任务。绝不能为了维持表面连续性，悄悄把一项任务变成另一份承诺。
2. **解析真实交付物集合。** 用 `manifest.md`、`task.md` 和 task root 枚举实际存在的具体文件；不要依赖记忆中的文件名、过期 assignment 或未解析 glob。仓库是当前代码行为的 source of truth；已批准任务交付物在 gate 被重开之前，是已批准意图的 source of truth。
3. **应用语义增量。** 只改新决定影响的内容，保留未触及部分，并保证幂等：同一个决定再应用一次，不应继续产生编辑。
4. **双向核对。** 读取被触及的交付物，以及所有依赖或约束其声明的现有交付物。设计改变可能反过来修需求，代码发现可能修设计和任务，验证证据也可能推翻更早的假设。记录矛盾、缺口和重复，不能只更新下游文件。
5. **路由并标记过期。** 每份修订交回它的 Owner。把受影响的 `manifest.md` 行标成 `needs_revision`，审批失效时在 Gate History 记录 `revised`，用更新后的 Action Context 替换过期 assignment；在交付物重新一致之前，依赖分支不得推进。
6. **把计划变化带进代码和证据。** 已有实现时，计划交付物一致并不等于完成：受影响部分要重进 implement、review 和 verification。无关且可逆的工作可以继续。

在 `task.md` 的 Artifact Coherence 下维护实时核对状态：触发修订、受影响交付物、当前状态和下一位 Owner。只有该台账为 `coherent`，任务才可以交付。

## 舞台所有者

- 交付编排者拥有所有三种模式的分类、模式选择、把关和最终交付。
- 交付编排者直接拥有“validate-requirements”并亲自编写工件 - 在进入该阶段时，加载“references/validate-requirements.md”（置信度、何时阻止、由发起人裁定的门；每个演示的形状、每个阶段目录的条形）。
- 在“部分委托”下，交付编排者还亲自编写“测试计划”、需要时的设计/诊断/合同、“实施计划”、“实施”和“验证”工件。
- 在“直接”下，交付编排者自行执行每个阶段；对于审查类型阶段，它会从相应的审查角度生成草案，并由发起人的人工门禁批准。
- 在“完全委托”下，测试计划者拥有“测试计划”；解决方案架构师流程ns 设计/分析工件；实施规划者拥有实施故事规范；实施工程师拥有“实施”；测试负责人负责验证协调；API 合约测试人员、E2E 测试人员和回归测试人员执行其指定的验证。
- “test-plan-review”属于测试计划审核者，“plan-review”属于计划审核负责人，“code-review”属于代码审核负责人，“acceptance-review”属于验收审核负责人——这在“部分委托”和“完全委托”下都适用；在“直接”下，这些审查的批准属于发起人的人工门禁。
- “审查研究”委托给审查研究员。当发现很多时，交付编排者会按代码域进行重复数据删除和集群，并并行调度多个审阅研究人员，每个研究员为其集群编写**每个问题**文件，然后编排者聚合“00-index.md”；工作人员不编写捆绑文件，也不自行扇出。“fix”同样由交付编排者并行编排：将文件分区为组，并行调度多个 Review Fixer 单个工作人员，运行合并验证，并聚合“fix-result.md”；Review Fixer 仅登陆为其分配的组，并且不会对其自身进行分区或分派。这在“部分授权”和“完全授权”下都适用，以保持作者区分“独立核查”和“忠实着陆”；在“直接”下，编排者按顺序进行研究和修复，并且通过让研究结论首先通过发起人的人工门禁来保留作者的分离。`fix` 总是出现在 `review-research` 之后，并且只消耗经过验证的 `review/research/`。

## 神器组织

当没有现有任务时，将“task_id”设置为“<YYYYMMDD-HHMMSS>-<desc-slug>”（时间戳在前，因此按名称列出的目录按时间顺序浏览），将“task_root”设置为相对于“workdir”的“teamspace/tasks/<task_id>/”，然后根据演示创建“task.md”和“manifest.md”。

## 阶段切换规则

对于委派阶段，交付编排者会在阶段开始之前写入分配文件。每个委托分配都带有一个“task_root”（相对于“workdir”的“teamspace/tasks/<task_id>/”）和一个相对于该任务根的“output_path”；当位置和工作空间不同时，相同的任务根也必须存在于`<code_worktree>/teamspace/tasks/<task_id>/`中。委托所有者将阶段工件写入分配的“output_path”，并写回指定工件路径和状态的 Markdown 收据。

派发前，在 assignment 的 Action Context 中填入具体值：source of truth；Owner 动手前必须读取的每个 context file；允许编辑的根目录；只读上下文；输出路径；行为约束。Context file 必须是实际存在的具体路径，不能是未解析 glob 或按惯例猜出的文件名。约束用于指导 Owner，不能原样抄进输出交付物。修订让 assignment 过期时，先替换它再重新派发，不要在旧 assignment 后面追加互相矛盾的指令。

对于收到的每张收据，交付编排者首先运行机械验证，然后进行质量判断 - 两者是分开的：

- **机械验证（信封一致性）**：运行“scripts/validate-handoff.py --pair <assignment> <receipt> --task-root <task_root>”（或在处理批次后运行“--sweep --task-root <task_root>”）。它检查收据的“artifact_path”是否确实存在，是否与作业的“output_path”匹配，“from_agent”/“phase”/“task_id”是否与作业对齐，工件的“author_agent”是否与所有者匹配，并且状态是否为非空。这一步解决了自由文本合同最容易出现的问题——“收据说已经完成，但工件不存在/错误的工件/缺失字段”（收据措辞≠实际情况）l 神器）。**非零验证退出被视为交接不完整**：将其作为“needs_more_evidence”发送回所有者，不继续进行下一步，也不将其视为门禁通过。
- **质量判断（阶段门禁）**：只有在机械验证通过后，编排者才会判断状态是否满足该阶段的质量门禁以及证据是否足够强大。通过机械验证并不意味着门禁通过了。当收据上有关于偏差、担忧或阻碍因素的注释时，请在做出门禁判断之前仔细阅读这些注释；不要只看状态字段。

发回不是逐字重试。发送回一个阶段后（机械验证失败、“request_changes”、“needs_more_evidence”或所有者报告被阻止），您必须在重新分派之前进行某些更改：提供其指定的缺失上下文、缩小或拆分任务、切换执行通道或升级到发起人 - 不进行任何更改的重新分派只会导致相同的失败。连续两次发送回相同的阶段后，停止并重新评估分区或计划本身是否错误，而不是重试第三次。

验证通过后，在“manifest.md”中记录分配、工件、收据、人工门禁结果和阶段质量结果，然后在工作区和位置之间同步更新的工件集。交付编排者会在活动的人工门禁处停止，直到发起人明确批准、跳过或重定向。

在“部分委托”下，交付编排者 亲自编写的阶段可能会省略分配/收据文件，但仍然需要阶段工件和清单条目；审查阶段仍然使用分配/收据，因为它们仍然是委派的。在“完全委托”下，委托阶段必须有分配/收据。在“直接”下，没有分配/收据，并且仍然需要所有阶段工件和清单条目。在任何模式下，经过验证的需求工件都是由交付编排者亲自编写的（请参阅“references/validate-requirements.md”）。在委托验证过程中，测试负责人可以在“verification/assignments/”下编写测试人员作业，测试人员在“verification/test-results/”下编写结果文件，测试负责人编写最终的“verification/verification-report.md”。

### 切换中的两种上下文保真度（耦合与独立）

“上游工件的名称和路径就足够了，引用而不是重述”的默认原则仅适用于一种切换，而不适用于所有切换。在编写作业并决定提供多少上下文时，首先确定它是哪种类型：

- **独立（独立/审查类型）**：`测试计划审查`，`计划审查`，`代码审查`，专家审查，`审查研究`，`接受审查`，验证类型。下游需要的是**独立的判断**，看到太多上游的结论其实是有偏差的（c合规性）。这里，保持默认的原则：传递**指针/路径**，让下游自己读取并判断；下游返回**提炼的结论**（发现/决定），而不是重新注入原始上下文。这保留了独立性并保存了上下文。

- **耦合（耦合/延续类型）**：`实施`、`修复`、`代码审查→审查研究→修复`修复主干，以及任何“下游建立在上游决策之上并导致错误级联”的交接。这里是**相反**：分配必须携带**完整的上游工件和先前的决策**（不是摘要，不是指针），因为操作携带隐式决策，并且未传递的隐式决策会产生冲突和返工。典型的必须包含的项目：“修复”必须包含“审查研究”针对该问题的**完整结论+根本原因+修复建议**（而不是一句“参见审查/研究/”）；“implement”必须携带已批准的 Story Spec 和相关合同的全文，而不仅仅是路径。

一行标准：**下游是否“独立地再次判断”或“建立在上游结论的基础上”**——前者传递指针以保持独立性，后者提供全文以保持连贯性。当不确定时，将其视为耦合（过度喂养比放弃决定更好）。

## 验证层次结构

1. 能力：每个模块必须具备，并且直接检查故障/边缘情况。
2. 集成/API：每个跨模块或公共合约流程都有成功和错误传播检查。
3. E2E：每一项面向用户的能力都至少出现在一个完整的用户目标中，既有快乐路径，也有错误路径。
4. 当变更涉及 UI 时，进行前端视觉/交互检查。

当所需的较低层检查仍然失败时，不要前进到较高层。当场景无法运行时，记录原因并将其视为间隙，除非明确接受。

## 并行执行协议

并行实现是“实现”阶段内的协议，而不是单独的阶段。仅当满足以下所有条件时才进行并行化：复杂度为 M/L/XL；至少可以独立构建两个子模块；子模块共享接口但不共享实现；架构或影响文档已经存在；当涉及公共/共享 API、模式、协议或跨模块合约时，“接口合约”是完整的；实施故事规范按合同划分任务，并仅保留每个子模块所需的集成上下文；每个子模块的 TestPlan 范围为 Must Haves、Need Haves、Failure/Edge Cases 和 Forbidden Zones。

任何并行或批量派发前，先形成一份汇总图：每个单元的硬依赖、required/provided contract、会触及的文件或表面、允许编辑的根目录。缺失硬依赖只阻塞依赖它的单元；重叠是必须通过合组、排序或互斥文件所有权解决的 warning，不会自动生成依赖。需要人工选择时，只展示一次汇总状态并获取一次确认。逐单元记录 success、skipped、blocked、failed；一个单元孤立失败后，其余独立单元继续。

每个并行实施会话均接收以下输入：

1. `STORY_SPEC_PATH`：已批准的实施故事规范的相对路径。
2. `DESIGN_DOC_PATH`：架构、影响、诊断或接口契约的相对路径。
3. `OWN_CONTRACT`：此会话实现的合约存根。
4. `DEP_CONTRACTS`：该会话调用但不执行的只读合约使。
5. `TESTPLAN_SCOPE`：该子模块的能力、边界、必须具备、需要具备、故障/边缘情况。
6. `FORBIDDEN_ZONE`：该子模块的显式红线。
7. `COMPLETION_SIGNAL`：协调器期望的确切完成信号。
8. `INTEGRATION_CONTEXT`：涉及该子模块的集成测试和跨界期望。

### 并行化`review-research`（按代码域进行重复数据删除分区，输出仍按项目写入）

当发现很多时，“审查-研究”也由交付编排者并行协调。这里必须注意一个区别：**调查单元可能是集群的，但输出单元始终是每个项目。**集群的存在只是为了“不会重复读取同一批代码”，并且绝不能将输出粒度折叠到集群文件中。

1. **获取结果**：获取所有结果以从代码审查结果中进行验证。
2. **按代码域进行去重和合并到集群**：当审阅者的多个发现指向同一文件/同一调用链时，将它们分组到一个集群中（例如“转换轮询”、“资产克隆锁定”、“批量状态合同”）。集群的存在使得一个工作人员可以读取共享一批代码一次并涵盖多个结果，而不是每个结果生成 33 个结果，然后重新读取相同文件的代理。
3. **并行调度`review-researcher`：每个集群一项作业，给出`FINDINGS`（该集群中每个发现的全文+审阅者证据，仅作为线索）、`CODE_SCOPE`（该集群的相关代码范围）、`DESIGN_PRINCIPLES`（记录的设计原则）、独立对抗性重新检查的规则和`OUTPUT_DIR=review/research/`。**将硬约束写入作业：工作人员为其集群中的每个发现写入一个每个问题文件“review/research/<number>-<slug>.md”，并且不允许写入捆绑/集群文件。** 遵守线束并发限制；如果不支持并行，则采用串行，协议不变。
4. **聚合索引**：每个工作人员为其少数发现编写每个问题的文件并返回收据。全部返回后，**Orchestrator 会聚合所有每个问题文件中的 `review/research/00-index.md`（索引形状，与评论研究人员自己的相同：每个发现一行，排序为 P0→P1→P2 — 一句话摘要 + 判决 + 一个空的人类决策列 + 一个指向每个问题文件的链接）。Orchestrator 不会发明合并的“SUMMARY.md”，也不会为索引分配自定义“artifact_type”——索引为“00-index.md”。
5. **仅在人工审查门禁之后进行修复**：默认情况下，在审查研究人工门禁处停止，让发起人确认判决和修复，然后继续“修复”；不要直接从研究到修复。

收到收据时，像往常一样使用“scripts/validate-handoff.py”进行验证（工作人员收据的“artifact_path”指向它生成的每个问题文件之一或“review/research/”），然后索引已聚合，运行完整的“--sweep”传递。

### 并行化“fix”（相同协议，按文件所有权分区）

“fix”也是由 交付编排者 编排的并行协议，而不是单独阶段之外的特殊情况。前提：`review-research`产生了`review/research/`，其中有判断为**确认/部分有效**的修正项。Orchestrator 执行以下操作：

1. **获取要修复的项目**：从`review/research/`中获取所有已确认/部分有效的项目及其修复建议（覆盖人工评论；误报和“需要人工确认”未修复）。如果找不到`review/research/`，请先停止并运行`review-research`。
2. **按文件所有权划分为互不重叠的组**：列出要更改的每个项目的主文件（以及可预见的溢出文件）；**接触同一文件的项目进入同一组**，由同一工作人员串行处理。两个组的文件占用集必须是不相交的。
3. **并行调度`review-fixer`单个工作人员**：每组一项分配，准确给出这些输入 - `GROUP_SLUG`、`FIX_ITEMS`（该组已确认/部分有效的项目和研究修复建议）、`OWNED_FILES`（该组有权编辑的文件集，其他组不会触及）、`REPO_CONVENTIONS`、`FOCUSED_VALIDATION`（重点关注的项目）检查该组应该运行），“OUTPUT_PATH”（“review/fix-records/<group-slug>.md”）。遵守线束并发限制：超过限制时排队并在插槽空闲时填充；如果线束不支持并行性，则转为串行，协议不变。
4. **收集+合并验证**：每个工作人员写入其组的“fix-records/<group>.md”并返回收据。全部返回后，Orchestrator 对合并的更改（语法/导入/类型/相关测试）运行一次完整存储库验证，以捕获跨组交互。发生在更改的文件上的故障→将相关组送回重做或升级；仅发生在无人更改的文件上的故障 → 将其记录为预先存在的故障。
5. **聚合**：Orchestrator 编写 `review/fix-result.md`：按 P0→P1→P2（作为需求研究/需求人类/误报参考落地/发回）聚合每个组的处置、合并验证结果和残留风险。该聚合属于 Orchestrator，而不属于任何单个工作人员。

工作人员不会在文件上发生冲突，因为他们的占用集是不相交的；如果某个工作线程的修复溢出“OWNED_FILES”，它必须停止并报告而不是跨越边界，并且 Orchestrator 会重新分区或连续重新运行。

## 经验教训

管道在“deliver”处终止，但课程会在“teamspace/learnings/”中的任务中保留 - 这是交付编排者的内置功能，而不是阶段，没有门禁，并且不会更改阶段目录。两个操作：在“intake”/“validate-requirements”开始时，通过任务关键字 a **搜索**先前的课程并将相关条目按路径馈送到下游分配中；在交付总结时或当合格课程在任务中出现时（意外的根本原因、重复返工、回购陷阱），**捕获**它。条形、形状、重复数据删除和回流规则位于“references/learnings.md”中。

## 总结导航

“交付”不仅仅是宣布完成。最终答复和“delivery/delivery-report.md”都必须让发起人知道任务是否可以结束、证据在哪里以及还有哪些自然后续行动。默认结构：

1. 状态：已交付/已交付但有风险/已阻止/已拒绝。
2. 交付内容：代码位置、关键工件路径、密钥验证。
3、偏差和残余风险：无则写无；否则给予业主或验收条件。
4. 建议下一步：一项明确的建议。
5. 可选后续措施：根据需要列出 2-4，例如关闭任务，创建后续任务，运行“演练”（发起人理解+测验门禁）或“变更逐块审阅者”（对本地 forge 的每个大块审核评论），进行另一轮验证，捕获/审查学习内容，返回门禁进行修改。

如果验收未通过或证据不足，建议的下一步不能“平仓”；它必须指出提供证据、修改、重新审查或发起人风险接受。
