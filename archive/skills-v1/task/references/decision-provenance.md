# 决策来源与修改权

决策标记表达的是谁拥有最终修改权，不是谁最先提出想法。用户直接作出的明确决定，以及 Agent 提议后被用户明确选择为后续约束的决定，都是 `UD-*`；查看、提问、参与讨论、表示知悉、审核、批准产物或没有反对，不会把 Agent 决定自动变成用户决定。文档中的 `UD-*` 标签只能记录已有授权，不能创造授权。

## `UD-###`：用户决定

会约束后续工作的用户价值、偏好、范围、非目标、完成条件、风险接受、兼容承诺和重要取舍，落入 Task Context 时必须分配任务内唯一的 `UD-###`，并在对应条目前缀标记。编号按当前最大值递增，已经失效的编号不复用。用户的问题、试探、模糊愿望和未确认候选不是决定，不分配编号。

Agent 已经向用户报告变化，也只有在用户明确选择了准备采用的范围、取舍、兼容或风险后才形成决定。用户已经明确选择但当前 Task Context 或专业产物漏记时，根据可追溯原话补齐源状态，不为了补文档要求用户重复作出同一个决定；无法取得原始表达且差异会改变承诺时保留证据缺口。

YAML 字段或表格值不能自然加前缀时，在行尾注释或 Evidence / Source 单元格写入 `UD-*`；不要为同一个决定再建一份平行正文。

没有 Active Task 时不伪造持久编号；在用户可见回复中把已经确认的内容明确列为“用户决定”。用户以后显式启动 Task 时，由 `shape` 把仍然有效的决定写入该 Task 的 `UD-*`。

Agent 可以用事实和专业判断挑战 `UD-*`，但不能静默改写。新的用户表达与现有 `UD-*` 冲突时，先指出冲突和影响，询问是否明确替换；确认后创建新的 `UD-*`，先修订受影响的专业产物，再通过带 `decision: UD-###` 的 `longrein_task_context` 更新承诺，并在事件摘要中写清 `supersedes UD-*`。Runtime 增加 `scope_revision`，旧决定继续在 [Task Timeline](task-timeline.md) 中可追溯。

## `AD-###`：Agent 决定

Agent 为满足 `UD-*` 和真实证据作出的关键设计、实现、评审或验证判断，保留在对应专业产物中，并写明 `Based on: <UD-*、事实或证据路径>`。`AD-*` 在所属产物内编号即可；跨产物引用时同时写文件路径。

新证据、实现反馈或评审结论可以直接修订 `AD-*`，不需要为了保护旧方案取得用户批准。只有修订会改变 `UD-*`、任务承诺或授权边界时，才回到 `shape` 形成用户决定。

只标记会被后续工作依赖、值得评审或可能被替换的关键判断。局部命名、机械编辑和可随代码自然调整的细节不建立 `AD-*`。

已有决定不因进入新 Phase、Review 或 Task 就复制、重新编号或重新讲解；当前工作通过原路径和 ID 引用它们。只有新证据推翻、当前变化准备修改，或其来源会影响本次授权时才重新核验。历史选择若只体现在代码中，先作为系统事实或既有契约调查，不从代码形态反推一个虚构的决定标记。

## 三种变化分别处理

- 事实错误：用证据纠正，增加需要的 `context_revision`；它不是任何人的决定。
- `AD-*` 不成立：专业 Skill 直接修订，记录替换关系和证据。
- `UD-*` 需要改变：Agent 提议并说明代价，只有用户明确确认后才能替换。

用户决定受到保护，但不免于质疑；Agent 决定可以自由修订，但不能越过用户决定。

## 让来源在长上下文中保持可验证

每个进入 Task Context 的新 `UD-*` 都通过 `longrein_task_context.decision_sources` 保存三项内容：决定 ID、原始用户消息或 transcript 的稳定位置、足以授权该决定的逐字原话。Agent 的总结、推荐、requirements、design、代码形态和压缩摘要都不能充当用户来源；同一段原话可以支持多个决定，但每个决定都要能从原话本身推出，不能用“用户允许调查”支持“用户授权实现调查中出现的所有能力”。

对话压缩、交接或恢复后，以 Runtime 中的 Decision Sources 和原始消息为准。摘要必须分开写用户已确认决定、Agent 提议、事实发现和开放问题；找不到来源的旧 `UD-*` 只能作为待核验承诺，不继续扩大下游行为。实现反馈提出新的拒绝行为、公共面、兼容承诺或范围时，先形成 finding 或候选决定，不通过补写 REQ、AD、Plan 或摘要追认授权。
