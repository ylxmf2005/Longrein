# Test 证据链标准

本文件用于非平凡测试、跨系统变化、异步终态、完整 E2E 和正式 Test Report。它保存详细契约；小改动只采用与风险有关的部分。

## 目录

1. 测试事实账本与风险镜头
2. 可执行 TestPlan 与逐用例记录
3. 缺陷、回归和公共契约
4. 浏览器、异步、并发与增量重测
5. 覆盖台账、总体结论与 Test Report
6. 产物与提交边界

## 一、先建立测试事实账本

执行前记录并核对：

- 被测对象：source/working/target 意图、inspected revision、未提交差异、构建物、镜像或实际进程；
- 环境：入口、端口、路由、配置模式、依赖服务、数据库、对象存储、缓存和隔离措施；
- 身份：persona、调用方、角色、owner、tenant/project/workspace 关系和凭据引用；
- 数据：真实资源 ID、创建来源、允许修改的范围和清理责任；
- 观察面：UI、API、CLI、数据库、日志、trace、审计、消息或外部接收方；
- 已知限制：进不去的页面、不可用服务、无法取得的权限和因此不能证明的结论。

读取已有的稳定环境说明和历史报告，但对本轮关键事实做轻量回读。文档与现场不一致时，以现场为准并修正文档；不要继续在旧地图上累积证据。

## 二、风险覆盖镜头

从任务承诺、Task Operating Envelope 和实际影响面选择相关镜头，不机械全跑：

| 镜头 | 需要证明的事实 |
|---|---|
| 主路径 | 用户能否从真实入口完成完整目标，最终状态是否正确 |
| 错误路径 | 无效输入、冲突、超时和依赖失败如何显式传播 |
| 权限与所有权 | 不同身份只能读写应拥有的资源，拒绝语义稳定且不泄露信息 |
| 生命周期 | 创建、读取、更新、软删除、恢复、硬删除和后续引用是否一致 |
| 组合操作 | 相邻操作是否因共享模型、权限或状态变化而改变 |
| 并发与幂等 | 重复请求、并发更新、回调重放和部分失败会否重复、丢失或卡死 |
| 数据与迁移 | 默认值、回填、新旧版本、回滚、审计和持久化终态 |
| 公共契约 | 请求/响应、错误语义、CLI、schema 与已有消费者兼容性 |
| 高级用户 | 合法但不常见的组合、长流程、完成后继续操作和跨资源关系 |
| 回归 | 原失败、受影响现有行为和明确不应变化的边界 |

表中的能力名称只是例子。测试真实产品，不为覆盖表格发明功能。

## 三、可直接执行的 TestPlan 用例

```markdown
### <用例 ID> — <场景>

- 背景与风险：为什么测，漏掉后会怎样。
- 优先级：P0 | P1 | P2，或项目已有等级。
- 环境与身份：入口、调用者、权限、依赖、允许的写入。
- 前置数据：真实资源或可直接执行的创建步骤。
- 实际动作：请求、命令或编号 UI 操作，保留字面输入。
- 预期：用户结果、status/business code 和持久化或外部终态。
- 观察面与窗口：在哪里、多久、用什么标识确认。
- 证据：响应、截图、SQL、日志、审计或外部确认的保存位置。
- 失败处理：停止、继续，或阻塞哪些后续用例。
- 清理：恢复动作和确认方法。
- 证据边界：通过后仍不能证明什么。
```

“调用列表接口并检查结果”不合格；写出方法、路径、参数、payload 和断言。页面操作使用可见文字和所在区域，不把脆弱选择器当成测试意图。对于原始缺陷，原输入、前置状态和失败表现必须逐字保留。

### 执行前攻击计划

在开跑前检查：

- 产品仍然错误时，是否存在全部用例变绿的路径；
- 每个 Must-Have 是否被直接观察，而非由邻近信号推断；
- 是否把 API 成功误当成 UI、异步任务或外部送达成功；
- 是否遗漏旧调用方、权限边界、生命周期消费者或失败补偿；
- 测试者是否仍需现场发明输入、数据或步骤；
- 清理失败是否会污染后续结论。

有一个答案为“是”，就先修计划或明确证据缺口。

## 四、逐用例执行记录

```markdown
### <用例 ID> — <场景> — <status>

- 背景 / 风险：
- 前置条件：
- 预期：
- 实际动作：
- 实际响应与观察：
- 终态回读：
- 清理与清理回读：
- 证据：
- 证据边界：本项证明……；不能证明……
```

每项先记录发生过的动作和观察，再下结论。记录实际时间、目标和资源标识；秘密脱敏，但不能删掉请求或操作本身。

## 五、缺陷与回归：让结论在变化两侧成立

能安全获得基线时，对原始缺陷使用同一输入和观察方式：

1. 明确基线与目标版本；
2. 在修前版本重放，捕获预期失败；
3. 在修后版本重放，捕获通过与终态；
4. 运行爆炸半径内的相邻检查；
5. 保存两侧命令、环境和证据，避免比较了不同条件。

若基线不可获得、无法安全切换或历史环境不能重建，只运行修后检查并把缺失的修前证明写为残余风险，不能称为完整的“修前失败、修后通过”。对从未损坏但可能回归的行为，比较变化前后的相同行为与契约。

绿色套件只证明它实际断言的内容。处于风险中的行为没有检查时，新增本地验证或采用其他直接观察；是否提交测试源码仍由用户授权和仓库规则决定。

## 六、API、CLI 与公共契约

真实契约结论来自打到真实目标的请求。mock、stub、生成服务器或静态 schema 只支持相应层级，不支持运行时兼容声明。

只对真实存在且 Task Operating Envelope 允许到达的表面选择正向和负向探针：

- HTTP：方法、路径、认证、头、请求/响应形态、状态码和业务码；无凭据、过期或越权凭据、缺字段、错类型、畸形 JSON、超大输入和不支持方法；
- RPC/A2A：方法、参数、id 回显、错误对象、流式中断和协议扩展；
- CLI：参数、stdin、stdout/stderr、退出码和机器可读错误；
- SDK/schema：旧调用形态、可选字段、省略字段、返回对象和版本兼容；
- 错误契约：是否把真实失败伪装成 200、空数组或静默降级。

保留准确请求、凭证模式、环境、status/code、关键 body、trace/request ID 和持久化回读。普通客户端无法复现浏览器会话、SSO 或 CSRF 行为时，从真实登录态执行，并说明证据只适用于该会话模型。

## 七、浏览器与完整用户旅程

用户界面属于承诺时，浏览器状态和真实交互是主要证据，API、DB 和日志用于解释与闭合终态。每条旅程采用“观察起点—执行动作—等待稳定—再次观察”的节奏，直到用户目标真正完成、明确失败或被阻塞。

只有真实用户表面与 Task Operating Envelope 需要区分使用方式时才选择 persona：

- 新手：依赖可见标签，容易在入口、反馈和恢复上受阻；
- 高级用户：组合功能、跨资源操作、继续编辑和追求一致性；
- 对抗性用户：重复操作、无效输入、越权、顺序变化和信息泄露。

页面记录包括 URL、字面输入、编号步骤、每一步实际状态、截图和必要的控制台/网络证据。纯接口产品不需要为形式制造浏览器测试；API 也不能冒充用户实际看到的页面证据。

## 八、异步、回调与外部终态

对任务、Plan、通知、调度、回调、消息和审计等行为，至少记录：触发时间、关联 ID、观察窗口、轮询或回调路径、中间状态、最终状态、持久化回读和外部确认来源。

区分三层：

```text
请求被接受 → 工作实际执行 → 用户/数据/外部系统到达终态
```

前一层不能自动证明后一层。观察窗口结束仍未闭合时，根据已完成部分标为 `partial`，或因关键条件完全不可得标为 `blocked`。

## 九、并发、重试和 flaky 结果

对幂等、回调重放、补偿和并发风险，记录参与请求、顺序或并发方式、各自响应、最终唯一状态和是否留下重复副作用。只检查单次成功不能证明重试安全。

出现不稳定结果时保存每次运行：时间、环境、命令、退出码、失败点和重跑结果。先判断是产品间歇性错误、环境不稳定还是测试本身不可信；在证据不足前使用 `partial` 或 `failed`，不要通过重复运行把历史红色洗掉。

## 十、证据沿用与增量重测

已有证据可以沿用，但必须先比较新 delta 是否改变：

- 测试入口或调用者；
- 经过的代码、配置或依赖；
- 数据模型、权限、状态转换或错误契约；
- 终态观察面和清理方式；
- 实际运行环境或构建身份。

任一项改变，重跑相关用例；均未改变时，在报告记录影响分析和沿用的证据来源。不要把“上游更新了”本身当成重跑理由，也不要把“文件没改”当成不受影响的充分理由。

## 十一、覆盖台账与总体结论

| 用例 ID | 场景 | 优先级 | 状态 | 执行记录 | 最强证据 |
|---|---|---|---|---|---|
| `<id>` | `<risk / user goal>` | `<P0/P1/P2>` | `passed / failed / partial / blocked / skipped` | `<anchor>` | `<path / link>` |

实际改动产生新风险时新增用例并说明来源。原用例失效时保留编号并标记 `skipped`，说明替代项。每个 Must-Have、受影响消费者和承重风险都能追到至少一个用例。

总体判断分别检查：

- **完整性**：承诺、消费者、风险和计划 ID 是否都有状态；
- **正确性**：证据是否直接证明所声称的行为；
- **一致性**：需求、设计、实现、计划和实际观察是否仍一致；
- **可重放性**：另一位测试者能否复现请求、失败和终态；
- **环境诚实度**：实际运行代码、数据库、路由和依赖是否与报告一致。

总体通过率不能覆盖承重失败。任何成功声明都应形成：

```text
声明 → 用例 → 实际动作 → 终态证据 → 支持边界
```

报告总体状态使用 `passed | failed | partial | blocked`，不使用通过百分比替代风险判断。`skipped` 只属于用例状态；它必须说明被放弃的证明范围，并参与总体结论。

## 十二、完整 Test Report

1. 结论摘要：能否交付、失败、阻塞和剩余风险。
2. 被测对象：commit/ref、未提交差异、构建或实际运行代码。
3. 环境与测试数据：服务、配置、依赖、隔离方式和真实资源 ID。
4. 身份、角色与资源：调用者、所有权、租户/项目/记录关系。
5. 必要业务背景：状态转换、派生值、审计和错误码。
6. TestPlan 覆盖台账。
7. 逐用例执行记录。
8. 失败分析、已知因果、未知边界和精确重测范围。
9. partial、blocked、skipped 与尚未完成的动作。
10. 清理证明。
11. 原始证据和重放入口。
12. 当前环境交接状态。

报告正文自包含地说明做了什么、怎么做、结果是什么。附件保存大体积原始证据，不替正文承担叙事。

## 十三、产物与提交边界

建议的本地结构：

```text
<task-workspace>/test/
├── test-plan.md
├── test-report.md
└── evidence/
    ├── requests/
    ├── responses/
    ├── logs/
    ├── sql/
    └── screenshots/
```

测试证据可以包含本地 harness、临时 fixture、快照和重放脚本；这些一次性资产默认不进入产品提交。用于永久保护产品行为的仓库级回归测试是实现资产，不因创建于本轮验证就自动清理；是否新增或提交服从当前任务授权和项目规则。提交前检查 staged files，区分两类资产；报告与原始证据是否进入版本库同样不自行扩大交付范围。
