# 研究包：当研究必须上手时

一份说"这个 SDK 能用"的报告，和你实际安装它、让它跑起来并把运行日志留在磁盘上的记录，是两种完全不同的证据。当决策依赖于"能不能跑 / 集成得好不好"的研究时，交付物不应该是报告——而应该是一个**研究包（research package）**：报告只是入口和索引，文件夹里存放的是可复现的实验、本地捕获的官方文档、以及建立在实证基础上的选型结论。下游架构师/工程师 agent 直接消费这个文件夹，无需再次搜索或踩同样的坑。

## 三种深度层级

按决策类型设定层级；不要每次都跑全套：

- `desk`：纯认知类问题（领域全景、概念澄清、市场扫描）。产出单个报告。
- `source-verified`：结论依赖于某个声明是否真实。在 `desk` 基础上增加来源级验证（参见 research-method.md）。
- `hands-on`：决策取决于"能不能跑 / 集成成本多大 / 选哪个 SDK"。产出一份完整研究包。

当指派单指定了层级时，跟随它；当没有指定时，自行使用上述标准设定层级，并在研究简报中记录层级和你的理由。不确定时问自己：发起者读完报告后的下一步动作会不会是"让人试试"？如果是，那么"试试"这一步正是你现在就应该做的。

## 目录骨架

```
research/<topic-slug>/
├── 00-report.md              # 决策报告（骨架见 research-method.md），整个包的单一入口点
├── experiments/
│   ├── README.md             # 实验索引：每个实验一行（问题 / ✅❌⏱ / 一行结论）
│   └── exp-01-<slug>/        # 序号 + 短名称；一个实验回答一个问题
│       ├── NOTES.md          # 参见"实验纪律"
│       ├── env/              # requirements.lock.txt + ENV.md（python/SDK 版本、复制粘贴安装命令）
│       ├── src/              # 实验代码，一条命令即可重跑
│       └── runs/run_01/      # output.log + meta.json + 本次运行的代码快照
└── docs-snapshot/
    ├── INDEX.md              # 参见"文档快照"
    └── <按主题分节的 .md 文件>
```

按需裁剪：如果没跑实验就没有 `experiments/`，如果没有捕获文档就没有 `docs-snapshot/`，但有的东西必须遵守本文的合同。

## 实验纪律

- **先问问题**：在触碰任何东西之前，先在 `NOTES.md` 中写出 `QUESTION`（本实验回答什么）、`SUCCESS CRITERIA`（什么算成功）和 `TIMEBOX`；运行后填写 `ANSWER` 和关键输出摘录。无法用一段话总结的实验就不算完成。
- **一条命令重跑**：每个实验必须通过一条命令重跑（例如 `python src/main.py`），没有隐藏步骤；将环境锁定到 `env/requirements.lock.txt` 并用 `pip freeze`，在 `env/ENV.md` 中记录 python 版本和安装命令。一个读了这份任务任何上下文的新 agent 也应该能仅从该目录中复现结果——这就是有效实验的门槛。
- **先跑基线**：先让官方的 quickstart/hello-world 按原样跑起来作为 run_01，然后再构建更贴近本任务场景的变体。如果基线就跑不起来，后续一切都是在沙子上建房子。
- **留下输出痕迹**：用 `tee` 将运行输出实时流式写入磁盘到 `runs/run_NN/output.log`，在报告中只引用摘录加指针——绝不要凭记忆转述；`meta.json` 记录 `{command, exit_code, duration_s, verdict: ok|failed, analysis}`；每次有意义的运行，将实验代码快照保存至运行目录（agent 会在运行之间修改代码，没有快照就無法追溯哪个代码版本产生了哪个结果）。
- **失败是一级交付物**：不要删除那些无法安装、无法运行或有版本冲突的运行——以原因命名目录（例如 `run_02-fail-version-conflict/`）。"我们试过 X，没成功，因为 Y"对下游决策的价值往往不亚于一个成功示例。
- **硬预算**：如果同一个错误在 3 次修复尝试后仍然存在，更换路线（不同的安装方法、不同版本、不同示例），而不是在一个死胡同中耗尽预算；TIMEBOX 到时必须把结论写到磁盘上，即使它是"没跑起来，卡在 X"。
- **实验代码用于验证**：在 `experiments/README.md` 中标注"仅用于验证，非生产质量"。它的任务是回答问题，不是交付产品。

## 执行边界

读取证据和运行实验是两个不同区域，有不同的红线：

- **证据读取区**（别人的克隆仓库）：保持只读——不执行它的任何脚本、setup.py 或 CI 配置。
- **实验区**（你自己写的代码 + 从官方注册表安装的依赖）：可以执行，但只能在隔离的 venv 或临时目录内执行，不污染系统环境；仅从官方注册表（PyPI/npm 等）安装依赖，安装前验证包名的拼写（typosquatting 是真实攻击面）；不使用生产凭证也不访问生产系统；收尾时清理实验派生的所有进程。

## 文档快照

目标是让下游 agent 能在离线状态下消费"这个决策需要的文档切片"——而不是镜像整个站点。分层进行：

1. 探测 `https://<docs-domain>/llms.txt`（注意它可能位于文档子域名上）。命中后将其保存到磁盘作为索引，并按需获取单个页面——许多站点支持在 URL 后追加 `.md` 直接获取干净的 markdown，优先尝试该渠道。
2. 文档源码存放在仓库中（mkdocs/docusaurus/sphinx 常见）：`git clone --filter=blob:none --sparse` 只拉取 `docs/`，并 checkout 匹配你正在研究的版本的 tag。自行判断：如果 `docs/` 中的 `.md` 文件打开可读，就用它们；如果里面全是模板指令和 autodoc 占位符，放弃并进入下一层。
3. 只存在渲染站点：对少量页面，用 `curl -s "https://r.jina.ai/<page-URL>"` 将每个页面转为 markdown；对整棵子树，使用本地工具如 crawl4ai。
4. 绝不要用 wget/httrack 做全站点 HTML 镜像——HTML 噪音会让 token 数膨胀，下游 agent 也无法使用。

`INDEX.md` 是快照对下游的合同，必须包含：来源 URL、仓库和版本/commit、捕获日期、捕获方法、文件清单（每文件一句话）、**明确的"未包含内容"**（防止下游误以为快照是完整集合）和已知缺口。

## 三种声明状态

在 `00-report.md` 的 Findings 中，将每条能力声明与其证据级别对应标记：

- **Verified（已验证）**：附上运行证据指针（`experiments/exp-01-*/runs/run_01/output.log`）。
- **Verification failed（验证失败）**：试过了，没跑起来——附上失败运行指针和原因。
- **Unverified（未验证）**：仅来自文档或源码阅读，从未实际运行。

在推荐理由中出现"这个 SDK 支持 X"却没有标记，是把伪装过的猜测当事实传递。

## Teamspace 晋升

研究包默认属于当前任务。收尾时做一次判断：文档快照或实验结论是否具有对**下一个不同任务**的复用价值（同一技术很可能再次出现）？如果是，将相关内容复制到 `teamspace/knowledge/<tech>/`，并在 INDEX.md 中注明来源任务和日期；遵循与 learnings 相同的去重规则——先检查 `teamspace/knowledge/` 是否已存在旧快照，高重叠时更新旧的而非另起新份。反之，**在每次研究的开头，先检查 `teamspace/knowledge/`**；命中则确认版本是否最新，直接复用，节省一轮获取。

## 交付前自检（研究包部分）

- `experiments/README.md` 索引覆盖每个实验目录；每个实验的 NOTES.md 都填好了 ANSWER。
- 每个实验都可通过一条命令重跑且环境完全锁定；每个失败运行都有存档。
- 报告中的每项能力声明都携带三态标记；每个"Verified"都附带可点击的日志指针。
- `docs-snapshot/INDEX.md` 的合同字段完整并声明了"未包含内容"。
- 已做出 teamspace 晋升判断（在报告末尾的 Follow-Up 附近一行：什么被晋升了，或为什么什么都没晋升）。
