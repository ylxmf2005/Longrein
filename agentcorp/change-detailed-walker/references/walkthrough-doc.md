# Walkthrough 文档骨架与代码阅读要求

动笔前重读本文；产出照这份骨架组装，交付前逐节核对没有丢信息。

## 读者假设：他没读过这片代码

这份文档的读者很可能从没读过被改动的代码——他掌握的是需求与设计层面的逻辑，现在要第一次真正看懂代码怎么改的。整份文档都按这个假设写：

- **先讲「原来是什么」，再讲「改成了什么」。** 每处改动动笔前问自己：读者知道这段代码本来是干什么的吗？不知道，就先用一两句话讲清它的职责和原行为，再展示改动。直接甩一段 before→after 而不交代背景，读者只能看到字符变了，看不出行为变了。
- **首次出现的模块、服务、关键标识符，给一句话介绍。** 「`foo_service` 负责文件的读写授权」这一句对读过代码的人是废话，对没读过的人是地图。宁可多写这一句。
- **不依赖读者脑中的调用图。** 说「调用点跟着变了」时，点名调用点是谁、在什么场景被触发，而不是假设读者知道谁调用谁。

## 阅读顺序：按理解的依赖序，不按文件序

raw diff 按文件序呈现，而那几乎从来不是建立理解的顺序——读者会先看到一个 helper，却还不知道它为什么存在。你的核心增值之一就是**重排**：

- **排序不变式：读者永远不该先遇到一个还没被介绍过的东西。** 任何段落引用某个函数、表或契约之前，它必须已在更早的位置被讲过，或当场得到一句话介绍。
- 默认依赖序链（可按任务调整）：**数据模型与 schema → 契约（API/类型/error/auth）→ 业务逻辑 → 调用点与入口接入 → 测试**。有自然入口（新的公开 API、配置项、UI 入口）时从入口开始讲也可以——关键是选定一条读者能跟随的线，而不是照抄 git 的文件序。
- 按 concern 组织叙述，不按文件。一个 concern 是一个内聚的设计意图（如「入参互斥校验」「私有副本派生」）；一个文件可以出现在多个 concern 下，一个 concern 也常横跨多个文件。

## 必备章节

下列章节是默认骨架。可以按任务调整标题，但不能丢失这些信息。

- **Reading Guide**：开头给一行改动面标尺——`N 个文件 · M 个模块 · ~L 行逻辑改动 · B 处跨模块/跨服务边界 · P 个新增公开接口`——让读者进正文前先有体量感。然后给**带编号的建议阅读路径**：按依赖序排成 stop 列表，每个 stop 是 `path:line` + 一句「这里是什么、为什么从这看」，读者拿着它就能在 IDE 里按序点过去。最后说明哪些章节对应主体能力、支撑改动、风险和维护入口。
- **Whole Change Story**：用连续叙述讲清本次实现整体改变了什么，为什么会牵动这些层。
- **Diff Surface Map**：按模块/文件组列真实改动面。每组至少写明职责、改动性质、行为影响、维护入口。**把纯机械改动单独标出、合并成行**——rename、格式化、import 重排、批量同质文件（如几十个本地化文件压成一行加一句白话描述）。一个 40 行的 rename diff 对读者来说行为上什么都没变，与语义改动同等展开会把真正重要的东西淹没。
- **Core Capability Walkthrough**：主体能力从入口到业务层、数据层、运行时/UI 接入的完整路径，按上面的依赖序组织。沿路径**点名每个被改的关键函数并展示它具体怎么改**（签名/分支/before→after），而不仅描述它做什么；同时写清主流程、状态变化、错误处理、权限/可见性、幂等或并发语义。
- **Supporting And Incidental Walkthrough**：主体之外被牵动的支撑服务、共享工具、上下文透传、后台任务、运行时包装、UI 接入、测试、配置、构建、文档或重构。每项写清「为什么被牵动」「如何支撑主体」「以后看哪里」，其中「如何支撑主体」要落到具体代码改动（改了哪个函数的什么、加了什么字段/分支），必要时贴 before→after。
- **Data And Contract Walkthrough**：数据表、数据模型、迁移、索引/约束、API/schema/error/auth、跨模块或跨服务契约、兼容性与迁移顺序。表/模型给出新增或变更的字段，API/契约给出签名或请求响应结构的前后差异。
- **Important Flows**：用流程叙述关键行为。每个流程写触发点、调用链、状态变化、失败/补偿、最终结果。流程涉及 3 个以上组件交互或异步协作时，配一张 Mermaid sequence/state 图；线性两三步的流程不要画图，文字已经够清楚时图只是复述。图与正文不一致时以正文为准，生成后核对一遍图没有漏分支、漏参与者。
- **Review And Verification Evidence**：列已有证据和缺口。没有执行证据就明确说没有，不声称跑过。
- **Risks And Maintainer Hotspots**：残余风险、review 必看点、维护者最容易误改的位置、线上观察点。**对 AI 实现的改动额外核对四类高发风险**，查到就写明，确认没有也说一句：引用了不存在的 API 或包（幻觉）；重复造了仓库里已有的轮子（新增 util 前先查有没有同职责工具）；改动面超出需求范围（对照 file list 与需求边界）；新增测试只覆盖 happy path。没有就如实说没有，不为凑数编造风险。
- **One-Page Mental Model**：最后用短段落或表格收束，让读者带走本次变更的核心模型。

## 代码阅读要求

从真实 diff 开始：

```bash
git diff --name-status <base>...HEAD     # 改动面地图
git diff --stat <base>...HEAD            # 规模分布
git diff <base>...HEAD -- <path>         # 读关键文件的真实 hunk（核心，不能只看 stat）
git log -p -L :<symbol>:<file>           # 需要时追某个函数的逐行改动
```

`name-status` / `stat` 只用来建立改动面地图；**写每一处改动之前，必须读它的真实 diff hunk**，看清新增、删除、改写的具体代码行。symbol-level 阅读也要落到 hunk，而不是停在「这个函数被改过」。对每个重要符号，从 hunk 里提取出文档要展示的「代码怎么改」：签名变化、增删改的分支与条件、返回结构变化、新增字段、调用点变化。读 hunk 时同步分辨它是语义改动还是机械 churn，归入 Diff Surface Map 的不同行。

再按 touched surface 横向阅读。对每个被触及的层次都至少读到足够判断其职责和行为影响：

- 入口层：handler/controller/route/command/UI entry。
- 业务层：service/use case/domain logic。
- 数据层：helper/DAO/repository/table/model/migration。
- 契约层：schema/types/API client/error/auth/permission。
- 运行时层：worker/interceptor/background job/runtime wrapper/scheduler。
- 界面层：frontend/mobile/user-facing wiring。
- 共享层：utility/common library/config/build。
- 测试层：unit/integration/E2E/regression fixtures。

对大型文件或大型 diff，允许用 symbol-level 阅读：先定位新增/修改的类、函数、路由、schema、表和测试，再围绕调用链读关键实现。不要因为文件大就只看 stat。

## 交付前核对

- 骨架逐节核对没有丢信息；`git diff --name-status` 里的每个文件都能在 Diff Surface Map 找到归属（语义改动逐组讲，机械 churn 合并行）。
- Reading Guide 的阅读路径满足排序不变式：顺着编号读，不会先遇到没被介绍过的东西。
- 抽查几段：假设自己没读过这片代码，每段能否先知道「这里原来是什么」再看懂「改成了什么」。
