# 规划

任务根目录的 `plan.md` 位于已经成立的需求、设计与下一段专业工作之间。它是 Shape 与 Dev 共享的当前路线：既让所有 Phase、Owner、依赖和完成边界保持可见，也保存执行过程中已经成立的状态、结果和证据。Plan 可以组织 Shape 调研、Dev 实现、Review 检查、Test 验证或其他真实需要的工作，不预设所有任务都以编码为中心。

## 只规划来源撑得住的内容

写计划前读取实际的 `shape/requirements.md`、`shape/design.md` 和被引用的契约、迁移、诊断或验证入口，双向检查需求、设计和真实对象是否一致。来源缺失、自相矛盾，或关键工作仍会迫使执行者自造范围、契约、数据所有权、失败语义或判断标准时，继续 Shape；不能用更详细的任务列表掩盖模型尚未成立。

计划通过路径和稳定 ID 引用来源，只带入当前阶段立即需要的少量事实。复制需求与设计会制造第二份权威来源。

## 按真实依赖形成 Phase

Phase 是能够独立取得结果并接受检查的工作单元，不是按目录、Skill 名称或仪式阶段切出的批次。数量由任务决定；没有依赖的 Phase 可以并行，因此可以同时存在多个 `in_progress`；需要前置证据、决定、契约、迁移或环境的 Phase 明确等待条件。

每个 Phase 都应说明：

- 完成后什么事实、决定、能力或验证结论真实成立；
- 它追溯到哪些 `REQ-*`、`AD-*`、Completion Evidence 或根因；
- 由哪个 Owner 或 Skill 承担，依赖什么输入和前置结果；
- 必须保持的行为、禁止区域和传播成本；
- 本阶段的完成检查、下游证明入口，以及什么反馈会修订后续 Phase；
- 包含实质实现时，阶段结束的轻量 Review 审查什么，以及结果从哪里继续；
- 当前状态、已经成立的结果、证据或阻塞。

计划可以指明已确认的对象、模块、接口和数据边界，但不替专业执行者预定尚需真实现场决定的局部方法。阶段之间应尽早接触高风险未知，并让中间状态保持可理解、可恢复和可继续判断。

Plan 只组织跨阶段关系和当前 Phase 事实，不接管专业 Skill 的详细计划或结论。Test 需要完整场景、环境和用例时仍由 `test/test-plan.md` 拥有；Dev 的局部施工步骤留在当前执行计划，最终实现结论进入 `dev/report.md`；Review 的覆盖模型、finding 资格和裁决由 Review 产物拥有。Plan 只保存 Owner、依赖、目标结果、当前状态，以及足以判断阶段是否成立的简短结果和证据入口；详细叙述留在所属专业产物。

## 当前模型随实现继续修订

Shape 创建初始 Plan，并在塑形调查、用户决定或方向变化改变需求、设计、公共或跨模块契约、任务承诺、Phase 目标、Owner、关键依赖或风险接受时，修订真正拥有这些结论的产物并同步更新受影响的 Plan。

Dev 直接维护自己执行所触及的 Plan：开始或结束 Phase 时更新状态；用实现事实补充当前结果、证据和阻塞；在既有用户承诺内拆分、合并或重排局部工作；把实际发现的调用者、文档、schema、Swagger、生成物和验证传播补进相关 Phase。实现反馈增加或推翻需求细节、`AD-*`、设计、契约或迁移判断时，Dev 同时修订真正拥有结论的 Shape 产物，不需要为了切换阶段而留下过期模型；改变用户承诺或授权边界时先取得用户决定。

Plan 保存当前有效状态，不保留已经失效的路线来充当历史。结构性变化和有交接意义的状态变化用 `longrein_checkpoint` 重新登记 artifact，Timeline 保存变化原因；普通工具调用和细碎施工步骤不进入 Plan 或 Timeline。

## 在 Phase 结束时接受轻量 Review

包含实质代码或系统模型变化的 Dev Phase，在标记 `completed` 前由 `review` 以 phase 模式重新读取本阶段实际变化、相关用户决定、受影响的 Shape/Dev 产物和 Runtime 状态。审查对象按 Phase 收窄，不默认生成逐 hunk 台账，也不为了显示完整展开所有不相关类型；决定授权、代码与产物一致性、产物与 Runtime 同步、Phase 完成资格不能省略。

Review 默认不直接改 Plan 或被审产物。Finding 回到 Dev、Shape、Owner 或产生状态的 Skill 修复，阻断项和未决 Owner decision 清除后再完成 Phase。极小机械变化可以把阶段 Review 与最终交付 Review 合并；Review、Test 或纯调查 Phase 不递归要求再审自己。

## 显式暴露传播成本

新增依赖、数据迁移、认证或权限变化、公共或跨模块接口变化、用户界面行为变化，以及同等传播面的工作，不论落在哪个 Skill 或 diff 大小，都要在所属 Phase 和跨阶段风险中显式指出。它们需要消费者、兼容、退出和验证路径，不能作为普通步骤夹带。

已经决定的契约需要同步调用者、文档、schema、Swagger、生成物或测试时，这是 Dev 应直接补齐并更新 Plan 的传播工作。若实现现场表明必须选择另一套公共契约、统一机制、数据所有权或任务范围，停止依赖旧前提的部分并回到 Shape。

## Plan 不替代 Current Work 或报告

`plan.md` 显示全部 Phase 及其当前状态；Runtime 生成的 `task.md` 只显示此刻的 Now、Next、Waiting on 和 Active work unit。Dev Report 说明最终交付的行为、变化面、验证和剩余风险。三者分别回答完整路线、当前工作和专业结果，不能互相复制。
