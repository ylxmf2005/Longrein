# 文档

## 描述

文档质量对可维护性至关重要。注释捕获代码本身无法表达的设计信息。没有适当文档，开发者会浪费大量时间重建原始设计意图，导致修改变慢且 bug 风险增加。设计时，使用此维度评判你要求的注释是否在适当细节层级提供了必要信息、有效描述了非显然方面，并支持而非污染抽象。

## 核心原则

### 注释描述代码无法表达的内容

代码只能表达形式语法与机制。它无法表达：
- 设计决策背后的理由
- 高层目的与意图
- 不变量与前置条件
- 边界条件语义（包含 vs 不包含范围）
- 何时以及为何使用特定接口
- 跨模块依赖与协调需求

这些元素存在于设计者的头脑中，但无文档则会消失。注释为未来的维护者保存这些知识。

### 在不同细节层级上撰写文档

有效注释在与代码本身不同的层级提供信息：

**更低层级的精确**：变量声明、参数与返回值需要精确澄清。说明单位、边界行为、空语义、所有权职责与不变量。像 "current offset" 这样的注释太模糊；"尚未返回给客户端的第一个对象的位置" 增加了必要的精确度。

**更高层级的抽象**：方法与循环的注释应描述意图，而非机制。不要逐行复述，而是解释代码整体完成什么以及为什么需要它。"尝试将当前 key hash 附加到已有 RPC" 比描述循环中每个条件测试更有价值。

### 接口 vs 实现文档

将用户需要知道的与内部运作方式分开：

**接口注释**定义抽象。它们描述类的能力、方法从调用者视角的行为、参数要求、返回值、副作用、异常与前置条件。用户应能在不阅读实现代码的情况下理解抽象。

**实现注释**帮助维护者理解内部运作。它们解释主要代码块完成什么以及为什么采取特定方法。然而，如果接口文档与代码足够清晰，大多数简单方法不需要实现注释。

接口文档泄露实现细节是浅抽象的危险信号。

### 避免与代码冗余

仅复述代码显然内容的注释浪费阅读者时间与维护精力。如果有人可以仅通过看相邻代码（而不理解更广泛的上下文）就写出这个注释，那这个注释没有价值。

在注释中使用方法或变量名的相同词汇是常见的冗余模式。不要重复名称，而是提供关于含义、约束或上下文的额外信息。

### 在设计时撰写注释

在实现代码之前先写接口注释。这样做：
- 在思路新鲜时捕获设计推理
- 在编码前强迫清晰思考抽象
- 作为评估接口质量的设计工具
- 防止注释债务积累

如果一个方法或变量需要冗长复杂的注释才能完整描述，这是危险信号，表明抽象存在问题，应重新考虑。

### 记录跨模块决策

跨越多个模块的设计决策特别容易出 bug，且难以有效文档化。当依赖跨越边界时，要么：
- 将文档放在开发者会遇到的明显中心位置（例如所有值必须一起考虑的枚举）
- 为复杂横切关注点维护中央设计笔记文件，并从受影响代码位置放置简短指针

## 危险信号

### 缺失接口文档
- 没有接口注释的公共类、方法或重要变量
- 方法文档遗漏行为描述、参数含义、返回值语义、副作用或异常条件
- 假设代码自解释

### 重复代码
- 在同一细节层级上逐行解释代码的注释
- 只是把实体名称重新排列成句子的注释
- 注释中没有代码声明或语句本身无法立即看出的信息

### 模糊或不精确的文档
- 变量注释缺失单位、边界行为、空语义或所有权
- 像 "current offset" 这样的通用注释，没有说明什么让它 current 或它是哪个东西的 offset
- 抽象描述没有提供足够细节来正确使用接口

### 接口注释中的实现细节
- 接口文档描述内部数据结构、算法或实现方法
- 在同一注释中混用"内部如何工作"与"如何使用"
- 把用户不应依赖的细节暴露给用户

### 过时或误导的注释
- 与实际代码行为矛盾的注释
- 代码变更后描述旧设计的文档
- 远离所描述代码的注释，导致同步困难

### 缺失"为什么"的解释
- 棘手代码没有解释其必要性
- workaround 与特殊 case 没有 rationale
- 非显然的设计决策没有记录推理

### 不充分的抽象文档
- 用户被迫阅读实现代码才能理解如何使用接口
- 没有记录整体抽象代表什么
- 缺失理解代码目的所需的高层概念框架

## Issue Tags

- **[Missing Documentation]**：未记录的公共接口、关键变量或复杂逻辑
- **[Redundant Comment]**：仅仅重复代码已显内容的注释
- **[Vague Documentation]**：缺乏正确使用所需细节的模糊注释
- **[Implementation Leakage]**：接口注释泄露内部细节
- **[Outdated Comment]**：与当前代码行为矛盾的文档
- **[Missing Rationale]**：未解释的非显然设计决策或棘手代码
- **[Poor Abstraction Documentation]**：用户不阅读实现就无法理解接口
