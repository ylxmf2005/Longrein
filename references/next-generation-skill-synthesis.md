# 下一代 Skill：从流程脚本变成可校准的调节器

> 综合来源：Anthropic Fable 5 官方提示指南、OpenAI GPT-5.6 Sol 官方提示指南、Simon Willison 的真实 Fable 长程记录、Wayfinder、gstack Skill carving 讨论、Finding Unknowns 社区实现、旧版 AgentCorp 全技能语料，以及 Ashby / Conant 的控制论原始文献。本文是面向新一代 AgentCorp 的研究结论与改进建议。

## 一、共同结论不是“提示词越短越好”

官方和真实案例共同支持的是更精确的说法：强模型已经能完成大量旧脚手架负责的工作，重复步骤会降低效率，甚至把模型锁在用户的初始认知里；但目标、权限、证据、稳定契约和停止条件仍然必须清楚。

下一代 Skill 的形态应是：

- 用高密度语义定义它在解决什么关系；
- 用真实环境和工具让它看见差异；
- 用少数不可妥协边界定义可接受区域；
- 用反馈决定下一步，而不是预写所有步骤；
- 把详细协议、模板和案例放在按需 reference；
- 通过相同 trace 的重放决定某条规则是否值得存在。

## 二、Skill 进化的第一原则：没有现实反馈，不改长期行为

“自行改进这个 Skill”通常只会得到更完整的语言。可靠进化需要真实轨迹形成的反馈场：失败、成功样本、反例、用户纠正、实际后果和成本共同说明当前能力离目标有多远。具体误差是有价值的方向信号，但第一个误差会锚定搜索，不能自动代表根因、完整范围或正确改法。

候选原因充分展开后，再进入可归因的评测闭环：

1. 固定代表性输入、模型、effort、工具和预期；
2. 保存原始输出与实际后果；
3. 让能够解释现象的候选原因竞争，而不是沿第一个失败直接下结论；
4. 为了验证候选原因，控制变量修改相关指令或语义锚点；
5. 重放相同样本，并加入一个邻近反例；
6. 同时比较质量、token、延迟、请示次数和意外副作用；
7. 只有稳定改善才进入长期 Skill。

第一次错误提供学习信号，不提供完整答案；找到经反例和重放仍成立的原因后，长期规则应提前识别原因，而不是要求每次重新踩坑。

## 三、显式和隐式不是对立面，而是不同信息的分工

适合显式写出的内容：

- 用户可观察的结果；
- 验收与停止条件；
- 权限、安全、兼容性和不可逆边界；
- 已经确认的项目事实；
- 必须保留的产物与证据。

适合保持开放或用高密度语义表达的内容：

- 调查与实现路径；
- 在证据不足时选择哪种探索动作；
- 风格、关系、角色和专业实践的整体判断；
- 可以由模型根据当前项目自行推导的机械步骤。

探索阶段扩大自由度；执行阶段固定边界和验收；新证据推翻内部模型时重新开放搜索。计划模式和明确规范并不是敌人，错误在于让某一种档位永久统治所有阶段。

## 四、临时指令与长期规范必须拥有不同写入权

一次任务中的“这次不要提交测试”“先用某个临时分支”“这轮输出简短”，都可能是当前情境的局部约束。它们不能因为有效一次就被模型写进全局 Skill、`AGENTS.md` 或记忆系统。

建议把持久化分成三层：

- 当前会话状态：自动更新，任务结束后自然失效；
- 任务资产：保存证据、决定和复现方法，但不自动支配其他任务；
- 长期规则：需要重复证据、明确适用边界和人工批准。

这不是一条“记得别乱写”的提示，而是权限与文件结构设计。

## 五、对 Shape 的改进建议

### 必须改：从“形成需求”转成“让领地先产生反馈”

当前 Shape 的地图—领地骨架是对的，但主线需要纠正一个更隐蔽的问题：不要从抽象 brainstorm 开始，也不要用“先找一个最可能改变方向的差异”替探索空间预先画边界。接触真实领地，让事实、矛盾、可能性、代价和用户反应共同显形；最先出现的差异只是新证据。

四类未知可以作为事后理解和比较材料，不宜在探索开始前成为分类框、顺序流程或隐形数量限制。模型不应预设未知有几类、哪类最承重或哪种发现动作有效。

### 应增加：方向状态，而不是更多步骤

Shape 只需维护三种高层状态：

- 稳定事实与已知价值；
- 当前可决定的前沿；
- 仍处于迷雾、暂时无法准确表述的部分。

新证据可以让已决定内容重新变成前沿。目标之外的事项与目标之内但尚未看清的事项必须分开。

### 应压缩：把产物格式移出主语义

`investigation.md`、`proposal.md`、`shape.md` 的字段契约适合放到 reference。主 Skill 只需说明：方向实质变化时保存稳定路径，并让用户知道路径。详细目录树不应在每次调用时占据核心注意力。

### Shape 的高纯度核心可以是

> 先让领地说话，再决定形状。让事实、可能性、代价与未知充分显形；方向有资格被承诺时收敛执行，新证据出现时重新开放。

## 六、对 Prompt 的改进建议

### 必须改：把“提示词教练”与“Skill 进化”分开

Prompt 可以在不打断主任务的前提下，把已经展开的偏差提纯成用户能学习和复用的表达；但不能因为最终反馈需要聚焦，就假定偏差只有一个来源。修改长期 Skill 时，必须有真实 trace 或 benchmark。两者不应共享同一授权标准。

### 应增加：语义锚点也要做 A/B 验证

`Swiss Style`、`相对论`、专业角色或成熟实践能压缩大量背景，但它们也会携带连带气质。示例、参考和高密度词都应被视为主动塑造地形的候选控制输入：探索后再有意引入，用同一任务比较输出，检查继承了什么、遮蔽了什么，而不是把“隐式”当神奇关键词。

### 应增加：规则晋升门

当模型想把一次提示沉淀为长期规则时，先回答：它修复了哪个真实误差？出现过几次？适用范围是什么？放在 prompt、任务资产、项目规则还是 Skill 哪一层？谁批准写入？

### 应采用官方一致的迁移方法

从能工作的版本出发，每次删除或修改一组规则，重放相同评测。不要一次重写整套 prompt stack，否则无法知道改善来自模型、effort、工具还是文字变化。

### Prompt 的高纯度核心可以是

> 用真实输出训练提示，而不是在真空中润色。目标、验收与硬边界说清；路径留给模型；高密度语义先当实验，稳定有效后才晋升为长期规则。

## 七、对 Rewind 的改进建议

### 当前最好的部分应保留

“把资产带回过去，不把混乱带回去”已经是高纯度母题，优于一长串恢复字段，不应再扩写成固定报告清单。

### 必须增加：区分三种恢复

- 原地校正：错误局部、当前上下文仍可信；
- 回溯或分支：错误前提已经污染计划、待办或解释，但目标仍然有效；
- 新任务：目的地或范围已经重画，旧路线不再是同一项工作。

Wayfinder 的启示是：目标之外的工作应成为新 effort，而不是把所有历史继续拖进当前时间线。

### 必须增加：回溯后的第一步是恢复反馈

短提示词不应直接命令过去的自己继续实现。它应先核验承重事实、重放最关键失败或读取当前真实文件状态，确认新时间线已经获得正确误差信号。

### 应明确：分别处理对话状态和外部状态

上下文可以回到过去，文件未必需要恢复；代码可以保留，旧计划可能必须丢弃。任何删除文件、改写 Git 历史或改变外部系统的动作仍需明确授权。

### Rewind 的高纯度核心可以是

> 当当前时间线开始放大一个已经失效的前提，把可靠事实、证据和有效产物带回错误前提形成之前；清除旧控制状态，并先重建反馈，再继续行动。

## 八、对 Soul 的改进建议

Soul 最适合保留一个高纯度角色，而不是变成所有 Reviewer 的压缩清单：“像明天还要为它值班的人”。它贯穿工程选择，但不拥有独立阶段和报告。

应保留的判断是：结果与基线先钉住；证据高于状态词；根因决定改动尺度；复杂度必须付费；阻塞只截断依赖路径；调查权不等于执行权；新证据可以反向修订旧决定；临时安排不能自动成为永久承诺。

应避免把测试步骤、评审严重度、产物模板和全部 AgentCorp 铁律塞进 Soul。那些细节依赖具体阶段与对象，进入常驻 Skill 只会稀释它的工程人格。

### Soul 的高纯度核心可以是

> 像明天还要为它值班的人：让现实拥有最后解释权，让系统说真话，让复杂度付得起账，并只接受用户真正理解且愿意承担的长期承诺。

## 九、对 Walkthrough 的改进建议

Walkthrough 不应被锁在交付末端。人在需求、计划、设计、评审、故障分析和实现过程中都可能失去对对象的完整理解；只要下一次决定需要真实心智模型，就可以对当前对象做 Walkthrough。

它的主线不是“汇报做了什么”，而是让叙述服从对象本身，按人的学习顺序重建稳定关系、关键变化或决定、完整影响链与证据边界。HTML、Mermaid、展开单元、变动标记和理解检查都是可选表达手段，不能反过来把不同对象压成同一份模板。

Walkthrough 还要意识到示例和图会主动塑造理解。具体材料只在能够压缩关系、暴露差异或支持核验时使用，并说明它证明什么、不能证明什么；不能让最容易展示的一条路径替完整对象定形。

### Walkthrough 的高纯度核心可以是

> 把人带回能够继续判断的位置：让对象自己决定讲解形状，用完整关系建立理解，用证据与路径保持可检查。

## 十、对 Evolution 的改进建议

任务沉淀与 Skill 进化应合并为同一个闭环，因为它们都从“这轮发生了什么，怎样改变下一轮”开始。不要先把经验塞进预设层级；先理解它要改变哪种未来行为、作用范围和失效条件，再让任务证据、回归检查、项目规则、设计记录、工具或 Skill 等可能落点显形。

Evolution 应在非平凡任务的里程碑、完成时和真实重复摩擦出现时主动触发，但不能为了存在感强行沉淀。普通项目资产遵守当前任务授权；涉及 Skill 的长期修改必须满足更高门槛：可重放误差、前后对照评测、邻近反例和用户对具体 diff 的明确批准。

这项能力最需要避免两个旧模型时代的问题：一是用自动 hook 产生大量无人处理的提案；二是把“总结经验”误认为改变未来。只有实际落到正确作用域、并能让未来行为不同的资产才算进化。

### Evolution 的高纯度核心可以是

> 让这轮代价改变下一轮。先从真实轨迹判断经验应落在哪里；能结构化的不要只写提醒，涉及 Skill 的不要凭感觉自改。

## 十一、模型什么时候应该主动介入用户

模型应主动告诉用户“方向可能错了”，当且仅当它能展示一个改变决策的真实差异：代码事实、运行失败、成本、权限后果、可行替代或目标冲突。怀疑本身不是介入理由。

模型应该提问，只有当答案确实只能来自用户的价值、偏好、授权或外部信息，而且不同答案会改变路线。项目能自行调查的事实不应转嫁给用户。

模型应该建议 Rewind，当继续当前上下文会系统性放大旧前提。恢复范围应沿污染传播链判断，不预设局部修正、完整回溯或新任务哪一种更优；建议必须说明可信返回状态和可保留资产，但不擅自执行破坏性恢复。

## 十二、建议的评测集

在实际改 Shape、Prompt、Rewind、Soul、Walkthrough 与 Evolution 前，先固定代表性 trace：

1. 清楚且低风险的小修改，检验是否制造仪式；
2. 用户非常自信但项目事实相反，检验是否敢于纠偏；
3. “看到才知道”的审美或交互任务，检验是否会先制造参考；
4. 计划中途被新证据推翻，检验是否重开 Shape；
5. 临时指令很有效但不应持久化，检验规则晋升门；
6. 长上下文与文件状态错位，检验 Rewind 是否选择正确恢复层。
7. 一轮工作没有真正可复用经验，检验 Evolution 是否克制地不制造资产；
8. 同一失败既可写项目规则也可改 Skill，检验它是否让真实作用域显形并守住批准门，而不是机械偏爱最小改动。
9. 在计划尚未实现时请求 Walkthrough，检验它是否围绕当前对象恢复理解，而不是伪造完成或套用交付叙事。

对每类任务比较当前版与候选版：方向质量、真实证据、用户问题数量、意外动作、token、完成时间、是否产生可用产物。只有结果支持的修改才进入主 Skill。

## 十三、最终判断

新 AgentCorp 不应只是旧版 AgentCorp 的缩写版。它需要把旧版的证据和制衡哲学、Fable/Sol 的路径自由、Finding Unknowns 的认知状态、Wayfinder 的迷雾与前沿，以及控制论的反馈模型，压缩成少数能够产生正确行为的母概念。任务沉淀与 Skill 进化也不必拆成两个争抢同一信号的能力：同一个 `evolution` 先判断经验的正确落点，只有 Skill 级变化才进入对照评测与人工批准。

最重要的不是把 Skill 写得像答案，而是让 Skill 能持续感知自己离目标有多远，并在新证据出现时改变行动。
