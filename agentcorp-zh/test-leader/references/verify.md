# 本地验证编排参考

在编写非平凡变更的测试员指派之前加载本文件：它定义了每个验证级别需要什么，以及如何处理环境。

## 验证级别

验证是分层的，层是有序的：在较低级别所需的检查通过之前，较高级别的证据尚不成立。

- **能力** — 每个 Must Have 都需要直接证据；如果存在自动化测试，运行它们；仅在测试计划指明手动确认的地方才手动进行。
- **集成/API** — 跨模块的数据流已被实际演练；边界的错误传播已被检查；公共契约通过真实请求、响应、schema 或 CLI 输出来验证，而不是凭信仰断言。
- **E2E** — 每个面向用户的能力至少有一个完整的用户目标贯穿其中；快乐路径和应该被演练的错误路径都已覆盖；每一步记录动作、预期结果、实际结果和证据。

## 处理环境

当测试计划有环境规格时，遵循它。当环境不可用时，诚实说明哪些检查被阻塞或降级，而不是通过阅读源码来编造证据。
