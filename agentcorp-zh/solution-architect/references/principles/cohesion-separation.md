# 内聚与分离

## 描述

这个维度处理最基本的设计问题之一：两段功能应该一起实现还是分开？这个决策适用于所有层级——函数、方法、类、模块和服务。目标是通过做出知情的"合或分"选择来最小化整体系统复杂度，从而改善信息隐藏、减少依赖并创造更深的接口。

关键关注点包括：
- **合或分的决策**：相关功能应该合并还是分开
- **代码重复**：暗示缺失抽象的重复代码模式
- **通用与特殊混合**：通用机制被特殊用途逻辑污染
- **连体方法**：无法独立理解的代码片段

## 核心原则

### 拆分自带复杂度

把系统拆成更小的片段并不自动更好。拆分带来成本：
- 更多组件需要跟踪与管理
- 组件之间的额外接口
- 跨组件的潜在代码重复
- 物理分离让依赖关系更难看清
- 协调多个片段的管理开销

这些成本只有在拆分真正通过更好的信息隐藏和更清晰的抽象降低复杂度时才有回报。

### 相关则合并

代码片段如果通过以下方式真正相关，则属于一起：
- **共享信息**：两者依赖相同知识（例如，文档格式、协议结构）
- **双向使用**：使用一个的人几乎总是也使用另一个
- **概念重叠**：它们属于一个自然的高层类别
- **理解依赖**：不理解另一个就无法理解这一个

### 为简化接口而合并

合并模块可以消除中间接口，并自动处理横切关注点。当部分解决方案被合并时，组合后的接口通常比原始接口之和更简单，某些功能可以被透明处理。

**示例**：合并文件 I/O 与缓冲消除了用户理解或配置缓冲的需要——它默认就工作。

### 为消除重复而合并

重复代码模式暗示缺失的抽象。当相同逻辑出现在多处时，考虑：
- 将通用代码提取为共享方法（如果模式足够大且有简单接口）
- 重构为只在一个地方执行代码
- 浅层提取（一行方法配复杂签名）是否增加比消除更多的复杂度

### 将通用代码与特殊代码分离

通用机制不应包含针对特定用例的特殊逻辑。混合它们会导致：
- 更难理解的更复杂机制
- 机制与特定用例之间的信息泄漏
- 修改一个用例需要改动通用机制的脆弱性

**解决方向**：把特殊代码向上拉到更高层，保持低层通用。上层通常处理应用特定逻辑，低层提供可复用的基础设施。

### 为更干净的抽象而拆分方法，而非长度

方法长度本身不是拆分的理由。接口简单的长方法可以是深且好的。只在拆分能产生更干净的抽象时才拆分方法：

1. **提取子任务**（父子拆分）：子方法实现一个相对通用的、可清晰分离的子任务。阅读者应能在不了解父方法的情况下理解子方法，反之亦然。

2. **划分复杂功能**（兄弟拆分）：原方法试图用过于复杂的接口做多个不相关的事情。每个结果方法应有更简单的接口，且大多数调用者只需要其中一个。

避免制造浅方法或迫使调用者协调多个方法调用并传递中间状态的拆分。

### 合并方法以降低复杂度

在以下情况下考虑合并方法：
- 用更深的方法替换浅方法
- 消除代码重复
- 移除依赖或中间数据结构
- 通过整合知识改善封装
- 简化调用者接口

## 危险信号

在你提出的设计中以及你阅读代码时注意以下模式：

### 重复
- 相同（或几乎相同）的代码出现在多处
- 复制粘贴后带有微小变体的逻辑
- 暗示缺失抽象的重复模式
- 例外：不要提取带有复杂上下文依赖的一行模式

### 特殊-通用混合
- 通用类或方法包含针对特定用例的逻辑
- "知道"其用户的机制
- 信息泄漏：修改用例需要改动通用基础设施
- 库级别模块中的应用特定代码

### 人为分离
- 只在一个地方被调用的浅方法
- 功能被拆到必须一起理解的类中
- 只是传递数据而不做转换的中间接口
- 迫使阅读者在多个位置之间翻转的代码分离

### 连体方法
- 无法独立理解的方法
- 物理分离但智力不可分割的代码
- 阅读者必须同时看两段才能理解任一
- 子方法需要了解父上下文

### 不恰当的分组
- 不相关功能被硬塞进一个类
- 缺乏连贯主题或职责的模块
- 无理由混合多个概念类别的类

### 过度细分
- 许多小类，每个只做很少的事
- 浅组件的爆炸
- 管理众多接口带来的复杂度，而非实际领域复杂度
- 基于僵化行数规则而非抽象质量拆分的代码

### 缺失合并机会
- 多个职责重叠的类
- 可被统一但重复的接口
- 散落在共享相同知识的模块中的功能

## Issue Tags

在设计产出中标记合或分问题时使用以下标签：

- **[Repetition]**：相同或相似代码未经抽象就被重复
- **[Special-General Mixture]**：通用机制包含特殊代码
- **[Conjoined Methods]**：无法独立理解的方法
- **[Artificial Separation]**：不必要的拆分制造了浅组件
- **[Over-Subdivision]**：过度拆分导致接口泛滥
- **[Missing Consolidation]**：相关功能散落在多个模块中
- **[Poor Method Split]**：未能改善抽象质量的拆分
