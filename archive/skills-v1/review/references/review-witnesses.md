# Review 证据构造法

评审时先从 Task Context 的 Task Operating Envelope 和实际改动表面选择相关检查角度。它们是构造证据的方法，不是固定审查角色清单；每个类型仍在 coverage ledger 中标明 `reviewed | not-applicable | evidence-gap`。

## Behavior & Boundaries

### 正确性

```text
真实输入或状态 → 实际分支/状态转换 → 错误可观察结果
```

检查边界值、空值、顺序、非法状态、错误传播和需求不匹配；只有 Task Operating Envelope 或实际调用者允许交错时才展开并发。测试通过不能替代自己走路径；测试也可能共享作者的盲区。

### 公共契约

```text
现有调用者依赖的承诺 → 本次形态或语义变化 → 调用者怎样失败
```

覆盖 route、请求/响应、必填/可选字段、错误码、CLI、schema、导出类型、认证方式和迁移窗口。版本号或 changelog 不是迁移路径；纯新增可选字段和兼容默认值通常不是问题项。

### 安全

```text
不可信入口 → 缺少或可绕过的控制 → 敏感操作/数据出口 → 可利用后果
```

没有可达攻击路径就不报漏洞。区分理论加固建议与当前可利用问题；检查身份、授权、注入、敏感信息、令牌/密钥和滥用规模。

### 可靠性

```text
依赖慢、失败或半成功 → 错误传播/状态残留 → 重试、泄漏、重复或级联损害
```

检查 timeout、retry、幂等、补偿、清理、资源释放、部分持久化和 shutdown/recovery。正常路径不能证明失败路径。

### 性能

```text
会增长的维度 × 有来源的真实规模 → 查询、内存、网络或延迟如何越界
```

没有真实规模、热路径和成本计算时，不把“可能慢”写成问题项。用查询计划、复杂度、基准或生产数据支持。

### 对抗与组合

```text
多个单独合理的行为/前提 → 特定顺序、组合或滥用 → 级联失败
```

用于高风险、跨组件、时序和可滥用变化。必须给具体场景，不使用“再想几个极端情况”制造噪声。

## Change Integrity

### 范围与变更卫生

```text
一个 hunk/文件/依赖 → 当前授权源 → 找不到归属或改变了未批准语义
```

区分本轮必要变化、分支历史残留、格式噪声和顺手重构。无授权内容通常应撤回或拆分，不因“顺便更好”留下。

### 明确标准

```text
项目逐字规则 → 当前代码或产物 → 可指认的违反
```

没有原文规则就不能把个人偏好冒充标准。规则与项目现实冲突时说明冲突和改变代价，不静默违反，也不机械服从。影响较小不代表可以忽略：优先级描述后果，是否在当前对象中修复由规则归属、修改成本和处理方式决定。

### 简洁性

```text
新增抽象、选项或间接层 → 它实际隐藏/解决什么 → 删除后承诺是否仍成立
```

通过删除检验才能报告多余复杂度：指出更简单的结构，并证明删掉以后承诺仍成立。正确性、安全、可观察性和显式失败真正购买的复杂度不是问题。

### Cleanliness 与 dead code

```text
未使用、不可达、重复或漂移的代码/注释 → 实际调用与任务承诺 → 删除后是否影响当前行为
```

搜索真实引用和入口后再下结论。新引入或被本次变化遗留的 dead code 属于当前 review；纯既有垃圾只有在本次变化扩散、依赖或掩盖它时才展开。注释只在错误、漂移或缺少代码无法表达的边界时形成问题，不把文案偏好当缺陷。

### 测试与文档资产

```text
本次行为或长期边界 → 现有测试/文档能否让未来修改者看见它 → 缺失怎样让错误安全通过
```

只报告会让真实缺陷逃逸或让维护者误解边界的缺口。完整 TestPlan、执行与覆盖结论属于 `test`；review 只判断本次变化留下的证据和维护资产是否足以支撑前进。

## Architecture & Stewardship

### 架构、形态与品味

```text
当前 hack 或错误概念 → 更诚实的结构 → 已检查的迁移与实现价格
```

最小 diff 可能在错误 schema、模块边界、依赖方向或命名上续命；更大改动也可能只是过度设计。先找到真实模型，再在诚实方案中选择最小者；给出更好形态以前检查它会触及的调用者、数据、迁移和实现价格。

### 项目长期承担

```text
新增公共面、依赖、债务或运维承诺 → 谁长期承担什么成本 → 谁有权接受
```

说不清成本承担者和授权者的担忧只是个人偏好。临时方案需要退出条件、触发器和负责人。

## Task Coherence

Task Coherence 检查同一项变化在用户决定、代码与运行结果、当前专业产物、Task Runtime 四个表面上是否同源。它不是文案校对；问题必须说明哪一项现实、授权、模型或状态因此失真，以及正确修复应落在哪里。

### 决定来源与授权

```text
用户原话或可追溯决定 → 是否明确选择了价值、范围、风险或取舍 → Task Context / UD → 下游专业决定与实现
```

Agent 报告过、用户看过或讨论过，不自动等于用户作出决定；产物里写了 `UD-*` 也不能创造授权。反过来，用户已经明确选择但 Task Context 和源产物没有同步，是模型遗漏，不需要为了补文档伪装成一次新决定。检查 `AD-*` 是否在现有用户承诺内，还是把范围、兼容、风险接受或长期责任偷渡成了专业决定。

同时核对 Runtime 的 Decision Sources：新 UD 是否有原始消息位置和逐字原话，原话是否真的支持当前承诺。特别检查调查授权被扩成实现授权、局部最小化要求被扩成全局拒绝行为，以及 Agent 建议经压缩摘要、Requirements 或 Design 被改写成“用户明确要求”的来源漂移。没有来源或来源只支持更窄解释时，使用 `owner-decision`，不能靠文档间互相引用补齐授权。

既有决定是当前变化的输入，不是默认审查对象。只有本次 diff 依赖、修改、违背它，或它的来源会改变当前授权判断时才沿引用回看；没有这些关系时不复述旧 `UD-*`、`AD-*` 或历史设计。若旧选择只隐含在代码中，先把它作为当前事实或契约调查，只有本任务需要明确保护、替换或继续依赖该专业取舍时才形成当前可审查的决定。

### 代码与专业产物一致性

```text
实际 diff / 运行行为 → 新增或改变的语义与结构 → 当前 REQ / AD / 契约 / 根因 → 受影响消费者
```

完整阅读审查对象的变化面，但不默认要求逐 hunk 台账。局部机械实现可以直接由既有决定解释；能够独立变化、影响消费者、改变契约或需要后续审查的选择若只藏在代码或 Dev Report 中，应形成或拆分所属 Shape 产物中的 `AD-*`。文档更新了但代码没有实现、代码实现了但文档仍描述旧模型、不同产物各写一套当前结论，都属于 coherence finding。

### 产物与 Runtime 同步

```text
当前专业产物变化 → Artifact 登记与 hash → Task / Timeline 投影 → 下一位实际读取的状态
```

`longrein_task_read doctor` 能发现部分文件或 hash 漂移，不能证明文档中的决定来源正确。检查应该更新的产物是否真实更新、是否登记、当前 revision 是否对应审查对象、Timeline 是否把语义变化保存为历史，以及 Runtime 是否把未同步或失效的产物继续显示为可信。

### Phase 完成完整性

```text
Phase 目标与来源 → 实际变化 → 源产物修订 → 聚焦验证 → Phase Review → completed 状态
```

包含实质实现的 Phase 在完成前接受轻量 Review；极小机械工作可以和最终交付 Review 合并。若 Plan 已标 completed，但代码、决定、文档、聚焦证据、阶段 finding 或 Owner decision 仍未收口，完成状态就是失真。Review Phase 自身不递归要求再建一轮 Review。

### 修复路由

- 实际行为错误或越界：`implementation`；
- 当前模型遗漏、过期或拥有位置错误：`source-artifact`；
- Artifact、Context、Timeline 或状态没有同步：`runtime-state`；
- 缺少用户明确选择：`owner-decision`；
- 实现证据推翻需求、设计或契约：`re-shape`；
- 错误前提已经持续污染多个下游：`rewind`。

一项 finding 可以有多个修复落点。比如代码和设计都扩大了范围但没有用户决定，需要先 `owner-decision`；用户拒绝时再同时修 `implementation` 与 `source-artifact`。不要只写“同步文档”或“和用户确认”，让下一位重新判断完整因果链。

## 对象特定见证

### 文档、设计和计划

代码之外的产物也需要能走通的失败路径：

- 需求：某个价值或边界不可观察，导致不同实现都能自称完成；
- 设计：缺少 owner、接口、状态、迁移或失败语义，迫使实现者临场决定；
- 计划：故事依赖未确定的结构，顺序无法验证，或把范围外工作静默吸收；
- 测试报告：用例缺失、证据路径打不开、动作与结论不对应，导致绿色状态无法复核。
- Longrein 当前产物：代码、用户决定、requirements、design、contract、migration、Plan、专业报告与 Runtime 投影互相冲突，或内容虽然同步却没有取得正确授权。

### Skill、提示词和长期规则

```text
真实触发输入 → description 是否发现正确能力 → 正文、reference 与相邻规则怎样影响判断 → 可观察行为误差或长期副作用
```

检查触发描述与正文职责冲突、多个 Skill 争抢同一信号、新旧规则互相抵消、临时任务要求被晋升为长期规则、示例或语义锚点外溢，以及复制安装后引用不可达。行为改进必须能在原失败和邻近反例中看见差异；只让一个样本更像预期、没有证明通用原因时，不把措辞偏好报告成改进。

## 问题项记录

问题项的完整字段和它在最终报告中的位置遵循 [Review 产物 demo](templates/review.demo.md)。本文件只定义怎样构造和审核证据，不维护平行模板。

被证据推翻的候选问题不分配优先级、scope relation 和处理方式；只有会影响审查可信度的高信号误报才进入 Finding Audit 的 `overruled` 部分。真实但不在本次落地的问题保持 confirmed/partial，并用 scope relation 与处理方式表达，不扭曲事实。

复现依赖未提供的 schema、配置或外部契约时，明确写出假设。一个在自选 fixture 中通过或失败的实验不能替代真实项目契约；该契约会改变裁决时，把它留作 evidence gap。

没有发现问题也是有效结果，但要说明实际检查的对象、覆盖面、镜头和证据边界；“未发现问题”不等于证明没有问题。
