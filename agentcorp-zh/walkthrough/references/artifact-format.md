# 讲解产物格式

SKILL.md 涵盖为什么；此文件涵盖如何。编写产物前加载。

## 形式

- **HTML（默认）：** 一个自包含的文件——所有 CSS 和 JS 内联，无网络请求，无外部字体或 CDN。响应式单页长文，有章节标题和目录；顶层结构不使用标签页。代码放入 `<pre>` 块（如果使用 styled `<div>`，必须设置 `white-space: pre-wrap`）。图表为内联 HTML/SVG 示例数据——简化的 UI 示意图、系统草图、前后对比；不用 ASCII 艺术。对关键概念、定义和边缘用例使用高亮框。
- **Markdown（`output_format: md`）：** 相同的五节；图表用 Mermaid（交付前验证语法）；测验选项和答案放在折叠的 `<details>` 块中，使答案在打开前保持隐藏。
- 命名：任务根下 `walkthrough/<slug>.html`，或独立运行时 `teamspace/walkthroughs/<YYYYMMDD>-<slug>.html`。slug 命名变更，不是分支。

## 章节契约

1. **背景** —— 讲授已经存在的内容，从宽到窄。从新来者进入此子系统需要开始的地方出发（它是做什么的、谁调用它、正常路径），然后窄到变更落地的确切上下文。在读者能将旧世界装进脑海之前，不得出现变更本身的任何细节。首次使用时展开每个任务内代码（故事 ID、发现 ID、内部产物名称），或者更好，根本不要用它们。
2. **直觉** —— 变更的本质，任何代码之前。说明变更解决的问题和解决方案的形状，用通俗语言，然后追踪**一个端到端的玩具示例**：一个具体输入、旧代码做了什么、新代码做了什么、以及为什么差异才是要点。更清晰时优先用小图加示例数据，而非 prose。
3. **变更，作为故事** —— 按*想法*展开的讲述顺序：关键决策优先或数据形状变更优先，无论哪种顺序让每个后续步骤感觉不可避免。不用文件顺序，不用字母顺序。Prose 在 hunk 之间带领读者：为什么这个片段存在、它实现了什么。展示的每个 hunk 引用其真实的 `path:line` 锚点（来自实际 diff——不要手数行）；承载 no idea 的 hunk（重命名、机械 fallout）每行一句话总结，带路径，不粘贴。
4. **行为变更和风险** —— 现在行为不同的紧凑、完整记录：新/变更的输出、错误行为、性能或兼容性影响，以及——最有价值的——与 diff 不显示的已有代码路径的交互。没有时各类写“none”。覆盖诚实：范围内的任何未解释事项，都在此处列为明确未覆盖并附原因。
5. **测验** —— 5-8 道选择题，每道 3-4 个选项，答案在读者提交前隐藏（HTML：点击揭示；md：`<details>`）。每个选项的揭示说明为什么对或错。难度：足以要求真正理解，无陷阱，无 trivia。

## 测验机制

- 瞄准意外表面：依赖已有代码路径的行为、边缘用例和失败模式、“追踪此输入”问题、“为什么选择形状 A 而不是 B”。至少一道题必须要求**背景**节的知识——这就是你检测跳过背景的读者的方式。
- 禁止：可通过扫 diff 回答的问题——文件名、行数、符号拼写、任何计数。
- 关卡：如 SKILL.md 的“测验关卡”（满分；重读后答对的变体清除 miss；`approved`/`skipped` 在 `task.md` 的 Gate History 中）。独立运行时，产物以最终“Gate outcome: approved <date> / skipped by sponsor <date>”行结束。

## 交付前自检——有任何一项命中，回去重写

- 读者尚未被教会周围系统之前，第一个代码块就出现了。
- 第 3 节的顺序镜像了 `git diff --stat` 的输出，而非想法的结构。
- 展示的 hunk 没有 `path:line` 锚点，或锚点是估算的而非来自 diff。
- 你知道的行为变更既未出现在第 4 节，也未出现在未覆盖列表中。
- 任何测验题无需阅读产物即可作答（trivia），或没有一道题要求背景节。
- 产物依赖网络资源、外部字体或赞助人可能没有的其他渲染器。
- 讲解被直接交付到终端行内，而非持久化到文件。
- 任何位置出现了未解释的任务内代码（T-xx、F-xx、ST、内部产物名称）。
