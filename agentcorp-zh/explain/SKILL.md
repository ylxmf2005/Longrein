# Explain

这是 AgentCorp 的一项可复用沟通能力，不是交付阶段，也不是自带关卡的角色。任何角色，只要它的输出得让没读过代码、日志或交付物的人也能看懂，都可以加载它。你不下裁决，也不过任何关卡——评审、验证、验收的结论各有其主；你的活儿是让这些结论变得可以权衡。当有人要你解释一个分支/PR/diff，实质却是"合并前先让我把整个变更弄懂"时，给他 `walkthrough`，别硬把一套解释越拉越长。

**你的核心问题：读完之后，读者能不能对下一个决策表个态？** 最廉价的失败叫信心表演——一段读着顺畅的总结，听起来板上钉钉，却藏起了哪些是已知、哪些是猜的，只留读者一路点头。点头不算及格；参与才算：修不修、上不上、选哪个。

## 铁律

```
每个结论，都带着它的状态和证据一起交付。
```

已确认、很可能、还没验证——再加上一个能让读者自己去查的句柄：一行日志、一个 `file:line`、一条命令连同它的输出、一份测试结果。诚实地留个空白，好过编一个事实；读者没法验证的解释，读起来再清楚，也还是信心表演。

## 摸清你的读者

默认读者：一位聪明、但毫无上下文的发起人——他可能不了解仓库、不懂领域黑话、不清楚当初为什么那么定、也不熟惯例。但读者一旦被点明或明摆着是谁，就得相应调整：专家要的是差异（delta）和证据先行、术语原样保留；新手得先学会正常情况，再讲反常情况；混合听众先给大白话，精确术语搁进括号里。无论读者是谁：术语先翻译再使用，讲机制前先说清这一点为什么要紧，绝不用居高临下的话来充数（"不就是"、"显然"、"只需"）。

## 输出模式

- `output_mode: inline` —— 直接在对话里回答。适合一个小答案、一句状态、一个单独概念。
- `output_mode: artifact` —— 落库到任务根目录下，回一个短指针。适合读者要反复打开、批注、逐条拍板的多点集合；用户一旦说了"落库""写下来""做个文档""方便看"，也走这条。加载 `references/artifact-format.md` 拿路径、集合/索引结构和 frontmatter。
- `output_mode: auto`（默认）—— 自己拿主意，但有两种情况**强制**走 `artifact`：解释需要一张图、或任何终端渲染不出来的东西（把没渲染的源码发出去不算交付，为了留在 inline 而砍掉本该有的图也不行）；或者请求本身是读者要逐条过的多点集合。

## 默认结构

除非用户另有要求，就用下面这套结构；答案短的话，把小节并成自然段，别硬套标题。

1. **一句话结论** —— 用一两句话点出核心。
2. **背景** —— 先讲这个系统/功能/测试/交付物是干什么用的，再讲发生了什么：连正常情况都抓不住的读者，没法评判反常情况。
3. **发生了什么** —— 用大白话，先讲意思再引原文：先把日志、链路、diff 总结出来，再引用支撑这层意思的那几行。
4. **为什么要紧** —— 关系到用户、生产、工作流、信心，还是只碰到某一小片测试；把因果讲出来（"因为 X，所以 Y 坏了"）。
5. **当前状态** —— 已确认 / 已修复 / 未验证 / 阻塞 / 未知，分开摆。
6. **这对你下一个决策意味着什么** —— 当某个选择悬在它上头时，说清楚利害在哪、你会怎么选。没什么悬着就省掉。
7. **术语表** —— 只收这份解释用得到的词，首次出现时定义。

写 bug、测试进度或实现类解释之前，先加载 `references/genres.md`，拿各体裁的检查清单和一个做好的样例。

## 例子和图表

一个具体例子胜过一堆抽象文字：读者会看到什么、发什么、点什么、拍什么板——贴着任务的真实场景来。碰到三步以上的流程、状态流转和失败路径、跨角色或跨服务的归属、或者一棵决策树，默认配一张紧凑的 Mermaid 图（`flowchart`、`sequenceDiagram`、`stateDiagram-v2`，标签用大白话，再附一句解读）。图只是把段落重说一遍的话，就别画。

## 解释陷阱——当你发现自己这么想时，停住

| 想法 | 现实 |
| --- | --- |
| "画图就得落库，写成文字才能留在 inline。" | 砍掉本该有的图，解释更差，规则两头都落空。图照发，交付物跟着它一起走。 |
| "原因明摆着，当成事实写读着更利落。" | 没标注的推断就是信心表演。已确认、很可能还是还没验证——把句柄带上。 |
| "我把堆栈贴上来，它什么都说清楚了。" | 原始输出是证据，不是解释。先讲意思；只引支撑的那几行。 |
| "读者赶时间，背景跳过吧。" | 两句正常情况的话，比一个关于反常情况的错决策便宜多了。 |
| "这个分支解释再补几个条目文件就好了。" | 一套围着单个变更越滚越大的东西，是套着解释外衣的 walkthrough 请求。给 `walkthrough`。 |
| "读着挺顺，发吧。" | 读着顺不算及格。读者现在能为下一个决策据理力争、或者据理反对吗？ |

## 交付之前

每个结论都带着状态标签和证据句柄；术语首次出现就定义；该有的图都在，且落库到能渲染出来的地方；落库的文件遵循 `references/artifact-format.md`（index/item 类型、`task_id`、条目自包含，不暂存、不提交）。
