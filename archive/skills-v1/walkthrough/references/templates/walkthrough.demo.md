# Walkthrough Markdown 产物 Demo

这份文件是 `walkthrough/walkthrough.md` 的唯一结构来源。中间章节必须跟随对象和人的理解顺序，不机械保留示例标题；开头的真相边界、连续主线、证据边界和交接信息不能静默缺失。

```markdown
# <对象名称> Walkthrough

## 先知道什么

- 对象与版本：<实际检查的需求、设计、代码、diff、分支、故障或成果>
- Task Context：context r<revision> / scope r<revision> | not-applicable
- 面向读者：<读者当前认知位置>
- 读完后的判断：<读者需要能够理解、质疑或决定什么>
- 真相边界：<证据对应的时间、状态和未覆盖范围>

## 背景与核心关系

<说明对象为什么存在、原来怎样工作，以及理解后续内容必须知道的局部概念。>

## <沿对象组织的连续主线>

<从入口或前提开始，连续走过权限、状态、数据、文件、异步过程、消费者和最终结果。根据对象拆成相邻章节，但不按文件列表或工作阶段复述。>

## 变化与传播范围

<只有存在前后变化时使用。分别说明新增、变动、删除、未变的行为怎样影响完整链路；没有变化时删去本节。>

## 证据与理解边界

- 关键证据：<代码、请求、数据、图、测试、评审或运行结果及其支持内容>
- 未覆盖：<没有检查或仍有争议的表面；没有则写 none>
- 开放决定：<仍需用户、Shape、Review 或 Test 介入的判断；没有则写 none>

## 接下来怎样判断

- 当前可以确定：<读者现在有资格作出的判断>
- 仍不能确定：<缺失证据或尚未完成的工作；没有则写 none>
- 下一步：<应读取、调查、实现、评审、验证或决定什么>

## 理解检查

<只有本文支撑高代价决定时使用：让读者预测一个关键输入、状态或失败会发生什么，以及为什么。否则删去本节。>
```
