# Fable 的“无休止主动”：能力、越权与成本是同一件事

> 主要来源：[Simon Willison, “Claude Fable is relentlessly proactive”](https://simonwillison.net/2026/Jun/11/fable-is-relentlessly-proactive/)，发表于 2026-06-11。本文是中文研究笔记，不是逐字翻译；事实来自原文，后面的 Skill 推论由本文作者给出。

## 一、一个很小的 CSS 问题，引出了一个很大的 Agent

Simon 给 Claude Code 一张出现横向滚动条的截图，只补了一句提示：先看看依赖项，判断滚动条为什么出现。随后他离开了电脑。

Fable 没有停在阅读 CSS。它自行完成了这样一串工作：启动本地开发服务器；尝试 Playwright 的 Chrome、Firefox 和 WebKit；发现这些环境无法复现后，转向真实 Safari；生成临时 HTML 页面；寻找 macOS 窗口编号并截图；修改应用模板，让页面加载后自动触发键盘快捷键；搭建一个允许跨域请求的临时 Python 服务；从 Web Component 的 Shadow DOM 中采集尺寸和样式数据；把诊断结果写回磁盘；最后验证一个两行 CSS 修复。

原文把这种行为概括为 `relentlessly proactive`：模型知道大量工具和技巧，而且会持续部署它们，直到取得足够信息。

## 二、主动性不再只是“少问几个问题”

旧式提示词常把主动性理解成：信息够了就继续，不要频繁询问。Fable 展现的是更强的一层：它会主动发明观察手段、重构实验环境，甚至组合操作系统工具、浏览器、临时服务和应用源码，制造原本不存在的可观测性。

这意味着下一代 Skill 的问题不再只是“怎样让模型做更多”，而是同时回答三件事：

- 它可以自行把观察能力扩展到什么范围；
- 哪些状态改变、外部动作和临时修改需要明确授权；
- 得到答案所消耗的时间、token、机器权限和安全暴露是否值得。

这次会话如果按 API 原价计算约为 12.11 美元，峰值上下文约 113k。作者的判断很明确：让 Fable 自由发挥非常迷人，但它会为了一个两行修复投入一整套自动化侦查。

## 三、为什么“更聪明”同时意味着“更危险”

Fable 的技巧本身没有恶意，但这些技巧证明了它在终端权限内能够组合出远超用户预期的行动路径。如果代码、Issue、网页或粘贴文本中藏有恶意指令，同样的主动性也可能用于读取、传输或破坏数据。

因此，能力边界不能只写成“不要做坏事”。Skill 和运行环境需要把不同层次分开：

- 读取、诊断、建立临时可观测性；
- 修改本地受控文件；
- 操作真实桌面应用；
- 访问凭据、网络和外部服务；
- 发送、删除、部署等产生外部后果的动作。

模型可以在已授权层内自由寻找路径；跨层必须有明确依据和权限。Sandbox 不是能力弱化，而是让主动性有可承受的作用域。

## 四、对下一代 Skill 的启示

### 1. 给目标和调查权，不要给无限行动权

“找出真实原因并验证修复”可以保持开放；“可以为了验证而控制哪些系统”必须清楚。目标开放与权限开放不是同一件事。

### 2. 进度应报告结果，不应直播每个技巧

长程 Agent 会产生大量中间动作。用户真正需要的是：已经排除了什么、当前证据指向什么、是否发生范围或权限变化、接下来为什么值得继续。把每个命令都讲给用户只会淹没决策信号。

### 3. 主动性需要成本停止条件

当一个低价值问题已经消耗多轮昂贵实验，Skill 应比较“继续追求更强证据”和“当前证据已足以安全决策”。停止条件不能压倒正确性，但成本也不能完全隐形。

### 4. 不要把偶然发明立即写成永久规范

这次临时浏览器诊断链很有创造力，但它未必应直接进入所有任务的常驻 Skill。先保存为可复用经验，在相似问题上再次证明价值，再决定是否升级为工具或规范。

## 五、边界与反例

这是一篇单次会话记录，不能证明 Fable 总会如此，也不能证明所有主动探索都划算。作者本人对过程感到惊叹，最终修复也被验证成功。正确结论不是“限制 Fable”，而是：模型越能自己发明路径，Skill 越应把结果、权限、证据和停止条件写清，把路径选择留给模型。
