# evolution

## 让这轮代价改变下一轮

Evolution 读取真实任务中的决定、行动、失败、证据和后果，判断其中是否存在能让未来工作不同的经验。总结发生过什么不算进化；只有经验进入正确作用域、并实际改变未来行为，才值得保存。

先保护当前任务。只有长期能力缺陷正在继续放大错误时才立即介入；其他发现等到自然停顿或任务收尾。

## 让经验找到正确作用域

先问这项经验准备改变什么未来行为、影响多大范围、何时会失效，再决定保存形式。

当前任务事实与承诺留在 `task.md`；稳定项目事实和决定进入项目规则或文档；稳定重复劳动优先变成结构、检查或工具；只有跨项目可重放的协作机制才修改 Skill。保存动作不取得新的产品范围、长期规则修改权或执行授权，能由系统执行的经验也不只写成更强语气的提醒。

## Skill 变化必须有证据

看到 Skill 相关误差时，先区分能力是否被发现、是否正确触发、加载后是否被遵循，以及 Skill 本身是否真的有缺陷。问题不在 Skill 正文时，修复实际失效的层。

行为性 Skill 变化读取 [evidence-driven-evolution.md](references/evidence-driven-evolution.md)，用真实轨迹、相邻反例和变化前后证据判断是否改善，并让用户批准具体变化范围或 diff。断链、格式、历史状态和 UI 元数据等机械维护使用能够直接证明结果的静态检查。

当偏差涉及调查授权被扩成实现授权、AD 被摘要或文档改写成 UD、或最小实现被假想能力扩大时，同时读取 [决定边界行为评测](references/decision-boundary-eval.md)，使用真实失败与相邻反例检查候选规则是否既能阻止越界，也不会把明确授权误判为必须再次询问。

未经批准的变化保持候选；没有验证结果只能称为已修改，不能称为已改进。

## 落地后保持一致

更新长期能力时，同时检查相邻 Skill 的触发和职责，避免重复争抢或相互否定。删除不再改变行为的旧规则，不用新规则覆盖旧规则。

## 产物

没有可复用经验时不制造文件。任务内需要保留候选、评测、授权或落地状态时，使用：

```text
<task-workspace>/evolution/
├── evolution.md        # 经验、作用域、状态、授权和验证结论
├── evals/              # 只有行为变化需要重放时创建
└── evidence/           # 必要的原始输出、diff、日志或静态检查
```

`evolution.md` 遵循 [Evolution 产物 demo](references/templates/evolution.demo.md)，记录真实行为偏差、准备改变的未来行为、选择的作用域、用户授权、前后证据和失效条件。该 Demo 是产物结构的唯一来源；没有相应内容时不创建空目录或占位文件。

独立研究 Longrein 自身、又没有所属任务时，沿用已有 `studio/evolution/<YYYYMMDD>-<slug>.md`；存在当前 Active Task 时记录使用的 `context_revision` 和 `scope_revision`，持久化前用 `longrein_work_start` 建立工作单元，落盘后用 `longrein_checkpoint` 登记 artifact 并 finish。不要直接编辑 Runtime 核心文件。

## 边界

Evolution 只决定这轮经验怎样改变未来，不重新打开已经完成的产品范围。
