# 用户流程测试参考

在运行完整的面向用户的流程时加载本文。

## 测试姿态

把自己当作一个有目标的真实用户：从外部驱动产品，报告你个人体验到的行为，而不是用"看起来没问题"的源代码作为流程通过的证据。

## 执行方式

SKILL.md 制定了规则——浏览器为主、API/DB/日志为辅，仅在 E2E 手册声明了降级方案时才能用 API 驱动执行。实践中：使用真实浏览器和已登录会话，遵循 E2E 手册（`test/e2e-test-plan.md`）中的步骤，将每一步的实际行为与手册指定的预期行为进行核对，并手册标注的截图点捕获证据。当环境未就位（服务未启动、路由未切换、数据未初始化）时，将对应流程标记为 blocked 并说明缺口；当使用了声明的降级方案时，结果必须说明该证据层无法证明什么。

## 如何运行一条流程

对于每条被指派的流程，先敲定前置条件——入口点、角色人设、环境、凭证、数据初始化；不要在现场编造手册未指定的任何前置条件（比如应使用哪个对象 ID）——将其作为缺口报告。然后让流程以"观察 — 行动 — 再观察"的节奏推进：首先看清起始状态，执行一个用户操作，等结果稳定后再执行下一个，并在每一步记录"预期 vs 实际"——直到达成目标、某一步的实际行为未能满足预期且后续步骤依赖它（记录失败步骤和输入，标记流程为 failed），或者你被阻塞（命名缺失的环境、数据或观察面）。绝不要在记录中未将其归类为 failed 或 blocked 就放弃一条流程。关键是验证流程的每一步，而不仅仅是最终结果。对于手册要求逐字输入的步骤（例如发送给系统的提示词），使用逐字文本，不要改写——改写输入测试的是计划所评估路径之外的其他路径。

## 按界面类型收集证据

- Web：URL、可访问的页面状态、状态变更或失败时的截图、相关的控制台/网络错误。
- CLI：命令、stdout、stderr、退出码，以及必要时记录延迟。
- API 作为用户流程：请求序列、响应、状态码、持久化结果。
- Desktop：在支持的情况下提供截图/窗口状态。

## 人工测试执行日志

该条目列表是执行记录的权威检查清单——SKILL.md 以此为准。每个已完成或被阻塞的场景写一条记录。把具体执行过程放在结论之前：

- 背景：角色人设、用户目标、环境、页面或入口点，以及这个场景为什么重要。
- 前置条件：所选数据、账户/凭证引用、功能开关、配置备份，以及允许哪些写入操作。
- 操作：精确的用户步骤、字面输入、命令行或脚本路径。如果页面上下文 API 是流程的一部分，需包含方法、路径、请求体、凭证模式和时间戳。
- 响应：状态码、响应消息、关键 body 字段、追踪/请求 ID，以及持久化状态的回读。大型 body 可以总结，但产生它的精确请求必须保持可见。
- 观察：测试者个人在页面、日志、审计、邮件/聊天/推送或其他通知面所看到的内容。对于人工观察，记录确认者、时间戳和内容摘要。
- 清理：恢复了什么、用于恢复的请求或 UI 操作，以及证明最终状态的回读。
- 证据边界：这条记录证明了什么，以及它没有证明什么。

对于反面检查，说明观察窗口和来源（例如"从 15:00 到 15:05 观察聊天通知，未看到匹配 X 的消息"）。如果没有可靠的观察面，将检查标记为 `needs_more_evidence`（按 SKILL.md 的交接要求记录在 Blocked checks 下）或 `blocked`；不要把反面路径称为 passed。

## 角色人设

- 新手（Novice）：跟随可见标签，耐心低，报告困惑和缺失的 affordance。
- 高级用户（Power user）：深入文档/设置/快捷键，尝试变通方案，对性能和一致性敏感。
- 对抗性用户（Adversarial）：探测边界——重复操作、无效输入、授权、信息泄露。

使用测试计划指派的人设；当未指派时，选择最符合用户可见风险的人设并说明原因。
