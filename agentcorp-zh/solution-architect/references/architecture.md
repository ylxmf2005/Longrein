# 架构设计

在实现之前设计系统结构。目标是在成本还低时敲定结构性决策，这样后续工作不会一直建立在早期错误之上。

## 你要对抗什么

复杂度是头号敌人。良好结构沿三个轴压制它：

- **变更放大** —— 简单改动不应 ripple 到多处。本地化变更：一个决策落在一个地方。
- **认知负荷** —— 开发者不应为了安全改动一处而把整个系统装在脑子里。深层模块把复杂度藏在简单接口之后。
- **未知未知** —— 一眼就能看出改动需要到哪里、以及你需要知道什么。暴露依赖关系，并在组件相遇的地方显式化契约。

 mindset：把复杂度向内拉入模块，不要向外推给调用者。最好的接口是"让调用者无需了解内部如何工作就能达成目标"的最简单接口。尽早把结构做对，因为每个建立在 leaky interface 之上的新调用者都是将来要多 rework 一个地方。

当具体判断（深度、隐藏、分层、命名）不明确时，从 `principles/` 中查阅相关文件。

## 这份产出必须达成什么

在需求之后，这是人类最直面的一份产出。发起者必须能信任这份设计，规划者必须能基于此进行下游构建而不需要反向工程代码。所以它必须清楚说明（以最能服务于设计的结构组织）：

- 要解决的问题，以及此设计的意图；
- 关键决策，以及为什么如此做出；
- 每个组件拥有什么、隐藏什么；
- 哪些接口与契约必须保持稳定；
- 数据或状态在影响边界的地方如何流动；
- 在相关处，存储/表结构/API 契约及其兼容性行为；数据表与数据模型的主体以贴近项目技术栈的代码块承载（DDL、ORM、Pydantic/TypeScript schema，或 pseudocode）；
- 引入了多大复杂度，以及这个结构如何压制它；
- 加上风险、约束，以及任何影响验证的因素。

给出足以让人信任设计的细节，不要更多；细节密集处使用代码块、表格或列表。按 `references/mermaid.md` 的指引使用图表：只要视图比文字更清楚地表达结构、流程、状态、所有权或变更前后，就画一张——但保持集合在传达设计的最小规模（通常 2–3 张）；对于变更标注图表与数据流序列，参见 `references/mermaid.md`。

如果需求或现有代码过于模糊，无法自信设计，返回 `blocked` 并说明具体缺失什么证据，而不是发明它。

## 产出

将产出写到任务分配的 `output_path`（通常是 `design/architecture.md`），遵循 `design-artifact` demo 模板。产出是语言无关的；重构时遵循被改动代码的语言。文本中的契约示例仅用于说明设计——不要写实现文件。

如果是重构，简要说明原方法有什么问题以及此方案如何解决。
