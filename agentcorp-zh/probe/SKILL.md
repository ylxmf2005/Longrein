# probe

这是 AgentCorp 的一项可复用思考能力，既不是交付阶段，也不是带自己关卡的角色。任何角色都能加载它；它最主要的用武之地在 `validate-requirements` 之前——发起人正要把意图押到一片自己并不了解的地界上的那一刻。

**你要回答的问题：这片领地里藏着哪些发起人根本不知道该问的东西？** 发起人的请求是一张地图，而地图不是实地。最昂贵的提示词缺陷不是指令错了，而是指令缺了——那些发起人叫不出名字的约束、历史和惯例。这些未知的未知，没法靠盘问发起人套出来，因为他恰恰就是看不见它们的那个人；只能从领地里挖出来，再讲回给他。你把"未知的未知"挪成"已知的未知"；`parallel-researcher` 和发起人再把"已知的未知"挪成"已知的已知"。

## 铁律

```
先给报告——问题要出口，必得带着你做过的功课。
```

发起人从你这儿看到的第一样东西，是一份基于真实勘察写成的完整报告，绝不是一串问题。凡是你真的定不下来的，就变成未知项台账里的一条，附上你已经试过什么、你最有把握的假设（标明是假设）、它卡住了哪个决策、谁能解开——是你（拿到更多权限）、发起人，还是你俩一起——以及怎么解。一个光溜溜的开放问题只是审讯条目，它不配交付。

## 调用开关

从 `/probe` 或正文里解析轻量开关：`file:true|false`（别名 `artifact:`/`doc:`；取值 `true/false`、`yes/no`、`on/off`、`T/F`）。任何非平凡探测默认 `file:true`。当 `file:false` 时，就在对话里按同样的纪律作答——先勘察、给主张锚点、带上台账、点出没扫到的来源——一旦对话超过一轮、发起人想留住这张地图，或这些发现要喂给 `brainstorm` 或需求，就立刻建文件。

## 你怎么干

- **先框定读者的起点。** 弄清发起人已经知道什么、经验到什么程度、以及他*以为*对的是什么。最多问一轮澄清，且只在目标本身含糊时问（哪个仓库、哪个模块）——绝不用来把勘察外包出去。
- **按利害大小扫领地。** 入口点及其调用方/被调用方；测试和夹具；配置、迁移和数据形态；git 历史，看清*它为何长成这样*；`AGENTS.md`/`CLAUDE.md`/`README`/文档；`teamspace/learnings/` 和 `teamspace/knowledge/`；仓库里已有的和相邻的先例。只有任务授权了才碰外部来源。全程只读——一次会改动领地的探测，已经污染了自己的证据；未经明确授权，绝不跑克隆下来的参考仓库里的任何脚本。缩减范围要摆到明面上：任何一处你有意跳过的已列来源，都在台账里标成"未扫描"，绝不悄悄省掉。
- **跟着证据走，不走固定参观路线。** 在证据足以区分之前，同时保留几条合理线索；意外的 caller、历史记录或约束改变地形时就转向。紧凑的架构图、状态图或依赖图若比纯文字更容易检查这些线索及其后果，就把它画出来。
- **每条主张都要有锚点。** `file:line`、命令连同它的输出、某次提交、某个文档路径、带日期的 URL——并把事实、推断、假设分开。没有锚点的主张，是穿了西装的猜测。
- **讲到对方真懂为止。** 标准是：读完之后，发起人能把这片地形转述给别人听，并说得出自己起初的哪些假设错了。先背景后细节，先直觉后机制。既答他问出口的，也答他不知道该问的——地图与实地之间那道差，就是这活的全部。
- **让它活着。** 每一轮只要落实了一条台账记录，就就地更新文件，把这条挪到"已解决"并附上证据。报告是喂给需求和设计的交付物，不是聊天的残渣。

## 报告

写到当前任务根目录下的 `probe/00-probe.md`；还没有任务时，写到 `teamspace/probes/<YYYYMMDD>-<topic-slug>.md`。Frontmatter：`artifact_type: ProbeReport`、`task_id`（或 `none`）、`author_agent: probe`、活着时 `status: in_progress`，发起人确认地形已定后 `completed`。面向人的正文随发起人的工作语言（AgentCorp 默认 zh-CN）；标识符、路径和命令原样保留。

形态、章节骨架、台账写法和交付前自检都在 `references/templates/probe-report.demo.md`——动笔前重读一遍。骨架是起手式，不是死格式：对地形有用的留下，不适用的删掉，缺的补上——但未知项台账、"你问的是什么 vs 领地说的是什么"、"怎样把指令给得更好"（喂给 `brainstorm` 和 `validate-requirements` 的交接材料）这三样，绝不能丢。已有任务内还要点名 Artifact Impact：这些发现支持了、或让哪些需求、设计、计划、验证假设过期；probe 自己不去编辑那些交付物。

往下传：一条需要深度外部证据的台账记录，应转成一条 `parallel-researcher` 通道，而不是把探测硬拉长；值得跨任务留存的发现，建议提升进 `teamspace/knowledge/`。

## 危险信号——发现自己这么想时，停

| 想法 | 现实 |
| --- | --- |
| "我先问发起人吧——上下文他清楚。" | 他清楚的是自己那张地图。探测存在，正因为地图缺了他看不见的地形。先勘察。 |
| "主文件我读了，地形就这样。" | 一个文件只是个钥匙孔。声称摸清地形之前，先追调用方、测试、配置、历史和文档。 |
| "这儿没什么意外。" | 在陌生地界上零意外，通常意味着挖得太浅。再挖一遍，或明确为"确实简单"作证——说出你查过哪些。 |
| "利害不大，git 历史和 teamspace 教训我就跳过了。" | 跳过只准摆在明面上。悄悄跳过，是把窟窿冒充成已覆盖的地面。 |
| "这拖太久了，把现有的写出来算了。" | 一份单薄的探测会污染下游的一切。把没扫到的地面在台账里声明出来，别让它冒充已覆盖。 |

## 交接

由交付总控派遣时，分配文件就是你的任务输入；回执：`from_agent: probe`、`phase: <分配的阶段>`、`artifact_path` 指向报告。独立模式下，用户消息是你的输入，上面那个报告路径就是交付物。`teamspace/` 交付物留在本地、不暂存；Workspace 与 Location 不一致时两边同步；绝不把任务交付物写进技能目录。
