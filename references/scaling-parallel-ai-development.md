# 扩展 AI 开发的真正方法：先减少人类介入，再增加 Agent 数量

> 来源：[atum《我用几千美金 token 换来的并行开发经验：让一群 AI Agent 帮你写代码》](https://atum.li/cn/blog/how-to-scale-ai-dev/)，发表于 2026-04-23。本文保留其框架、案例、实践细节和限制，用另一套语言重新组织。
>
> 边界说明：文章中的时间线、模型能力和并发数量来自作者截至 2026 年的个人实践；“TPDD”是作者为便于与 TDD 对照而使用的名称，不是通行行业标准；文件重叠比例等数字是经验规则，不应当作普遍定律。

## 一、三条看似激进的团队规则

作者在团队中提出了三条规则：

1. 不手写代码；
2. 不盯着 AI 写代码；
3. 不在传统代码编辑器里逐行 review AI 代码。

这些规则的目的不是否定编程能力或代码审查，而是迫使团队重新设计质量保障方式。如果 AI 只是替你敲键盘，而需求澄清、过程监督、逐行检查和架构控制仍由人串行完成，那么增加 Agent 数量不会带来真正的并行。

人仍然是单点瓶颈。

并行开发的核心问题不是“怎样同时启动更多 Agent”，而是：

> 怎样让每个 Agent 在更少的人类实时介入下，独立完成可以被验证、可以维护的工作？

只有回答这个问题，并行才有意义。

## 二、限制并行的三个瓶颈

### 1. 需求如何准确传给 Agent

人的想法通常模糊、隐含并携带大量没有说出口的上下文；Agent 需要明确、可以执行的目标。双方都可能误以为已经理解，直到成品出现才发现方向不同。

### 2. 如何证明功能正确

AI 代码往往“看起来像对的”，真正运行时却可能存在边界遗漏、状态错误或集成问题。过去的解决方式是人逐行检查，再把问题反馈给 AI 修改。

### 3. 如何控制长期可维护性

一次能运行不代表代码库能持续演进。如果模块边界、抽象、命名和工程纪律不统一，多个 Agent 会快速制造结构混乱、职责泄露和修改连锁反应。

三项瓶颈过去都依赖 human-in-the-loop，而人的注意力是串行且有限的。五个 Agent 同时工作，最后仍然要排队等同一个人回答问题、审代码和决定架构。

图示表达的是：旧模式中，所有 Agent 的工作流都汇聚到人身上；新模式要用机制承担大多数日常检查，人只设定方向、标准并处理关键例外。

## 三、从 Copilot 阶段到自主开发阶段

在较早的 AI Coding 阶段，人是驾驶员，AI 是副驾。

需求通常靠两种方式传递：编写尽量完整的 specification，或者通过多轮对话不断纠正。正确性依靠人工 code review；架构与代码组织也由人逐项指定。

这个阶段的工具重点是改善协作界面：更方便地聊需求、更方便地查看 AI 修改。并行并没有解决根本问题，只是让多个等待队列同时出现。

作者认为，2025 年下半年前沿模型在四个方面跨过了一个实践门槛：

- 自主调试：能运行测试、阅读日志、追调用栈并反复修复；
- 需求遵循：需求理解正确时，更容易通过自测得到正确实现；
- Skill 约束：能够在多数常规情况下遵守结构化工程规范；
- Computer Use：能操作终端、浏览器和 GUI，不再局限于修改源码。

这并不表示人可以完全退出。变化是人的参与形态从“逐行监督”转向“设定规则、审核验收方式、检查关键节点”。

## 四、失败变便宜以后，策略也应该变化

人类实现两套候选方案的成本通常太高，因此过去必须在设计阶段尽量一次想清楚。Agent 的尝试成本低得多，于是出现了新的工程策略：

> 想不清楚时，不一定继续空想；可以让多个 Agent 实现不同方案，通过真实测试和结果比较再决定。

“多路并试，择优使用”不只是速度上的量变，而是决策方法的变化。并行有时不是因为任务天然可拆，而是因为实验比争论便宜。

这并不等于无脑生成多份代码。候选仍需共享同一组需求、验收标准和工程约束，否则比较的只是不同方向的随机产物。

## 五、并行开发的三把钥匙

作者用三套机制分别替代三个需要人类实时参与的环节：

1. 用结构化需求对齐降低沟通依赖；
2. 用独立审核的测试计划保障功能正确性；
3. 用持续演化的 Skill 注入工程纪律。

这些机制都不能安装即用。团队必须通过真实任务磨合：记录 Agent 经常误解什么、哪些类型可以放心、哪些自信表达需要打折、哪些环节仍须人工判断。

文档化规则只是显性部分。人对模型能力边界形成的直觉，也是决定能放手到什么程度的重要资产。

## 六、第一把钥匙：把需求对齐变成独立阶段

需求对齐有两条互补路线。

### 路线一：连续追问，主动暴露盲区

先用自然语言说明目标，但明确要求 Agent 暂时不要编码。让它列出所有不确定、缺失或存在多种解释的地方。

回答一轮以后，不立即开始实施，而是继续问：根据新增回答，还有哪些新的不确定？直到问题开始重复、琐碎，核心模糊区才算基本覆盖。

最终把确认内容整理为结构化需求文档，作为后续测试计划与开发的输入。

这种方式彻底，但需要用户持续提供信息，复杂需求可能消耗十到三十分钟的深度注意力。

### 路线二：生成候选方案，由人筛选

如果很多外围细节并不值得用户从空白页回答，可以让 Agent：

1. 找出描述中的模糊点；
2. 必要时调查同类产品；
3. 为每个模糊点给出三种合理方案；
4. 排序并说明推荐理由。

例如，“实现用户注册”隐含邮箱或手机号、验证方式、密码规则、失败提示等决定。Agent 可以根据常见产品实践提出完整默认方案，人只修改真正关心的部分。

用户由“回答所有问题的人”变成“审阅完整方案的人”。审方案通常比凭空定义全部细节更快。

### 两条路线怎样组合

- 核心体验、商业规则和高风险边界：连续追问；
- 不重要的外围选择：Agent 提方案并给默认推荐；
- 最终产物：足以直接生成测试计划的完整需求文档。

需求对齐是流程中最难并行的环节，因为它需要问题所有者的判断和深度注意力。可以先串行完成多个任务的对齐，再让执行阶段并行。

## 七、第二把钥匙：测试计划驱动开发

不逐行读实现代码以后，正确性不能建立在“模型说完成了”上，而要建立在预先定义的验收契约上。

作者把这种方式称为 Test-Plan-Driven Development，简称 TPDD。

流程是：

1. 根据已经确认的需求先写测试计划；
2. 测试计划覆盖单元、集成和端到端层级；
3. 在实现开始前审核测试计划；
4. Agent 根据计划开发；
5. Agent 自行运行测试并进入调试—修复循环；
6. 所有验收条件通过后才交付。

测试计划描述“正确的系统必须发生什么”，实现代码描述“具体怎样做到”。对需求提出者而言，前者通常更容易判断，也更接近他真正拥有的知识。

因此，人类审查可以从每行实现转移到测试计划、测试结果和少量高风险模块。

## 八、Agent 必须拥有真实操作系统的能力

TPDD 成立的前提，是 Agent 能像工程师一样操作目标系统。只给源码和一个能启动的容器远远不够。

不同系统需要不同工具：

- CLI/API：Shell、日志和测试框架；
- Web 应用：运行中的服务，以及 Playwright、Chrome/CDP 或其他 Browser Use；
- 桌面/GUI：显示系统、输入控制和截图能力；
- 状态机、异步、并发和长进程：调试器、断点、变量与调用栈，而不只是不断加 print。

一个简单判断标准是：如果一名只能使用这些工具的新工程师无法复现线上问题，Agent 也无法做到。

工具能力必须与交给它的责任匹配。无法观察真实行为的 Agent，只能对正确性进行语言想象。

## 九、为什么测试计划必须独立审核

如果同一个 Agent 同时理解需求、设计测试、实现代码，它的一项误解可能同时污染测试和实现。代码错了，测试也按照同一个错误理解编写，于是全部通过。

这是 TPDD 最重要的逻辑漏洞。

测试计划进入实现阶段以前，必须经过独立审核：

- 人类审核：拥有业务语境，适合高风险任务，但消耗深度注意力；
- 新会话 Agent 审核：只看到需求和测试计划，不继承实现者上下文，便宜且可并行，能发现大量理解偏差；
- 组合审核：Agent 先过滤明显问题，人再检查关键业务边界。

低风险内部工具可以主要依赖第二个 Agent；线上关键路径、金融、安全或多人共享核心模块，应保留人的业务审核。

人需要特别检查：

- 边界条件是否完整；
- 错误与异常路径是否覆盖；
- 测试验证的是“应该发生什么”，还是把现有实现行为重新断言一遍。

关键不是审核者一定是人，而是审核者不能与编写测试和实现的是同一上下文主体。

## 十、功能测试以外，还需要专门验证 Skill

单元、集成和 E2E 主要覆盖功能正确性。并发、性能、安全、恢复能力和长期架构劣化，需要单独的方法论。

这些方法可以沉淀为领域 Skill：

- 压力测试：负载阶梯、p50/p99/p999、SLA 与退化曲线；
- 混沌测试：数据库、下游服务和网络故障注入，验证降级与恢复；
- 安全测试：XSS、SQL 注入、越权、CSRF 等常见攻击面；
- 演进回归：文件规模、函数复杂度、循环依赖和模块边界劣化。

Skill 的本质是把“在这个领域里，做对是什么样子”结构化，让 Agent 可以重复执行。

但 Skill 仍无法替代所有判断。几种架构都能通过测试时，哪一种更适合当前业务、团队和未来变化，仍是价值取舍。

## 十一、TPDD 与 TDD 的区别

TDD 的经典流程是细粒度的红—绿—重构：先写一个失败测试，以最少代码通过，然后重构，持续小步推进。它是一种指导人类开发过程的纪律。

TPDD 先为完整需求制定整体测试计划，再把编码、运行和修复闭环交给 Agent。它的粒度更粗，目标是让人能够退出实时编码过程。

可以这样区分：

- TDD：人类开发者的编程纪律；
- TPDD：人类为 Agent 设定的验收契约。

TPDD 不是对 TDD 的替代，也不是业界标准术语，而是为自主 Agent 工作流描述一种不同的质量控制层级。

## 十二、让 AI 以不同用户身份做真实测试

Computer Use 使 Agent 能按用户故事直接操作页面或 GUI，发现布局、流程、文案和交互层的问题。

重要技巧不是简单说“模拟用户”，而是定义不同用户能看到什么、不能看到什么：

- 新手：只认识屏幕上的按钮和标签，看不到控制台，失败几次就放弃；
- 熟手：主动找替代路径和快捷操作；
- 对抗性用户：尝试异常输入、超长字段、错误顺序和重复操作。

角色的盲区比角色的标签更重要。新手报告只能说“保存以后页面白了五秒”，不能用工程师视角直接解释成初始化失败。这样产生的反馈才接近真实用户。

不同角色暴露的问题不同：新手发现术语和主流程，熟手发现效率障碍，对抗性用户发现校验和稳定性问题。

## 十三、第三把钥匙：用工程纪律控制复杂度

功能正确靠测试，可维护性依靠持续执行的设计原则。作者主要借鉴 John Ousterhout 的《A Philosophy of Software Design》，把控制复杂度的原则写入 Skill。

### 1. 模块深度

好的模块接口简单，内部承担足够多的复杂度。Agent 容易过度拆分，产生大量几乎没有价值的浅模块。不要为了“单一职责”机械创建类和文件。

### 2. 信息隐藏

模块边界按“谁拥有这项知识”划分，而不是按执行时间顺序划分。一个模块不应要求其他模块了解其内部决策。

### 3. 抽象分层

每一层应提供不同的思维模型。只把调用原样转发给下一层的中间层没有价值。复杂度应尽量由下层吸收，让上层更简单。

### 4. 内聚与分离

必须共同理解的逻辑放在一起；通用规则和特例混合后彼此干扰时再分开。避免必须跳到另一个函数才能理解当前函数的耦合。

### 5. 尽量把错误从设计中消除

无处不在的异常处理会积累复杂度。能通过调整语义、默认行为或接口让某类异常不再出现时，优先修改设计，而不是继续增加 try/catch。

### 6. 命名与显而易见性

读者第一眼应能正确猜出代码作用。命名精确且全局一致，不违反项目使用者的直觉。

### 7. 文档与注释

注释解释代码本身无法表达的设计意图、取舍与抽象语义。重复下一行代码的注释是噪声；缺少关键“为什么”同样有问题。

### 8. 战略设计与反过度设计

每次修改应对整体结构有长期责任，但不能为纯粹假想的未来创建抽象。快速补丁和无需求的宏大设计都会增加未来成本。

## 十四、工程原则不能变成机械 checklist

这些原则之间存在真实张力：

- 深模块要求功能深、接口少；小而组合强调单一职责；
- 战略设计鼓励为未来投资；反过度设计要求只解决当前问题；
- 信息隐藏减少泄露；过度隐藏又可能制造看不懂能力的浅接口。

资深工程师也无法机械地永远判断正确，Agent 更不可能仅靠一份通用 Skill 解决全部权衡。

现实做法是：让 Agent 在常规场景执行规则，人 spot-check 架构级决定；观察项目中反复出现的错误，把具体反例和项目边界写回 Skill。

Skill 库随项目成长，不是一次编写后永久正确的宪法。

## 十五、可维护性的三阶段操作

Agent 获得需求文档和测试计划后：

1. 先依据工程原则设计模块、接口、层次和文件组织；
2. 开发、运行测试并修复；
3. 重新对照原则自审，检查浅模块、知识泄露、纯转发层、命名不一致和无价值复杂度。

人类在最后针对架构和长期维护风险做抽查。随着项目 Skill 吸收更多真实错误，抽查频率可以降低，但不应假设归零。

## 十六、四种并行调度模式

### 模式一：跨项目并行

不同仓库、不同 Agent，依赖最少，管理成本最低。人可以在项目 A 生成测试计划时，去完成项目 B 的需求对齐，再回来审核 A。

这种模式最适合刚开始练习并行调度。

### 模式二：同一仓库的不同方向

使用 `git worktree` 为不同任务或不同候选方案创建独立目录与分支。Agent 互不覆盖，完成后按优先级合并。

适合两种情况：

- 不同且耦合较低的功能并行开发；
- 同一需求用多种实现竞争，测试后择优。

任务修改的文件越重叠，合并成本越高。作者给出的经验判断是：重叠低于约 20% 通常适合并行；超过约 50% 更适合串行或重新拆分。这个比例应根据仓库和修改性质校准。

### 模式三：同一工作项的不同事务类型

一个功能内部，可以按工作性质并行，而不一定按模块拆分。

例如订单导出功能：

- Agent A 负责后端边界和并发测试；
- Agent B 用 Playwright 负责 UI 流程、下载和错误状态；
- Agent C 修复已经确认的现有缺陷。

三项工作使用不同工具链、影响不同区域，天然适合交叉并行。

### 模式四：Agent 自行拆分和调度

把任务直接交给主 Agent，由它识别可并行子任务、派发子 Agent、协调和合并。

作者把它视为前沿实验，而不是稳定默认。当前问题包括：子 Agent 之间上下文共享脆弱、任务边界不清时相互撞车、合并仍需人工介入。

为未来准备的方法是今天就建立清晰的模块边界、接口契约和数据流。如果几个 Agent 需要在编码中不断重新讨论接口，并行通常会比串行更慢。

四种模式可以嵌套，例如在 worktree 中再按后端测试、UI 测试和缺陷修复交叉并行。

## 十七、新瓶颈：人来不及消化 Agent 产出

并行数提高后，人可能不再审代码，却开始整天阅读测试报告、复盘、用户反馈和缺陷清单。瓶颈换了形式，又回到人身上。

理想闭环是：

1. 人给出需求；
2. Agent 并行开发和测试；
3. Agent 批量消化所有产出；
4. 合并重复问题；
5. 按类型派发后续任务；
6. 根据影响范围和临时替代方案排序；
7. 只把真正需要价值判断的高优问题升级给人。

作者坦诚表示，这一自动 triage 环节目前仍不可靠：分类会失焦；合并会误删独立问题；优先级缺少业务视角。

在闭环成熟以前，只能用缓冲措施：

- 强制每个 Agent 使用“摘要 + 严重程度 + 建议动作”格式；
- 把反复出现的低价值问题写入 Skill，让 Agent 自行过滤；
- 集中在固定时间批量阅读状态，不让报告持续打断；
- 提高审查粒度，从逐条产出升级为一批产出的综合判断。

## 十八、从多少个 Agent 开始

没有并行经验时，从两个开始。熟悉需求对齐、测试计划审核、任务隔离和状态轮转后再增加。

作者认为初学者通常在三到四个 Agent 时出现明显的上下文切换压力；熟练拆分和批量审查后，六到八个可能可行。

真正决定上限的不是意志力，而是：

- 任务是否提前拆得足够独立；
- Agent 是否能长时间自主推进；
- 产出是否使用一致结构；
- 是否可以批量审查；
- 是否只有关键问题才打断人。

并发数不是生产力指标。被大量低质量任务持续打断，六个 Agent 可能比一个更慢。

## 十九、需求对齐串行，执行并行

执行可以齐发，需求对齐需要问题所有者的深度思考。

更合理的节奏是：

1. 串行完成任务 A 的需求对齐；
2. A 开始生成或执行测试计划；
3. 对齐任务 B；
4. B 开始执行时审核 A 的测试计划；
5. 在不同 Agent 的低等待窗口轮转。

作者用“装弹与射击”比喻：装弹需要专注，射击可以齐发。

## 二十、所有 Agent 必须共享项目专属规范

工程规范、架构原则、测试要求和团队偏好应形成项目 Skill，并在每个 Agent 启动时加载。

并行产物能够合并的前提，是大家对命名、边界、验证和设计原则有同一套默认。如果三个 Agent 各自采用一套风格，技术上能合并，维护上仍是一件百衲衣。

共享 Skill 不能只有抽象美德。它应逐渐吸收项目中出现过的真实错误、反例和局部阈值。

## 二十一、审查强度必须与风险匹配

低风险原型和内部工具可以主要检查需求、测试计划和最终结果；线上关键路径、性能、安全、金融和多人共享模块需要更强审核：

- 逐项检查测试计划；
- 抽查架构决策；
- 阅读关键模块代码；
- 运行专门的性能、安全和可靠性验证。

减少逐行 review 不是禁止人在任何情况下看代码，而是把有限注意力从所有代码平均分配，转向真正承担风险的地方。

## 二十二、并行开发不会让人更轻松

并行 Agent 放大产出的同时，也放大单位时间内的判断需求。

手写代码包含操作节奏：写、运行、等待、修改，脑力与手上工作交替。调度多个 Agent 后，人持续处理高浓度决定：需求是否正确、候选选哪个、测试够不够、问题优先级如何、复盘意味着什么。

时间未必更长，但精神负荷更高。工作从“体力加脑力”变成“纯判断力”。

因此需要主动留白：

- 分批读取状态；
- 不把每个 Agent 输出当即时聊天；
- 保留不被打扰的方向设计时间；
- 限制同时活跃的决策数量；
- 不把“放手给 Agent”误解为“人不再工作”。

AI 节省的是手上的执行，空闲不会自动出现。人已经从键盘操作者变成协作系统的设计者与运行者。

## 二十三、完整的并行开发流程

可以把全文整理成以下操作链：

1. 人用自然语言提出目标；
2. Agent 暂停编码，扫描歧义和未知；
3. 对核心部分追问，对外围部分生成候选方案；
4. 人完成关键选择，形成需求文档；
5. Agent 生成单元、集成与 E2E 测试计划；
6. 新会话 Agent 或人独立审核测试计划；
7. Agent 按项目 Skill 做架构设计；
8. 通过 worktree、事务类型或跨项目方式并行执行；
9. Agent 运行测试、操作真实系统并自主修复；
10. Agent 按工程原则自审可维护性；
11. 人抽查高风险测试、架构和关键代码；
12. 结构化汇总全部产出，批量 triage；
13. 把新的反复错误和有效方法写回项目 Skill；
14. 下一轮减少同类人工介入。

流程的衡量指标不是启动了多少 Agent，而是每个 Agent 在完成一项高质量任务前需要打断人多少次。

## 二十四、最终结论

并行 AI 开发不是同时打开更多聊天窗口，而是重新设计人与 Agent 的责任边界。

需求对齐把模糊想法变成可执行契约；独立审核的测试计划把正确性从逐行代码审查转移到可验证行为；项目 Skill 把工程纪律变成所有 Agent 的共同默认；worktree 和事务拆分才在这些基础上提供真正并行。

人没有从循环中消失，而是从每个 Agent 的 pair programmer，变成多个 Agent 的技术负责人：确定问题、设定标准、裁决关键取舍、检查高风险证据，并持续改进整套协作系统。

这种模式同样可以迁移到研究、文档、设计、数据分析和其他可拆分的脑力劳动。代码开发只是较早跑通的应用场景。
