# 实现与修复判断

本文件用于代码变化存在真实风险、根因不在表面，或当前代码可能已经偏离既有设计和评审结论的情况。它帮助 Dev 取得足够反馈，不是一份固定步骤清单。

## 从准确现场开始

确认正在修改的真实仓库、工作树和代码状态，读取当前范围的项目规则，分清本轮目标与已有未提交变化。沿入口、调用链、数据或状态定义、接口、消费者和现有检查建立足够的当前模型；只看任务摘要、计划或 diff 都可能在错误位置实现正确文字。

开始编码所需的理解与任务风险相称。局部机械修改不需要预演整个系统；跨模块行为、持久化、权限、公共接口和迁移必须看到它们实际依赖的边界。

## 不同入口，共用一条反馈循环

实现功能时，从可观察结果和必须保持的契约出发，让代码结构决定局部做法。修复缺陷时，保留原始触发条件，沿输入、状态变化和错误结果找到错误从哪里产生，而不只处理最后出现的症状。落实评审问题时，把结论与当前代码重新对齐；代码漂移或新证据推翻原路径时，让结论回到评审或设计，不机械套用旧补丁。

这些不是三套流程。它们都要求：当前理解能够解释行为，修改落在拥有事实的位置，结果能够反过来检验理解。

## 让改动形状服从根因

先寻找项目已经拥有的能力和惯例，避免平行实现。不要为尚不存在的未来建立抽象，也不要为了少改几行而保留错误边界。必要重构只覆盖当前承诺真正依赖的部分；临时兼容、双读、双写、开关或迁移层需要拥有可执行的退出方式。

失败应该表达真实状态。宽泛捕获、吞错、无依据重试和静默降级如果只是隐藏错误，就不是可靠性设计。

## 用实现过程取得新证据

实现中出现计划外事实时，判断它是否改变了承诺。局部实现选择自行解决；施工顺序失效但结构仍成立时调整路线；公共行为、数据、权限、迁移、范围或责任变化时，把决定带回用户或 `design`；目标本身被推翻时回到 `shape`。阻塞只停止依赖它的工作。

重复尝试只有在引入新假设、新观察或新证据时才有价值。没有新增信息时，回到根因、环境或设计，不继续沿同一轨道消耗。

## 证明与变化相称

选择真正经过改变路径的证据：相关构建或静态检查、聚焦测试、原始缺陷重放、必要的数据或状态回读，以及最终 diff。记录决定结论的命令、结果和证据边界；环境缺失时说明因此无法证明什么。

实现者负责证明自己的改动确实发生且没有明显破坏近邻行为。完整风险覆盖、真实用户旅程、并发权限组合和最终发布资格仍交给 `test`、`review` 与用户判断。

交付时让下一位参与者能够直接看到：什么行为改变了，代码在哪里，实际检查了什么，哪里偏离了原地图，还有什么没有证明。其余格式由当前任务自然决定。
