# 提案路径

此参考仅用于模式 2：多路径提案。

下面的提案透镜并非新的 AgentCorp 角色。它们源自现有 AgentCorp 职责、阶段门和工件预期的需求塑形视角。不要将它们作为可分派的角色呈现。用它们来创建可供赞助方选择的不同需求路径。

## AgentCorp 提案透镜

选取 2-4 个适合当前模糊度的透镜。

- **最小变更透镜**——塑造满足赞助方意图的最小安全需求集合，同时保留现有行为。根植于 AgentCorp 的范围控制和变更规范纪律。
- **已验证旅程透镜**——优化目标用户/系统参与者、可观察旅程、成功指标和非目标。根植于 `validate-requirements`。
- **文档/上手透镜**——在投入产品/代码变更之前，先通过文档、示例、模板、可发现性或交接清晰度来实现赞助方的目标。仅当它确实是备选方案时使用，不要作为填充。
- **设计边界透镜**——仅在局部行为变更无法支撑时，才提出变更边界、契约、数据/状态流或架构。根植于解决方案架构师职责。
- **验证风险透镜**——偏好证明策略最清晰、隐藏回归风险最低的路径。根植于测试计划员、测试负责人和验收证据要求。
- **研究优先透镜**——当外部事实、已有先例、客户行为或代码库未知因素决定需求时，推迟承诺。根植于并行研究员和审查研究员的证据纪律。
- **交付范围透镜**——围绕 AgentCorp 本次任务能干净交付的内容塑造路径：阶段数、工件负担、审查门和剩余风险。根植于 Delivery Orchestrator 的所有权。

如果某个透镜对当前任务没有有意义的差异，就不要使用它。仅在措辞上不同的两条路径其实是一条路径。

至少将一条路径推到赞助方已声明偏好的边界之外——隐藏偏好只有在边界处才会浮现，一组全部安全地停留在请求内部的候选路径无法教会你真正的界限在哪里。明确标记它为过界选项，让赞助方知道它是故意的。

当未定形状是视觉或交互式时，将每条路径渲染为一次性的单文件原型（自包含 HTML，使用逼真的假数据——lorem ipsum 会掩盖布局问题），放在任务根目录下的 `brainstorm/prototypes/` 中，像所有 teamspace 工件一样排除在 git 之外，这样赞助方就能对真实的东西做出反应，而非对其的描述。原型是反应材料，从来不是产品代码。

## 每条路径的必备结构

每条提案路径必须包含：

- **名称**——简短、具体，不应当是角色名称，除非它命名了该透镜。
- **使用的透镜**——上述 AgentCorp 提案透镜之一。
- **变更内容**——可观察的用户/系统行为，而非实现机制。
- **目标参与者与旅程**——谁会经历这一变化，以及如何达到成功。
- **Must-haves**——如果该路径被选中，会出现在已验证需求中的要求。
- **MVP 边界 / 非目标**——该路径刻意排除的内容。
- **验收信号**——什么证据能证明这条路径有效。
- **主要风险或权衡**——代价、失去的好处、模糊度或验证风险。
- **何时选择它**——这条路径是正确赞助方决策的场景。

## 推荐规则

在展示路径后，推荐一条。仅在以下情况下允许混合：

- 混合仅使用已提出的路径中可见的片段
- 赞助方能理解什么被组合了、什么被舍弃了
- 混合仍有清晰的 MVP 边界和验收信号

在赞助方选择一条路径或明确授权混合之前，绝不要编写已验证需求——无论是单一路径还是多条。你的推荐路径不是选择；只有赞助方的选择才是。
