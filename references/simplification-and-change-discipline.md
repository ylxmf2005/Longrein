# 变更收缩与项目长期成本：来自 Simplify 和 AgentCorp Reviewer 的启发

> 来源范围：Claude Code 的本地 `/simplify` 命令，以及 AgentCorp 的 `simplicity-reviewer`、`change-hygiene-reviewer`、`taste-reviewer`、`project-steward-reviewer`。本文保存可复用的判断关系，供后续设计 Longrein 优化能力时参考；它不是当前 Skill 的规则副本。

## 一、共同对象不是“代码质量”，而是这次变化新制造的成本

这些能力看起来分别讨论复用、简化、效率、形态、diff 卫生和项目维护，实际都在追问同一个问题：为了当前承诺，这个变化新引入的结构、传播和长期责任是否划算。

审查对象应从真实 diff 开始，而不是从整个仓库寻找可以顺手改善的地方。否则优化会脱离当前授权，把“代码可以更好”误当成“这次交付应该修改”。

## 二、几个判断维度不能互相替代

### 复用

新增代码是否重新实现了仓库已有能力。可靠判断需要实际搜索相邻模块、共享工具和调用点，并指出可以复用的现有对象；“看起来可能重复”不构成结论。

### 简化

新增的层、状态、配置、分支、helper 或抽象是否真的替调用者隐藏了复杂度，是否有当前需求为它买单。移除之后所需行为和验收标准仍然成立，才说明它是可删除成本。

### 效率

变化是否引入重复计算、重复 I/O、不必要串行、启动或热路径阻塞，以及生命周期过长的对象和捕获环境。性能判断需要真实调用频率、路径和规模，不因理论可能制造优化。

### 正确层次与诚实形态

局部补丁是否只是绕开了错误的数据结构、边界、状态模型或缺失概念。更小不一定更诚实；当 workaround 会被多个调用者复制、进入持久 schema 或公共类型时，正确修复可能合理地更大。Taste 的作用是给这种更大形态定价，而不是用偏好否定朴素实现。

### 变更卫生

每个 hunk 是否能追溯到当前需求、设计、计划、评审发现、测试失败或工具约束。纯改名、格式扩散、无关消费者修改、旧分支残留和顺带契约变化都会扩大审查面。即使一项变化“更好”，如果回退后本次验收仍然成立，它通常也应拆出当前交付。

### 项目管护

变化是否增加了公共表面、兼容承诺、依赖、运维责任、无退出条件的临时层或长期债务。判断必须点明谁将承担什么成本，以及谁有权接受；说不出成本和所有者的担忧只是偏好。

## 三、最重要的张力：删除多余结构，但不能把必要形态削成 hack

Simplicity 与 Taste 不是同一个方向上的强弱关系：

- Simplicity 防止没有当前需求买单的结构进入项目；
- Taste 防止为了小 diff 而把错误模型固化进项目；
- Change Hygiene 防止两者借优化之名夹带无关变化；
- Project Stewardship 判断正确且有用的变化，项目是否仍然养得起。

因此不能用“越少越好”统一裁决。更可靠的问题是：留下当前形态的长期成本，与现在建立诚实形态的成本，哪一个更高；两边都要用真实消费者、契约、迁移和维护面定价。

## 四、作者侧收缩与独立裁决必须分开

实现者应该在工作过程中持续复用、收缩和检查传播面，因为这时删除多余结构最便宜。特别需要注意纯改名、公共契约传播、新文件、新接口、单用途抽象和无关消费者修改。

但作者侧检查不构成批准。实现者已经沿自己的模型工作，容易继续为现有形态寻找解释；独立 Review 仍需从当前 Task Context 和 Git baseline 冷读，用搜索、调用者、契约与移除测试证伪发现。

这意味着未来 Longrein 的专门优化能力可以负责发现并在授权范围内修复 changed code，但不能替代独立 Review，也不能把行为变化、公共契约调整或跨范围重构包装成清理。

## 五、可复用的证据动作

- 从 merge-base、上游分支或用户指定对象锁定真实 diff；需要时明确纳入未提交变化。
- 搜索已有 helper、模块惯例、符号调用者和消费者，验证“重复”“单用途”“无人使用”等否定式主张。
- 对可疑结构做移除测试：拿掉后，当前需求、契约和验收是否仍然成立。
- 对可疑 hunk 做依赖测试：回退后，本次交付是否仍然完整；若完整，默认拆分而非保留。
- 对更诚实但更大的形态核算触点：类型、schema、接口、调用者、迁移、测试和运维成本。
- 区分局部等价优化与会改变用户行为、公共契约、数据所有权或长期责任的变化；后者必须回到 Shape 或用户决定。

## 六、对 Longrein 的当前启发

- 根目录 `plan.md` 应显式标记高传播成本，让这些问题在便宜时被看见；Shape 建立结构，Dev 用实现现场持续补齐真实消费者和传播工作。
- Dev 在每个负责的 Phase 内持续检查复用、无必要传播和多余抽象，把 Phase 状态、结果和证据更新到共享 Plan，并在 `dev/report.md` 交付最终变化面与验证；这仍不是自我批准。
- Review 保持综合冷读，通过行为风险、Change Integrity 和架构维护等簇独立裁决，不必照搬 AgentCorp 的多 reviewer 组织图。
- 新的专门优化 Skill 等真实使用边界稳定后再设计；它应聚焦 changed code 的收缩与修复，明确与 Dev、Review、Shape 的权限关系。
