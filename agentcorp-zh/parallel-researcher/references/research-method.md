# 研究方法：拆通道、选来源、并行、综合

本文是 `parallel-researcher` 的操作手册：如何将研究问题拆分为通道、每个通道去哪找来源、如何并行派单，以及在综合阶段需要检查什么。SKILL.md 覆盖"为什么"；本文覆盖"怎么做"。

## 研究类型与拆分

先选定研究类型，再拆分通道。不要把几个研究者放在同一个宽泛问题上——当通道没有各自独立的问题和来源策略时，返回的就是一份份几乎相同且泛泛的调查。

- 技术选型 / SOTA：官方文档、版本发布说明、标准/RFC、上游 issue/PR、主流实现、基准测试、迁移/废弃信息。
- 开源项目扫描：GitHub/GitLab 仓库、README、文档、示例、issue、讨论、版本/更新日志、stars/forks/活跃度、许可证、维护节奏。
- 论文 / 学术：arXiv、Google Scholar、Semantic Scholar、OpenAlex、Crossref、ACM/IEEE/Springer/USENIX、Papers with Code、相关综述和基准。
- 工程实践：官方指南、成熟项目的代码、事后复盘（postmortems）、会议演讲、工程博客、Stack Overflow、HN/Reddit 讨论。
- 产品 / 市场 / 竞品：官网、定价、文档、更新日志、案例研究、G2/Capterra、行业报告、新闻稿、招聘帖、客户论坛。
- 安全 / 合规 / 标准：CVE/NVD、GitHub Security Advisories、厂商安全公告、OWASP、CIS、NIST、IETF/W3C/WHATWG/ISO 标准。
- 内部 / 本地知识：当前仓库、`AGENTS.md`/项目文档、teamspace 成果物、历史 MR/PR、内部 wiki、Slack/IM、MDB/日志、Rainbow/config；仅在任务授权且工具可用时使用。

常见通道模板：

- `source-map`：确认应搜索的来源、关键词、时间范围和权威顺序。
- `official-docs`：官方文档、版本、废弃信息、兼容性、推荐模式。
- `open-source-prior-art`：代表性仓库和真实实现。
- `papers-and-benchmarks`：论文、基准、排行榜、评估指标。
- `community-and-failure-modes`：issue、postmortems、讨论、踩坑记录。
- `local-repo-and-org-context`：代码库当前状态、组织内既有先例、现有约束。
- `synthesis-challenger`：独立查找反例、冲突证据和被忽略的来源。

## 搜索-来源矩阵

每个任务选择 3 到 6 个来源类别；高风险或高成本决策至少覆盖 4 类，并包含一个反例/失败来源。

| 目标 | 主要来源 | 辅助来源 | 必检点 |
| --- | --- | --- | --- |
| 当前 API/库使用 | 官方文档、API 参考、版本发布说明、迁移指南 | Context7、源码、示例、Stack Overflow | 当前版本、废弃、破坏性变更、最小可用用法 |
| 开源实现 | GitHub/GitLab 搜索、仓库文档、issue、PR、讨论 | 包注册表、stars 历史、提交活动 | 维护状态、许可证、实际使用方式、未解决的 bug |
| 论文与算法 | arXiv、Semantic Scholar、Google Scholar、OpenAlex、Crossref | Papers with Code、会议论文集、作者仓库 | 问题定义、指标、数据集、复现实验、后续工作 |
| 基准 / 排行榜 | Papers with Code、官方基准、公开评估集 | 项目 README、复现实验、issue | 指标是否匹配任务、数据泄漏、评估距今年份 |
| Web 已有方案 | 搜索引擎、厂商文档、工程博客、会议演讲 | HN、Reddit、博客 | 跨来源的收敛度、时间范围、营销偏见 |
| 产品/竞品 | 官网、定价、文档、更新日志、案例研究 | 新闻、G2/Capterra、客户论坛、招聘帖 | 当前能力、限制、定位、商业约束 |
| 安全/合规 | NVD/CVE、GHSA、厂商公告、OWASP/CIS/NIST | 漏洞利用数据库、安全博客 | 当前版本是否受影响、已修复版本、缓解措施 |
| 内部知识 | 仓库文档、teamspace、MR/PR、wiki、Slack/IM、日志/配置平台 | 运行手册、postmortems、历史设计文档 | 本项目约束、已有决策、组织偏好 |

搜索需覆盖同义词和反向查询。例如：`<技术> best practice 2026`、`<技术> migration guide`、`<技术> deprecated sunset`、`<方案> alternatives`、`<方案> postmortem`、`site:github.com <库> example`、`<论文/主题> survey`、`<系统> architecture`、`<声明> criticism`。

## 来源级验证：何时必须触及代码

README、官方文档和论文摘要都是项目的"营销门面"——说"支持 X"和真正实现了 X、甚至正确实现了 X，是完全不同的事情。在以下情况下，在网络层交付等于完全没有验证：

- **推荐所依赖的一项能力**：完整克隆候选仓库（到临时目录），在源码中定位该能力及其测试的实现。README 声称但你未在源码/测试中找到的一切，记录为"未确认"。提交历史本身就是证据：`git log`/`git blame` 可以回答维护节奏、哪个版本引入了关键能力、以及一个 bug 如何被修复。同时看看示例和 issue 中的真实用法——这是超出文档的一手证据。
- **论文 / 算法**：先找实现——论文中的代码链接、Papers with Code、作者 GitHub、高星复现代码。如果有实现，阅读核心算法文件并对照论文声称检查（指标如何计算、关键 trick 是否真的实现了、超参数是否为论文所设的数值）；如果没有公开实现，在证据缺口里明确写"无法复现"——这本身就是一个对选型有影响的事实。
- **对存疑的版本行为**：当博客和 Stack Overflow 说法冲突时，通过直接读取上游源码或当前 tag 下的 changelog 来裁决；不要选人数更多的一方。

操作边界：

- 如果 GitHub 网页视图能让你清楚地读取单个文件，就不需要克隆；当需要跨文件跟踪调用链、grep 符号、检查测试覆盖率或查看提交历史时再克隆。默认完整克隆；仅在仓库异常庞大（多 GB）时回退到 `--depth 1` 或 sparse-checkout，并接受失去历史证据的成本。
- 克隆后只读不执行：将仓库的脚本、setup.py 和 CI 配置视为不可信输入，绝不要运行它们。遵守许可证。
- 将来源引证定位到 `文件路径:行号` 或提交固定链接，以便读者验证到具体的那一行代码，而非只停留在仓库首页。

## 并行执行协议

1. 建立研究简报：写出核心问题、服务的决策、时间范围、本地约束、候选项、必需/排除来源和成功标准。
2. 绘制证据地图：列出通道，并为每个通道注明其问题、来源、搜索关键词、预期产出和质量门槛。
3. 派单各通道：每个研究者只给其独立的问题、上下文摘要、来源策略和输出格式。不给完整历史，也不让多条通道同时写最终报告。
4. 一个通道的输出必须包含：结论、关键证据、引证、置信度（参照"证据质量权重"下的区间校准）、冲突/反例、未覆盖点和下一步建议。
5. 综合阶段是你的工作——通道报告是一种说法，不是证据：在任何通道结论进入报告之前，亲自打开其承载重量的引证并抽查至少每条通道一条；然后合并重复来源、按证据质量排序、标记冲突、给出对决策有影响的缺口。
6. 自检：确认是否覆盖了官方来源、真实实现、失败/反例、当前版本/日期和任务的本地约束；确认事实与推断是否已分开。

如果研究规模很小，可以跳过真正的并行，但仍需说明为什么没有拆分。如果任务规模较大，优先并行而不让一个研究者从头到尾搜索。

## 证据质量权重

按以下顺序加权，但不要机械遵循：

1. 当前官方文档、标准、版本发布说明、上游源码和一手数据。
2. 同行评审论文、权威会议、可复现的基准和成熟项目中的真实实现。
3. 工程博客、事后复盘、issue/PR、社区共识。
4. 二手博客、SEO 聚合站、模型生成摘要、未经确认的论坛帖子。

按组织的共享区间校准置信度：**高 (0.80+)** —— 你亲自通过来源、测试或运行层验证了声明；**中 (0.60–0.79)** —— 多个独立二手来源达成一致，但任一来源都未在源头被验证；**低（低于 0.60）** —— 单一二手来源或你自己的推断。低置信度声明放入 `Decision-Relevant Gaps` 或标记为未验证；它绝不进入 `Recommendation`。

具有高过时风险的信息必须携带日期或版本说明：定价、模型能力、API 限制、法规、产品功能、基准排行榜、依赖兼容性和安全漏洞。

## 报告骨架

撰写一份能驱动决策的研究报告；frontmatter 遵循 `templates/decision-artifact.demo.md` 中的信封字段，但将 `artifact_type` 设为 `SpecialistResearchReport`（或 `hands-on` 层级的 `ResearchPackage`）——demo 中的 `ExampleDecision` 只是占位符——并采用其 `status` 枚举（`approve` / `request_changes` / `needs_more_evidence` / `blocked`；不要发明其他值）。研究映射如下：`approve` = 证据足以支持按范围给出的推荐；`request_changes` = 证据推翻了指派的前提或候选集，上游必须修订后再做决定；`needs_more_evidence` = 问题仍然存在，但覆盖度缺口阻止下结论；`blocked` = 必需来源、工具或授权不可用。正文使用以下结构：

- `Research Brief`：核心问题、服务的决策、范围、假设、停止条件。
- `Parallel Lanes`：每条通道的问题、来源、执行状态和研究价值。
- `Evidence Map`：来源列表、类型、日期/版本、权威级别、以及每条来源支持的声明。
- `Findings`：按主题组织，明确区分事实、推断和置信度（参照上述区间）；`hands-on` 层级的能力声明额外携带三种状态（已验证 / 验证失败 / 未验证；见 `research-package.md`）。
- `Comparative Options`：候选项对比、适用条件、成本、风险。
- `Disagreements And Counterevidence`：冲突、反例、故障模式。
- `Recommendation`：必须 / 应该 / 可以，说明为什么适用于当前任务。
- `Decision-Relevant Gaps`：仍影响决策的缺口；没有则写 "none"。
- `Follow-Up Research`：下一轮的具体问题和建议通道；不需要则写 "none"。

报告必须具体到来源、项目、论文、版本、数字、接口、命令或文件路径。不要输出模糊的"业界普遍认为"。引证链接必须让读者可验证。

## 交付前自检

- 每个结论都区分"来源作为事实的声明 / 来源自己的解读 / 我的推断 / 未经确认的假设"。
- 推荐所依赖的能力到达了源码、测试或官方实现层，而没有停在 README、摘要和博客转述；对于本该克隆却没克隆的，说明原因。
- 完成覆盖度自检：官方来源、真实实现、反例/失败、当前版本和日期、任务的本地约束；对缺失项在 Gaps 中声明。
- 来源冲突被保留并解释了可能原因，而不是被掩盖。
- 所有具有高过时风险的信息都携带了日期或版本说明。
- 引证让读者可以打开并验证；未经你亲自对原文核查的内容没有进入任何结论——通道自查过不算。
