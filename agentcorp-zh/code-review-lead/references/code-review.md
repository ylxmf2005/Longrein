# 本地代码评审参考

如何评级发现并确定决策——代码评审负责人所遵循的标准，而非布局格式。评审人员选择（五个始终启用的维度和风险触发的专家）位于 SKILL.md 的"你召集谁"部分：按变更实际风险添加专家，绝不默认全部启用。

## 发现评级

评级依据是"是否存在可操作的故障路径"，而非一个发现来自多少位评审人员。

- **必须修复**：可复现的行为 bug；安全或数据丢失风险；破坏契约的变更；违反明确需求；无法追溯到已批准来源制品的超范围语义/契约变更；管家发现中会将错误的长期承诺写入项目核心的问题；以及任何阻止有意义验证的评审阻塞项。
- **建议修复**：可维护性、可靠性、性能、覆盖率或长期维护风险且具有合理的故障模式；或一个合理的变更卫生发现，应当从本次 MR/PR 中拆分出来。
- **可选**：有用的清理工作，但不阻塞交付。
- **否决**：风格意见、重复项、与本次变更无关的既有问题、以及无可操作路径的猜测。

专家 `Confidence` 值是 0–1 的自校准值，各角色有特定的报告底线（安全从 0.60 起报告；大多数其他角色保留 0.60 以下的内容不报告）——将其视为分诊优先级，绝非证据：评级仍然取决于可操作的故障路径。

合并重复发现时，归入证据最强且 file:line 最精确的那一个。

## 决策

- `approve`：没有剩余的必须修复项；可以进入验证阶段。
- `request_changes`：仍有一个或多个必须修复项。
- `needs_more_evidence`：由于缺少 diff、需求、测试或设计上下文，评审无法完成——且一个具名证据请求可以解除阻塞。
- `blocked`：评审完全无法进行，且没有任何证据请求可以解除阻塞（例如，diff 或工作树不可用，或阶段本身被取消）。

高风险变更（安全/权限边界、公共/共享契约、数据丢失/不可逆发布）在决策前需获取一次跨模型族二次意见——完整规则在 SKILL.md 的"高风险二次意见"部分；结论仍归代码评审负责人所有。

在没有证据的情况下，绝不声称某位评审人员、某个命令或某个测试已经运行过。
