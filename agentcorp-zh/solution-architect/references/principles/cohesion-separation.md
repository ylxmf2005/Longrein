# 内聚与分离

## Description

这一维度要解决的是最基础的设计问题之一：两段功能应该放在一起，还是分开？这个决策适用于所有层级——函数、方法、类、模块和服务。目标是通过审慎的合与分的决策来降低整个系统的复杂度，提升信息隐藏能力、减少依赖，并打造更深的接口。

核心关注点包括：
- **合与分的决策**：相关功能应该合并还是拆分
- **代码重复**：重复的代码模式往往意味着缺失抽象
- **通用与专用混用**：通用机制被专用逻辑污染
- **连体方法**：代码片段无法独立理解

## Core Principles

### 拆分本身也会带来复杂度

把系统拆成更小的块并不总是更好。拆分是有代价的：
- 需要跟踪和管理的组件变多
- 组件之间的接口增加
- 跨组件可能出现代码重复
- 物理上的分离让依赖关系更难看清
- 协调多个部分带来管理开销

这些代价只有在拆分真正降低了复杂度、实现了更好的信息隐藏和更清晰的抽象时，才是值得的。

### 相关就放在一起

如果代码片段真正相关，就应该放在一起：
- **共享信息**：两者依赖相同的知识（例如文档格式、协议结构）
- **双向使用**：用到其中一个的，几乎总会用到另一个
- **概念重叠**：它们天然属于同一个更高层级的范畴
- **理解依赖**：不理解其中一个，就没法理解另一个

### 合并以简化接口

合并模块可以消除中间接口，并让横切关注点的自动处理成为可能。当部分解决方案被合并后，合并后的接口往往比原来各个接口的总和更简单，而且某些功能可以透明地处理。

**Example**：把文件 I/O 和缓冲合并在一起，用户就不需要理解或配置缓冲——默认就能用。

### 合并以消除重复

重复的代码模式说明缺少抽象。当相同逻辑出现在多个地方时，考虑：
- 如果模式足够大且接口简单，就提取成共享方法
- 重构代码，让逻辑只在一处执行
- 警惕浅层提取（一行代码的方法配上复杂的签名），这种提取增加的复杂度可能比消除的还多

### 把通用代码和专用代码分开

通用机制不应该包含针对特定场景的专用逻辑。混用会导致：
- 机制更复杂、更难理解
- 机制与具体场景之间的信息泄漏
- 脆弱性：修改一个场景就要动通用机制

**Solution**：把专用代码往上层抽，保持下层通用。上层通常处理应用专属逻辑，下层提供可复用的基础设施。

### 拆分方法是为了更干净的抽象，而不是为了长度

方法长度本身不是拆分的好理由。接口简单的长方法可以是 deep 的、好的。只有在拆分能带来更干净的抽象时才拆分：

1. **提取子任务**（父子拆分）：子方法实现一个可以清晰分离的、相对通用的子任务。读者应该能在不了解父方法的情况下理解子方法，反之亦然。

2. **拆分复杂功能**（兄弟拆分）：原方法试图做多件不相关的事，导致接口过于复杂。每个结果方法应该有更简单的接口，而且大多数调用者只需要其中一个。

避免拆出 shallow 方法，或者迫使调用者协调多个方法调用并传递中间状态。

### 合并方法以降低复杂度

以下情况考虑合并方法：
- 用 deeper 的方法替代 shallow 的方法
- 消除代码重复
- 移除依赖或中间数据结构
- 通过整合知识来提升封装性
- 简化调用者接口

## Red Flags

在你正在提出的设计里、以及你为设计而阅读的代码里，留意以下模式：

### Repetition
- 相同（或几乎相同）的代码多次出现
- 复制粘贴后只改了少许的逻辑
- 反复出现的模式，暗示缺少抽象
- Exception：不要提取带有复杂上下文依赖的一行代码模式

### Special-General Mixture
- 通用类或方法里包含特定场景的逻辑
- 机制“知道”自己的使用者是谁
- 信息泄漏：修改使用场景就要修改通用基础设施
- 库级别的模块里出现应用专属代码

### Artificial Separation
- 只在一个地方被调用的 shallow 方法
- 功能跨类拆分，但两者必须一起理解
- 中间接口只是透传数据，没有任何转换
- 代码分离迫使读者在多个位置之间来回跳转

### Conjoined Methods
- 无法独立理解的方法
- 物理上分离，但逻辑上不可分割的代码
- 读者必须同时看两部分才能理解其中任何一部分
- 父子方法中，子方法需要了解父方法的上下文

### Inappropriate Grouping
- 不相关的功能被硬塞进一个类
- 缺乏统一主题或职责的模块
- 类在没有正当理由的情况下混合多个概念范畴

### Over-Subdivision
- 很多小类，每个只做很少的事
- shallow 组件的爆炸式增长
- 管理的复杂度来自大量接口，而非真实的领域复杂度
- 代码被按照僵化的行数规则拆分，而非基于抽象质量

### Missing Consolidation Opportunities
- 多个类的职责重叠
- 可以统一但被重复的接口
- 功能散落在共享相同知识的多个模块中

## Issue Tags

在设计产物中标记合并/拆分问题时使用以下 tag：

- **[Repetition]**：相同或相似的代码被重复，但没有抽象
- **[Special-General Mixture]**：通用机制里混入了专用代码
- **[Conjoined Methods]**：无法独立理解的方法
- **[Artificial Separation]**：不必要的拆分，产生了 shallow 组件
- **[Over-Subdivision]**：过度拆分导致接口泛滥
- **[Missing Consolidation]**：相关功能散落在多个模块中
- **[Poor Method Split]**：方法拆分没有提升抽象质量
