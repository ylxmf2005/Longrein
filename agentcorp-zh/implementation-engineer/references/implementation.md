# 本地实现参考

SKILL.md 承载角色规则；本文承载工作纪律——如何走通故事规格，保持 diff 最小的关卡，以及交付代码审查前必须通过的门禁。

## 执行故事规格

先通读整个故事规格，消化其 Story、验收标准、任务/子任务、实现约束、验证预期、审查重点和状态，然后加载它引用的代码和项目上下文。再按顺序处理任务/子任务——除非故事规格或审查者明确允许重新排序。

对每个任务做出"最小且正确"的变更：尽量贴近已批准的故事，不要顺手塞入相邻改进；保持现有模块边界不变，除非已批准成果物要求改动。尽量让每个文件只承载一个连贯的变更，在再次改动前先重读一遍。当行为、约定、bug、数据、认证或公开接口发生变更时，添加或更新聚焦的测试。在 `implementation/implementation-result.md` 中边做边记录进度、变更文件、命令、偏差和阻塞项——边写边干，不要在最后再凭记忆写。

## Diff 最小化关卡

更少的变更胜过更多的变更。在添加任何内容之前，用以下问题把关，防止 diff 膨胀：

- **复用优先于新建。** 在添加函数、文件或抽象之前，先在仓库中搜索是否已有东西能做这件事（grep 关键符号、相似名称、可比工具类）——复用它，而不是在旁边搭一个平行拷贝。
- **不为想象中的未来做泛化。** 不要留开关/选项/插件点，也不要为"以后可能用得上"写通用结构；当前只有一个用例时，就为这一个用例写。
- **不提早提取。** 单调用方的逻辑保持内联；不要把它提取成共享函数，除非已批准的故事规格显式要求该接口。
- **不动无关代码。** 不动任务没要求你改的现有代码——保持偶然的改名、重构和格式化远离本次变更。
- **每次新增都可追溯到规格。** 每个新增的能力、文件或分支都应能追溯到故事规格或验收标准；任何无法追溯的都在范围之外，不要做。

## 交付前门禁

在交接给代码审查之前，确认以下每一项——列表末尾的部分正是代码审查负责人（Code Review Lead）和 review-researcher 验证变更所需要的东西，而且这也是最常跳过的部分：

- 每个必需的任务/子任务已完成，或明确列为阻塞。
- 代码能编译，且相关静态检查（如有）通过。
- 聚焦测试通过，或失败被诚实记录为阻塞项。
- 结果行为已对照验收标准、测试计划或诊断标准进行了检查。
- 实现结果列出了所有新增、修改和删除的文件，以及所有与已批准故事规格的偏差。
- 代码审查负责人能够访问变更文件、命令、测试证据和已知风险——每一项都是可核查的句柄（命令 + 输出、file:line），而不是一个状态词。
