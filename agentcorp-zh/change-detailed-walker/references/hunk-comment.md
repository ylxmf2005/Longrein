# 评论撰写规则

本文档定义了如何在 forge PR 中撰写每条评论。评论质量与可追溯性在此规范；评论如何发布、行如何锚定、覆盖率如何保障，详见 `references/pipeline.md`。

你以**上帝视角**撰写：你已经读过 diff 和周围代码，该查的都查完了。评论是**递给同事的一个结论**——不是你的排查过程，也不是给对方的待办清单。改动已经交付；你的职责是清晰、果断地解释**这是什么**以及**为什么这么做**，就像一位已经看过全部代码的 senior 同事直接告诉你那样。这不是风险审计。

## 单条评论的字段

传给 `post_review.py` 的每条评论格式为 `{path, body, new_position 或 old_position}`：

- `path`：相对于仓库的路径（新增/修改用新路径，纯删除用旧路径）。
- `body`：markdown——结论本身。Body 使用 requester 的工作语言；AgentCorp 的默认是 zh-CN。
- `new_position` / `old_position`：从 `diff_outline.py` 实际值中取的锚定行号。新增/上下文行锚在 `new`，纯删除锚在 `old`。二选一。

## 每条评论必须说清楚的两件事

始终以结论形式陈述（不是疑问，不是待验证项）：

1. **这是什么 / 做了什么**——用大白话：新增或改了哪个函数/逻辑，它负责什么，大致流程如何。即使代码注释里已经写了，也要用更直白的话重述一遍。缺了这一层，评论就不合格。
2. **为什么这么做**——意图：这次改动或重构想达成什么。如果它是某个模式的一部分，就点出来（\"跟上面两个 endpoint 一样，都是把逻辑下推到 service 里\"）。

然后，**只有当确实存在真实问题时**，再追加**一句结论**——真正的 bug、残留代码、顺手重构、越界改动。把它当作你已经发现的事实陈述，不是让你去调查。没有问题 → 完全不写风险内容。

这就是整条评论的全部。两层 + 必要时一句结论。保持紧凑。

## 写结论——不写你的过程，不把活丢给读者

这是最重要、也最容易被违反的规则。

- **陈述结论，不是你的排查过程。** ✗ \"我已对照 remix_service.py:592-617，四项都搬过去了\"——这是你在展示工作过程。✓ \"四项逻辑在 service 里完整保留，行为等价。\" 读者不想看你验证，他们要的是 verdict。
- **不要把活推给读者。** ✗ \"（必看）\"、\"逐项核对 service 是否完整复刻了旧逻辑，否则静默丢功能\"、\"维护者要确认 X\"。你拥有上帝视角——*你*来做确认，然后写下结果。如果确实有东西没搬过去，直接作为事实陈述：\"X 没搬过去，这里会丢 Y 功能。\"
- **不要堆砌风险或对冲。** 已交付的改动不需要风险清单。如果有一个真问题，用一句话点明——不要\"可能\"、不要\"否则\"、不要\"要注意\"。如果没有，就什么都不说；不要写\"此处无新增风险\"。
- **不要 meta。** 不要引用早期版本、你自己的迭代过程或 review 流程。评论要独立成文。

跨文件的事实可以写，而且很有用——但要把它写成**你已经完成的检查的结论**（\"四项已在 service 完整复刻，行为等价\"），而不是让读者去验证的契约，更不是对冲（\"若 X 改成抛错则全失效\"）。

## 粒度：锚定在语义单元上，不要按行铺排

- 一个函数 / 一个分支 / 一组有意义的常量写一条评论；逻辑密集时，细到单行也可以。
- **大 block 必须拆分**：hunk 超过约 20 行改动时，按函数拆成多条评论，每条锚定到该函数内的真实改动行。
- **不要为了凑密度而把文件切成连续的等长线段。** gate 只要求密度下限，不是让你\"把文件切成连续 block，每个 block 一条评论\"。如果你的评论按排序后把整文件无缝铺满，那就是在铺 block——重新锚定到真实语义单元上。宁可一个函数写多条互补的评论，也不要把一条评论盖到无关范围上。
- 纯机械性改动（格式化、import 重排、批量重命名）：一条简短评论说明是什么机械改动。如果本身是个大 block，密度也要够。

## 分类（决定你怎么写；不是贴在 body 里的标签）

每条改动心中要有分类。**不要在 body 前加 \"normal\" 标签**（`core`/`supporting`/`mechanical`）——读者要的是解释，不是标签。只有三类\"需要谨慎\"的才需要在 body 开头加醒目标记，作为一行结论：

| Classification | 判定标准 | Body marker |
| --- | --- | --- |
| `core` | 直接承载主能力或修复 | 无 |
| `supporting` | 必要的支撑（call-site 透传、辅助查询、fixture） | 无 |
| `mechanical` | 机械性改动，行为意图弱或无 | 无 |
| `suspect-refactor` | 看起来像顺手重构，授权不清 | `【可疑·顺手重构】` |
| `suspect-residue` | 疑似早期方案残留，或越界语义 | `【可疑·残留】` |
| `untraceable` | 找不到可信的授权/意图证据 | `【追溯不到】` |

- `suspect-*` 标记是结论：用一行话陈述可疑事实（\"这是顺手重构，和本次需求无关\"），不要展开风险讨论。
- `untraceable`：不要编造来源；说明你查了哪些证据（需求、设计、commit、周边代码）但追溯不到。
- 抓出残留和越界改动是这个工具的核心目的之一——但它是 verdict，不是对冲。**几十个文件的改动中零 suspect，通常意味着你没认真看**，而不是它真的干净。

## 可追溯性

轻量处理：当它有助于解释某处为何存在时，指向授权来源（需求/设计条目，或 commit）。没有设计文档的分支，用 `<repo alias> <short sha> <subject keywords>`。当一个大 commit 覆盖多处时，不要把同一个分支标签到处贴——要么给这处点明子意图，要么用\"配合 `<file>:<symbol>`\"串起因果链。不要把 trace 变成 clutter。

## 反面示例

### 反面示例 1（最常见）：排查过程 + 风险审计 + 把活推给读者

> 核心：update_remix_cache 是改动最大的一处。原 handler 里塞了管理员校验、按 compilation_content_id/file_id 分流、文件存在性校验、asyncio.create_task 起异步日志，现在全删掉压成一次 await remix_service.update_cache_for_user(...)。
> 风险 / 行为等价性（必看）：这次搬迁要逐项核对 service 侧是否完整复刻了旧逻辑，否则就是静默丢功能：①②③④……我已对照 remix_service.py:592-617，四项都搬过去了。但有个隐患……维护者要知道这条日志旁路不可靠……

问题在哪：\"（必看）/逐项核对/否则静默丢功能/维护者要知道\" 把验证推给读者；\"我已对照……四项都搬过去了\" 是 walker 在展示工作过程；整段是对已交付改动的对冲式风险审计。

目标写法（上帝视角结论）：

> `update_remix_cache`：原来在 handler 里直接做的四件事——管理员鉴权、按 compilation_content_id / file_id 分流、文件存在性校验、异步操作日志——整体下沉进 `remix_service.update_cache_for_user`，handler 只剩一行转发。和上面两个删除接口是同一套「逻辑收进 service」重构，四项在 service 里完整保留、行为等价；那几行 `logger.info` 只是单行字符串拆成多行拼接，无行为变化。一处遗留：异步日志仍是裸 `asyncio.create_task`，出错会被静默吞、请求结束可能被取消——旧代码就这样，本次没动。

为什么有效：what + why 作为结论；\"行为等价\" 是你得出的 verdict（不展示检查过程）；唯一真实的遗留用一句话点明，不是\"必须检查\"的审计。

### 反面示例 2：逐行复述（按代码顺序念稿）

> 先校验源归档就绪，再生成不冲突 id（最多重试 5 次），再复制归档（失败先删目标再抛），然后在 try 块里创建任务、创建记录、写引用、创建提交、初始化对象，任一步异常走整组回收并 raise……

问题在哪：按代码顺序念了一遍；读者仍然不知道**为什么**重试 5 次，**为什么**先删再抛。

目标写法：

> 复制失败后先删目标对象再抛，是为了不让半拷贝留下的孤儿对象在下次重试里被当成已就绪。

### 反面示例 3：只夸，或过度对冲

> （只夸）补偿写在 clone 旁边，登记失败就回收私有副本避免孤儿，和 task_file 分支对称，设计很好。
> （过度对冲）……接手前要确认它幂等、失败可吞；若 create_asset 改成撞键抛错，这里的回收逻辑全失效……

目标写法（一个真问题一句结论）：

> 补偿紧贴 clone：登记失败时回收本次铸的私有副本——只有这层握着 `clone_result`，外层不知道铸了什么。一个真问题：`cleanup_cloned_forge_project` 失败时会用清理异常盖掉原始的 create 异常，上层看到的是「清理失败」而非真正的登记失败。

最终自检，通读全文：我会不会用这些话，当着接手代码的同事的面说？如果读起来像我的备忘笔记、风险清单或留给对方的作业——重写成一句大白话结论。
