# Information Hiding

## Description

信息隐藏是将设计决策封装在模块内部的一种技术，使这些决策不会暴露在模块的接口中，对其他模块不可见。每个模块都应该隐藏与实现机制相关的知识，包括数据结构、算法和设计假设。与之相对的是**信息泄露**，即相同的设计决策出现在多个模块中，形成需要同步修改的依赖关系。

良好的信息隐藏可以简化接口，让系统更容易演进。当信息被隐藏时，对该信息的变更只会影响到包含它的那个模块。

## Core Principles

### Encapsulation of Design Decisions

- 每个模块都应该封装特定的知识片段，这些知识代表了设计决策（例如如何解析 JSON、如何在 B-tree 中存储数据、如何实现 TCP）。
- 被隐藏的信息包括数据结构、算法、底层细节（如页面大小）以及高层假设（如"大多数文件都很小"）。
- 目标是让信息完全隐藏，使其对模块使用者来说无关紧要、不可见。部分隐藏（仅少数使用者需要的信息，通过独立方法访问）也有其价值。

### Private vs Information Hiding

- 将变量和方法声明为 private 并不等同于信息隐藏。private 有帮助，但不能保证信息隐藏。
- public 的 getter 和 setter 暴露私有变量信息的程度，可能和直接把这些变量设为 public 差不多。
- 真正的信息隐藏意味着内部数据的性质和使用方式根本不会通过接口暴露出来。

### Information Leakage

- 信息泄露发生在某个设计决策出现在多个模块中时。一旦该决策发生变更，所有涉及的模块都需要修改。
- 泄露可以是显式的（通过接口）或后门的（两个模块都了解某种文件格式，即使它不在它们的接口中）。后门泄露的危害更大，因为它不容易被发现。
- 如果你在类之间发现了泄露，请重新组织，使相关知识只影响单个类。可选方案包括合并类，或将共享知识提取到一个具有简单、抽象接口的新类中。

### Avoiding Temporal Decomposition

- **时序分解**按照操作发生的时间顺序来组织系统结构（例如用单独的类分别负责文件的读取、修改和写入）。
- 这会导致信息泄露，因为设计决策（如文件格式）在执行流程的多个节点上都会体现。
- 关注完成每个任务所需的知识，而不是操作的顺序。即使某些知识在不同时间点被使用，也要把相关知识整合到单个模块中。

### Interface Design for Hiding

- 接口越简单，信息隐藏通常就越好。复杂的接口会暴露更多信息，产生更多依赖。
- 不要暴露内部数据结构（例如返回用于存储参数的内部 Map）。内部表示的变更不应该影响到接口。
- 提供更深层次的方法来隐藏内部表示，减轻调用方的负担（例如提供 `getParameter(String name)`，而不是返回内部 Map 的 `getParams()`）。

### Defaults and "Do the Right Thing"

- 为常见场景提供合理的默认值。默认值是一种部分信息隐藏的形式——在正常使用中，调用方无需了解它们。
- 类应该自动"做正确的事"，而无需被显式要求。例如，文件 I/O 的缓冲应该是自动的，而不是需要手动开启。
- 让常见场景尽可能简单。罕见场景才需要提供特殊方法。

### Making Classes Larger for Better Hiding

- 通过让类稍微变大一些，将与特定能力相关的代码集中在一起，往往可以改善信息隐藏。
- 将相关功能组合起来，可以提升接口的抽象层次（例如用一个方法完成整个计算，而不是用三个方法分别完成三个步骤）。
- 这样可以得到接口更简单的深层模块。

## Red Flags

### Information Leakage Indicators

- **知识重复**：多个类理解相同的文件格式、协议或数据结构
- **后门依赖**：类依赖相同的设计决策，即使该决策不在它们的接口中
- **联动修改**：修改一个设计决策需要同时改动多个模块

### Temporal Decomposition Indicators

- **按操作顺序组织**：不同的执行阶段（读取、处理、写入）被拆到不同的类中
- **共享格式知识**：reader 和 writer 类都需要理解相同的格式
- **基于阶段的命名**：类名如 `RequestReader` 和 `RequestParser`，反映了执行步骤的先后顺序

### Shallow Interface Indicators

- **暴露内部结构**：方法返回用于存储状态的内部数据结构（Map、List）
- **暴露内部结构的 getter**：返回内部参数 map 的 `getParams()`，而不是 `getParameter(name)`
- **表示泄露**：内部表示变更时，接口也随之变更
- **增加调用方负担**：调用方在获取数据后还要进行额外的查找或转换

### Missing Defaults

- **强制指定**：调用方必须提供本可自动确定的值（例如 HTTP 响应版本应该与请求版本匹配）
- **罕见选项变必填**：即使在常见场景中，也必须指定很少用到的配置项
- **过度暴露**：使用常见功能时，却迫使用户去了解很少使用的功能

### Over-Decomposition

另请参见 `cohesion-separation.md`，了解完整的"在一起还是分开"的红旗标志和标签。

- **一个阶段一个类**：多步流程（如读取和解析）的每一步都在单独的类里
- **重复解析**：相似的解析或校验逻辑出现在多个地方
- **复杂使用模式**：调用方必须按特定顺序调用多个方法才能完成一个任务

## Issue Tags

在设计产物中标记信息隐藏问题时使用以下 tag：

- **[Information Leakage]** - 相同的知识或设计决策出现在多个模块中
- **[Temporal Decomposition]** - 系统结构反映的是执行顺序，而非知识边界
- **[Exposed Internals]** - 内部数据结构或表示在接口中可见
- **[Shallow API]** - 接口暴露了过多细节，或提供的抽象不足
- **[Missing Defaults]** - 常见场景需要显式指定本可自动确定的值
