# Claude Fable 5 官方最佳实践：把困难交给模型，把边界留给人

> 主要来源：[Anthropic 官方《Prompting Claude Fable 5》](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5)与[通用 Prompting Best Practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)。本文按实践主题重新组织，不是逐字翻译。
>
> 时效：2026-07-16。Fable 5 的可用范围、API 参数、安全分类器和回退模型可能继续变化；使用前应复查官方页面。

## 一、先改变任务选择，而不是先改提示词

Fable 5 最重要的升级不是“同一个任务答得稍好”，而是可承担的任务跨度发生变化。Anthropic 建议从能力上限附近开始测试：把过去需要数小时、数天甚至数周的人类工作，作为端到端目标交给模型，让它自己调查、拆解、执行和验证。

如果只拿简单工作测试 Fable，很容易低估它。它的优势主要出现在：

- 长时间保持目标和指令；
- 一次完成复杂且定义清楚的系统；
- 阅读密集截图、技术图像和 Web 页面；
- 处理财务、表格、幻灯片和文档等企业工作；
- 跨代码库和版本历史查错；
- 在多线程、存在歧义的任务中判断下一步；
- 调度并持续管理并行子代理。

一个好的首轮实验，不是“改一个变量名”，而是选择过去会分成几天处理、存在真实验收标准的任务，让 Fable 先界定范围、提出关键问题，再端到端执行。

## 二、Fable 的回合更长，运行系统也要跟着改变

在高 effort 下，一次困难请求可能运行很多分钟；自主任务可能持续数小时。迁移到 Fable 时，先检查运行它的系统：

- 客户端超时是否足够长；
- 是否使用流式输出；
- 用户能否看到可信的进度；
- 长任务能否异步运行，而不是一直阻塞连接；
- 是否支持定时检查或后台继续；
- 中间状态和最终证据能否保存。

长回合不等于让模型无限规划。歧义已经消除、信息足以行动时，应要求它开始行动，不要重新推导已确认事实，不要反复争论用户已经作出的决定，也不要在用户可见的消息里枚举不会采用的方案。真正存在取舍时，给出推荐，而不是无边界调查。

## 三、effort 是能力、延迟和成本的主旋钮

官方建议大多数任务从 `high` 开始；最依赖能力的任务使用 `xhigh`；日常任务可用 `medium` 或 `low`。低 effort 的 Fable 仍可能超过旧模型在高 effort 下的表现，因此不要默认所有任务都开到最高。

选择方式：

- 任务做对了但明显太慢：降低 effort；
- 需要快速交互或反复来回：用 medium/low；
- 任务包含长程调查、多步骤推理、严谨验证：用 high；
- 结果价值很高，少量质量提升值得额外时间：用 xhigh。

高 effort 的副作用是可能调查过多、顺手整理周边代码、增加抽象或为假想未来设计。对此不需要写几十条禁令，一段简短边界就够：只做任务要求的改变；缺陷修复不附带周边重构；一次性操作不创建不必要的 helper；不为假设需求设计；只在用户输入、外部 API 等系统边界做防御性验证。

## 四、指令跟随更强，所以提示词应该更短

Fable 对短指令的响应已经足够强。旧模型时代常见的长清单，可能在新模型上造成过度约束或冗余。

例如，与其逐项禁止“背景铺垫、长篇根因、复杂 PR 模板、逐行注释、过多标题”，不如说明输出原则：

- 先说结果；
- 第一段回答用户最想知道的“发生了什么”；
- 只保留会改变下一步行动的细节；
- 可读性优先于机械压缩；
- 不用碎片化短语、箭头链和内部黑话冒充简洁。

同样，暂停条件也不用穷举。只在以下情况停下来找用户：不可逆或破坏性动作；真正的范围变化；只有用户能提供的信息。其余可逆、在原请求范围内的动作应继续推进。

## 五、明确授权边界，防止“能力强”变成“做得太多”

Fable 偶尔会执行未被要求的动作，例如用户只在讨论问题，它却直接修改代码；用户只让分析，它却起草邮件；为了安全感自行创建备份分支。

因此需要把请求类型与授权范围分开：

- 用户在提问、描述问题或思考：产物是评估与发现，报告后停止；
- 用户明确要求修改、构建或修复：执行范围内的可逆本地更改并验证；
- 重启、删除、外部发送、配置变更等会改变系统状态的动作：必须先确认事实真的支持该动作；
- 看到一个像已知故障的信号时，先诊断，不要直接套用熟悉的修复。

高能力模型最需要的不是更多实现步骤，而是清楚的权限与结果边界。

## 六、长任务中的进度必须来自工具证据

长时间自主运行最危险的错误之一，是模型根据计划或记忆报告“已经完成”，实际工具结果并不支持。

官方建议在系统或任务指令里要求：每次汇报进度前，把每个声明对应到本次会话中的工具结果。没有验证的内容明确标记为未验证；测试失败就报告失败输出；跳过步骤就说明跳过；完成且验证过的内容直接确认，不用含糊措辞。

这条规则把进度消息从叙事改成证据索引：

- “已经修复”必须有修改记录和相关测试；
- “部署成功”必须有部署结果和健康检查；
- “未发现问题”必须说明检查了哪些范围；
- “任务完成”必须逐项对照验收标准。

## 七、并行子代理：独立工作并行，语义判断集中

Fable 更愿意主动派发子代理，也更能维持长时间协作。适合委派的任务是相互独立、边界明确、可以并行产生证据的工作，例如：

- 分别调查官方文档、代码实现和失败案例；
- 对不同组件运行独立测试；
- 让独立审阅者检查安全、正确性和性能；
- 把大规模样本分片处理。

主代理不应派完就阻塞等待。它应继续做自己的综合工作，通过异步消息纠偏。长期存在并保留上下文的子代理可以复用缓存与历史，不必每次重新解释背景。

但最终判断仍应集中：子代理报告是一项声明，不是自动成立的事实。主代理需要抽查承重证据，处理冲突，决定哪些结论能进入最终产物。

## 八、建立可修订的记忆系统

Fable 特别适合从过去运行中记录和复用经验。最简单的记忆系统可以只是一组 Markdown 文件，但必须防止它变成垃圾堆。

官方建议的原则包括：

- 每个文件只记录一项经验，顶部有一句摘要；
- 同时记录纠正和已验证的有效方法，并解释为什么重要；
- 仓库、提交或聊天里已经存在的事实不要重复保存；
- 新证据与旧经验重叠时更新原文件，不创建副本；
- 被证明错误或过期的经验要删除；
- 新任务开始时明确告诉模型去哪里读取这些经验。

还可以让 Fable 回顾历史会话，用子代理归纳反复出现的主题与教训，再由主代理去重、核验后写入稳定位置。

记忆的目标不是“记住更多”，而是让未来行为因过去证据发生可检查的改变。

## 九、处理两种少见但重要的停止问题

### 1. 只说“接下来会做”，却没有调用工具

长会话深处，Fable 偶尔会用一句计划结束回合，例如“我现在运行测试”，但并没有真的运行。自动管线可以加入回合结束检查：如果最后一段是计划、问题、承诺或下一步清单，就立即执行相应工具；只有完成或确实缺少用户独有输入时才能结束。

### 2. 因上下文预算而过早建议新会话

如果 harness 把剩余 token 倒计时直接展示给模型，模型可能提前总结、交接或建议开新会话。能不暴露精确预算就不暴露；必须展示时，应明确说明上下文仍充足，不要仅因预算数字停止工作。

## 十、给理由，而不只是给命令

Fable 在理解任务意图时表现更好。对长程、跨工作流任务，提示词应包含：更大的工作是什么；产物给谁使用；产物将帮助对方完成什么；然后再给具体请求。

理由不是礼貌性背景，而是决策信息。它帮助模型在遇到未规定的细节时，选择与最终价值一致的路线。

例如，“生成一个表格”信息不足；“这是给财务负责人在预算会上决定下季度削减项目用的，突出不可逆承诺、现金流影响和数据缺口”会改变模型的字段、排序和验证方式。

## 十一、最终总结必须重新把用户带回现场

模型经过几十次工具调用后，掌握了一套用户没看见的内部词汇和工作线索。最终消息不能像内部工作笔记的续集。

官方建议：

- 第一段重新说明结果；
- 接着说明用户需要知道或决定的一两件事；
- 使用完整句子；
- 内部简称、临时标签和复杂术语要重新介绍，否则删除；
- 文件、提交、开关和标识符分别用通俗语言说明；
- 在“更短”和“更清楚”冲突时，选择清楚。

如果异步 Agent 需要在不中止任务的情况下，把一段内容原样展示给用户，可以提供专门的 `send_to_user` 工具。它只传递必须让用户直接阅读的交付物、回答或量化进度，不用来传内部推理和日常旁白。仅定义工具还不够，需要在系统指令中明确何时调用。

## 十二、迁移到 Fable 时，应主动删除旧脚手架

为旧模型编写的 skills、system prompts 和 agent harness 往往过度规定。Fable 默认能力增强后，这些脚手架可能反而降低质量。

迁移时逐项审计：

1. 先运行一个简单、干净的基线提示；
2. 对比现有复杂 skill 的结果；
3. 每次删除一组旧指令、示例或工具；
4. 用相同评测验证质量、成本和延迟；
5. 只保留能修复已测量缺陷或表达真实产品要求的部分。

特别要删除要求模型复述、展示或逐字解释内部推理的指令。Fable 的安全机制可能把这类请求视为 reasoning extraction。若应用需要过程透明，应读取结构化 thinking 输出或展示工具证据、进度和决策摘要，而不是要求模型公开内部思维链。

## 十三、自我验证应成为运行结构，而不是结尾口号

在长任务里明确规定验证节奏，例如每完成一个阶段，就让独立、全新上下文的 verifier 对照规格检查。官方观察是，新上下文验证者往往比同一代理的自我批评更可靠。

验证计划应该在实现前确定：

- 验收标准是什么；
- 哪些阶段需要检查；
- 使用测试、截图、日志、diff 还是人工规则；
- 验证者能看到哪些证据；
- 失败后是修复、回滚还是请求用户决定。

“最后检查一下”不是验证系统。验证必须产生可引用的证据，并能阻止错误交付。

## 十四、可直接复用的 Fable 工作法

1. 选择过去不敢一次性交给模型的真实难题。
2. 用目标、意图、边界和验收标准描述任务，少写实现步骤。
3. 从合适的 effort 开始，以评测结果升降，不迷信最高档。
4. 允许长回合，但让系统支持超时、流式进度和异步继续。
5. 信息足够时行动，不重新讨论已确认决定。
6. 明确哪些动作可自主执行，哪些必须暂停确认。
7. 每个完成声明都指向工具证据。
8. 独立子任务并行委派，最终结论由主代理核验。
9. 用可修订 Markdown 记忆积累经验，去重并删除错误经验。
10. 最终总结重新建立用户上下文，而不是倾倒内部工作记录。
11. 逐步删除旧模型遗留的冗余提示和工具。
12. 通过独立验证者、测试和规格对照证明完成。

Fable 的正确用法不是给它更长的提示词，而是给它更大的问题、更清楚的责任边界和更真实的反馈回路。
